The process — recommend how agents get tested

1 · The 8-layer test strategy — a staircase to production

Microsoft's guidance layers agent testing from unit to accessibility. Shift left, automate in CI/CD, always finish in staging.

Layers 1–4 (build quality in)Unit tests per topic/tool → happy path conversations → knowledge tests (does grounding answer correctly?) → regression (yesterday's fixes still hold).
Layer 5 — adversarialEdge cases, unknown intent, deliberately false information — probe how the agent fails, not just how it succeeds.
Layers 6–8 — production-readinessPerformance/load, then security & compliance (RBAC, DLP, sensitivity labels, data leaks), then accessibility.
The three process rulesShift-left · always validate in staging before go-live · automate in CI/CD so tests gate promotion, not humans.

2 · Non-deterministic AI changes the method

The same prompt gives different words each run. Classic assertion testing breaks — replace it deliberately.

✗ Stop doing

  • Exact-match assertions on generated text — the words change every run even when the answer is right.
  • Single-run verdicts — one pass or one failure proves nothing about a stochastic system.

✓ Do instead

  • LLM-judged comparisons, rubrics, thresholds / pass rates, and repeated runs.
  • Assert the plan and tools chosen rather than exact wording — behavior over prose.
  • Multi-turn tests for conversational end-to-end scenarios.
💡 Hook: deterministic methods (exact / keyword / tool-use) still have a home — compliance-critical paths where the wording is regulated. LLM-judged methods own everything open-ended.

4 · The seven test methods — pick the right judge

Agent evaluation offers seven scoring methods. The exam names a need; you name the method.

General quality (default)LLM-judged relevance / groundedness / completeness / abstention — needs no expected answer.
Compare meaningSemantic comparison against an expected answer — same meaning passes, different words don't matter.
Tool usePass/fail on whether the expected tools/topics were invoked — “prove the orchestrator picks the right tools”.
Keyword match & Text similarityDeterministic textual checks — keywords present, or similarity above a bar.
Exact matchStrictest check — for fixed, regulated wording only.
CustomYour own evaluation instructions — you write the rubric the LLM judge applies.

Testing in Foundry — evaluations as metrics

6 · The evaluator map — which evaluator answers which question

Three families. The exam phrases a question in plain English; you name the evaluator.

Plain-English questionEvaluatorFamily
“Did the agent understand what the user wanted?”Intent ResolutionAgent
“Did it do what the instructions said?”Task AdherenceAgent
“Did it call the right tools with the right parameters?”Tool Call Accuracy (+ tool-selection/input/output, task-completion variants)Agent
“Is the answer supported by the sources?”Groundedness (+ Groundedness Pro)RAG
“Did retrieval find the right passages — and is the answer on-topic and complete?”Relevance · Retrieval · Response CompletenessRAG

💡 Hook: evaluations run local (SDK), cloud at scale, or in the portal, from an evaluator catalog governed by RBAC. Safety evaluators report a defect rate, not a 1–5 score.

7 · Scales, thresholds & release gates

Three numbers and one pipeline pattern carry most Foundry testing questions.

1–5LLM-judged quality score, pass ≥ threshold (default 3)
0–7content-harm severity scale behind safety evaluators
ASRAttack Success Rate — output of AI Red Teaming Agent (PyRIT-based) pre-launch scans
  • Run evaluations in CI/CD via GitHub Actions / Azure DevOps with baseline comparison and statistical significance — a release ships only if it's provably not worse.
  • AI Red Teaming Agent runs adversarial scans before launch producing the ASR scorecard — pair automation with human red teaming.

Validation criteria for custom AI models

8 · The validation-criteria matrix — per model type

A direct exam bullet: “create validation criteria of custom AI models.” One row per model class.

ModelValidation criteria
AI Builder predictionGrade A–D (A best). D = worse than random — or suspiciously ~100% accurate = data leakage / overfit. Accuracy on a held-out test set; inspect the confusion matrix and most-influential columns.
Numerical predictionGraded on R².
Category classification / entity extractionPerformance score from precision, recall, F1 + quick test.
Document processingPer-field / per-table accuracy score + improvement recommendations (accuracy shown for fixed-template models).
Fine-tuned models (Foundry)Training curves (next figure) → automatic safety evaluation (preview) can block deployment → Foundry evaluations vs expected pairs → deploy to a Developer deployment first → use the same system message as in training at inference.
Threshold philosophyThere is no absolute pass mark — optimize precision when false positives are costly, recall when false negatives are costly, F1 for balance. Acceptance is a business call.

9 · The overfitting curve — read the training charts

Healthy fine-tuning: both losses fall together. Overfitting: validation loss turns around while training loss keeps falling.

  • Watch train_loss ↓ and train_mean_token_accuracy ↑, plus valid_loss / full_valid_* per epoch.
  • Divergence between training and validation curves = overfitting → the fix is fewer epochs / lower LR multiplier.

12 · Building test cases with Copilot — the strategy

A direct exam bullet: let AI generate breadth, keep humans on review, regenerate when the agent changes.

Agent evaluation generatesTest sets built from instructions / knowledge / topics — and from analytics Themes, turning production telemetry into regression tests.
Agent Kit bulk creationAuthor hundreds of cases in Excel, import in one go.
Azure DevOps MCP server + Copilot“Create test cases for…” and coverage-gap analysis straight into Azure DevOps Test Plans; GitHub Copilot generates code-level tests.
The architect's rulesAI breadth → human review · LLM-judged methods for open-ended answers, deterministic for compliance-critical paths · store cases in Azure DevOps · regenerate when knowledge or instructions change.
💡 Final hook: map need→method instantly — “regression-test 500 Q&A pairs each release” → Agent Kit / evaluation in pipelines · “prove the right tools get picked” → Plan validation / Tool use · “model shows 100% accuracy” → data leakage, grade D.
