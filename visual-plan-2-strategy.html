A · The CAF AI adoption process

6 phases + 4 agent phases

1 · The six CAF AI-adoption phases

Three run in order, three never stop. Phase names are exam answers by themselves.

1CAF AI adoption = Strategy → Plan → Ready in sequence, then Govern, Secure, Manage run continuously.
2Know which artifact belongs to which phase — e.g., landing zones = Ready; NIST AI RMF alignment = Govern; AI CoE = Manage.

2 · Inside the Strategy phase — four steps

Strategy produces four outputs before anything is built.

1Use-case identificationbusiness problems → use cases → individual work vs business automation → generative vs non-generative
2Technology strategythe Microsoft AI decision tree — four adoption models trading customization for simplicity
3Responsible AI strategydefine responsible AI policies up front
4Data strategygoverned, high-quality data on a unified data platform
3Strategy step 1: AI use-case identification (the funnel from page 1 lives here).
4Strategy step 2: AI technology strategy via the Microsoft AI decision tree.
5Strategy step 3: responsible AI strategy — policies defined before build.
6Strategy step 4: data strategy — governed, high-quality, unified data platform.

3 · The four technology adoption models — the staircase

Each step up trades simplicity and speed for customization and control. The exam gives you a phrase — you name the step.

7Ready-to-use Copilots (SaaS) — "fastest ROI, minimal customization" → M365 Copilot, prebuilt D365 agents.
8Low-code SaaS development — Copilot Studio + M365 Copilot extensibility → "department workflow with SaaS governance."
9Managed PaaS — Microsoft Foundry (agents, RAG, fine-tuning; Azure ML / Fabric for training; Foundry Tools for prebuilt models) → "model choice, pro-code, evaluations."
10Azure infrastructure (IaaS) — bring-your-own models, most control, most ops → "strictest isolation."
11Weigh every option on four criteria: capabilities · data needed · skills required · cost.

4 · Inside the Plan phase — six steps and the maturity ladder

Plan turns strategy into staffing, resources and proofs of concept.

12Plan phase = assess skills → acquire skills → access resources → prioritize use cases → create PoCs → implement responsible AI (the startup checklist is shorter: access resources, establish responsible AI).
13PoC discipline: start with internal-facing use cases and add a +20–30% time buffer to estimates.
14Maturity ladder: L1 = ready-made Copilots · L3 = RAG apps in Foundry · L4 = training custom models on AKS/VMs.

5 · Inside the Ready phase — the landing-zone rules

Ready is where governance becomes Azure architecture.

Management groups

  • separate groups for online (internet-facing) vs corporate (internal) AI workloads

Azure Policy

  • AI-specific policy for Foundry, Foundry Tools, AI Search

Placement

  • deploy AI resources in workload subscriptions / application landing zones
15Ready: put internet-facing ("online") and internal ("corporate") AI workloads in separate management groups.
16Apply AI-specific Azure Policy for Foundry, Foundry Tools and AI Search.
17Ready also covers AI networking, reliability, foundation, and architecture choices — resources live in application landing zones.

6 · The continuous three — Govern, Secure, Manage

These loops never finish. Each has a fixed internal sequence the exam can quiz.

↻ Govern AI

  • assess organizational risks
  • document governance policies
  • enforce policies
  • monitor risks

↻ Secure AI

  • discover AI security risks
  • protect AI resources & data
  • detect AI security threats

↻ Manage AI

  • operations: AI CoE · standardize SDKs · sandboxes
  • MLOps (classic ML) vs GenAIOps (generative)
  • deployment · models · costs · data · continuity
18Govern loop: assess risks → document policies → enforce → monitor; enforcement uses Azure Policy + Purview.
19Governance aligns to the NIST AI Risk Management Framework; reviews: quarterly for high-risk, annually for low-risk.
20Secure loop: discover risks → protect resources and data → detect threats.
21Manage: establish the AI CoE; choose MLOps for classic ML vs GenAIOps for generative workloads.
22Keep a sandbox environment separate from dev/test/prod; onboard models sandbox-first, then the production catalog; manage drift, cost, continuity.

7 · The CAF agent adoption scenario — four phases

The companion track for agents specifically. Don't confuse its four phases with the six AI phases.

23CAF agent adoption = Plan → Govern & secure → Build → Manage (four phases, versus six for AI adoption).
24Plan-for-agents covers business plan, technology plan, organizational readiness, and data architecture; Build covers single-vs-multi-agent and the build process.

B · The build strategy

SaaS first · validate · scale

8 · The build decision tree — SaaS first, then pick your platform

CAF's primary question, then three build lanes. Trigger phrases in quotes.

25CAF's primary question: "Does a SaaS agent meet functional requirements?" If yes — adopt, don't build.
26Build lanes: Copilot Studio (low/no-code, built-in responsible AI, fast process transformation) · Foundry (PaaS, own models, deep integration, strategic transformation) · GPUs & containers (IaaS, compliance-sensitive).
27On IaaS: ACA serverless GPU for SLMs, AKS for LLMs.
28Hybrid pattern: Copilot Studio's low-code interface with a bring-your-own Foundry model — balances speed and capability.

9 · Validation discipline — experiments and metric gates

Strategy without gates is hope. CAF prescribes time-boxed proof and go/no-go metrics.

⏱Time-boxed experiments1–2 weeks per option, findings documented, stakeholder review; assess single- vs multi-agent before scaling
📏Baseline & gatesset baseline KPIs before building; use metrics as go/no-go decision gates at each stage
✂Post-deploymentevaluate against targets; retire or redirect underperforming agents
29Validate platform choices with time-boxed experiments (1–2 weeks per option), documented findings and stakeholder review.
30Baseline KPIs first, then use business metrics as go/no-go gates at every stage.
31After deployment, evaluate against targets — retire or redirect agents that underperform.

10 · Building at scale — the CAF build process

Once past the pilot, four pillars keep an agent factory healthy.

1Standardized environmentsconsistent, governed places to build and run agents
2Model selection strategymatch model capability to task complexity — small models for routine tasks; catalog + leaderboards; Model Router for productivity/action agents; quota & cost governance
3Evaluationsvalidate quality before deployment
4CI/CDrepeatable, automated delivery
32Scale pillars: standardized environments, model selection strategy, evaluations, CI/CD.
33Model selection: smaller models for routine tasks, premium models reserved for complex reasoning; pick via catalog and leaderboards; validate before deployment.
34Use the Model Router for productivity and action agents; enforce quota and cost governance.
35The architect profile expects "agentic-first" roadmaps built on open standards: MCP (Model Context Protocol) and A2A (Agent2Agent).

C · The AI Center of Excellence

responsibilities · evolution · operating model

11 · The CoE wheel — eight core responsibilities

CAF "Establish an AI Center of Excellence" lists these. Each spoke is a possible exam answer.

36Define AI strategy — align to business goals, use the AI decision tree, own the responsible AI strategy.
37Develop AI skills (assessments, learning pathways, hands-on experimentation) and lead pilot projects prioritized by impact/feasibility.
38Define and enforce standards: governance policies, security, data quality, model lifecycle, bias/transparency reviews, audits.
39Intake & prioritization workflows — structured project requests, consistent criteria, a prioritized backlog.
40Reusable assets (templates, code repos, compliance checklists on an internal platform) and measure & report KPIs: adoption rates, compliance levels, project cycle times.
41Optionally the CoE manages AI services: deployment governance, model monitoring, lifecycle.

12 · CoE evolution — centralized → advisory

The CoE starts as a central function and must let go. Watch for the three inflection signals.

42Shift centralized → advisory when you see approval delays, knowledge bottlenecks, or CoE-vs-team friction.
43The move: embed AI delivery into platform teams, distribute expertise to product/platform/enabling teams, keep guardrails.

13 · The three-way responsibility split (agent adoption)

Who owns what once agents are in production — a favorite scenario-question setup.

Platform team

  • technical foundation
  • governance guardrails & enforcement
  • observability

Workload teams

  • own agent lifecycles end-to-end
  • build & operate within guardrails

AI CoE

  • centralized advisory body
  • embeds responsible AI into policy
  • expert guidance · training
44Operating model: platform team enforces guardrails · workload teams own agent lifecycles · the CoE advises.
45Skills the CoE develops in the org: prompt engineering, agent optimization/evaluation, AI ethics & governance, data engineering for AI (RAG/vector indexes), AI security (prompt-injection/jailbreak defense).

14 · The agentic CoE — four functions, risk tiers, and the anti-patterns

The M365 agent-guidance flavor of the CoE, plus the three failure modes the exam names.

The 4 agentic-CoE functions

  • Govern — release gates, audit logs (who built/approved each agent)
  • Enable — golden paths, templates, maker training, community
  • Optimize — quality / cost / value monitoring
  • Scale — a repeatable pipeline

The 3 anti-patterns

  • Gatekeeper — every decision bottlenecks through the CoE
  • Ghost — sets standards nobody sees enforced
  • Perfectionist — polishes forever, ships nothing
46The agentic CoE governs, enables, optimizes and scales agent work — complementing existing security/cloud/Power Platform CoEs.
47Risk-tiered governance: knowledge-only agents = light review; write-access agents = cross-disciplinary review.
48Power Platform CoE core roles: executive sponsor, CoE lead, SMEs, developers/IT pros, change management specialists, technology admins, champions — and the CoE Starter Kit remains the tenant-governance toolkit.

15 · One customer record, four agents — the shared-Dataverse journey

CE apps share one Dataverse environment, so a record flows through app-specific agents without integration work. ERP joins via Dataverse and Fabric.

49CE apps (Sales, Customer Service, Field Service, Customer Insights) share one Dataverse environment — a record flows lead → opportunity → case → work order across app agents.
50Finance & operations data integrates through Dataverse virtual tables and Link to Fabric for near-real-time analytics without ETL.

16 · Cross-app agent scenarios and design considerations

The combinations the exam likes, and the four design considerations for multi-app solutions.

Cross-app combinations

  • Sales Qualification Agent research + Customer Insights journeys/events
  • Supplier Communications Agent (SCM) + Account Reconciliation Agent (Finance) sharing Copilot for Finance context
  • Sales agent in M365 Copilot synthesizes CRM + M365 data; Copilot for Sales / Service bring D365 (or Salesforce) into Outlook/Teams
  • Customer Service knowledge configured in Copilot Studio reused across Contact Center channels
  • a "front door" agent routes to app agents via connected agents; Foundry/Fabric agents add analytics

Design considerations

  • environment strategy — which apps share environments
  • master data — duplicate detection, one source of truth
  • consistent security roles across apps
  • consumption billing — many D365 agent capabilities meter Copilot Studio credits or pay-as-you-go
51Many D365 Copilots are built on Copilot Studio — e.g., the F&O in-app help agent uses the conversational boosting topic, so you can add files/SharePoint knowledge and enable general knowledge there.
52A multi-app knowledge strategy centralizes sources in Copilot Studio; D365 Customer Service can reuse Copilot Studio-configured knowledge.
53Use connected agents for a front-door agent that routes to app-specific agents; interoperate with Foundry and Fabric agents for cross-app analytics.
54Design checklist: environment strategy · master data/duplicate detection · consistent security roles · consumption billing.
55Positioning sentence to remember: D365 "combines intelligent AI agents, Copilot experiences, and built-in AI across ERP and CRM" — apps work alone or connected.
