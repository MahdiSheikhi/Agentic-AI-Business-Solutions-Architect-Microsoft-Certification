1 · The money formulas

1ROI = (benefits − costs) ÷ costs; classify benefits as tangible vs intangible.
2Payback period = cost ÷ monthly net benefit; also know NPV, CAPEX vs OPEX, net cash flow from the business-case glossary.
3Benefit categories: efficiency/cost reduction, speed/responsiveness, scalability; measurable results: lower operating costs, improved CSAT, faster innovation.

2 · The four-pillar value framework — and honest measurement

Microsoft's agent business-value guidance: quantitative + qualitative metrics per pillar, leading vs lagging indicators.

Framework mechanics

  • four value pillars, each with quantitative + qualitative metrics
  • leading indicators predict; lagging indicators confirm
  • time saved = leading — track where reclaimed time goes

Measurement discipline

  • baseline before rollout
  • keep a non-user comparison group
  • avoid vanity metrics: seats purchased ≠ value
5The value framework uses four pillars with quantitative and qualitative metrics, plus leading vs lagging indicators.
6Time saved is a leading indicator — the value story depends on where the reclaimed time goes.
7Honest ROI: baseline before rollout, keep a non-user comparison group, reject vanity metrics like seats purchased.

3 · Agent Assisted Hours — the published formula, worked

Weight sessions by knowledge grounding → minutes saved → hours → dollars. The exam can hand you these numbers.

8Agent Assisted Hours: weight sessions by knowledge grounding, multiply by minutes saved, convert to hours × loaded hourly rate → dollar value.
9Worked example: 14,400 weighted sessions × 6 min ÷ 60 = 1,440 h/month × $72 = $103,680/month (~$1.24M/yr).
10The Savings calculator (time or money saved per run/tool) feeds the Analytics tab; admins can disable money rules; 16 use-case blueprints + a metrics reference library support estimates.

4 · The TCO iceberg

Licenses float above the waterline; most of the cost is under it.

11TCO = licenses + consumption (credits, tokens, storage, compute, grounding connections, AI Search) + everything below the waterline.
12Below the line: implementation/integration, data preparation and AI-readiness, fine-tuning training + hosting, testing/evaluation.
13Ongoing: monitoring/administration, D365 agent consumption or pay-as-you-go, training and change management.
14Classic TCO component list: hardware, software/licensing, maintenance, energy, personnel, downtime, replacement — vs the platform's consumption cost (credits/tokens/PTU).
15Quote to remember: "the license isn't the investment — adoption is."

C · The ROI analysis method

define → measure → tell the story

7 · The ROI pipeline — eight steps for any business process

The repeatable exam method: value defined before build, gates in the middle, analytics at the end.

25Define value before you build: four discovery questions, a prioritization grid, a baseline protocol (current handle time, volume, cost per transaction), plus a licensing/consumption plan and prebuild governance checklist.
26Estimate benefits as hours saved × loaded cost, deflection %, revenue uplift; compute TCO as licenses + credits/consumption + build + run + change management.
27Produce ROI, payback and NPV, then set decision gates before building further.
28Measure post-deployment with Copilot Studio Analytics, the savings calculator, and Power Platform admin center consumption reports — then iterate or retire.
29"Tell the value story" uses three measurement tiers, the value curve over time, and an executive scorecard template.

9 · Build · buy · extend — decided per use case

Buy sits at the top, extend in the middle, build at the bottom — with tradeoffs at every level.

OptionWhenTradeoff
Buy / adopta prebuilt SaaS agent covers ≥ the functional need (M365 agents, D365 agents, GitHub Copilot, Fabric/Azure/Security Copilot)fastest value, least customization
Extendthe gap is knowledge/actions on top of an existing Copilot/agent — M365 extensibility, D365 agent + Copilot Studio, agent templatesrapid, "clear and measurable ROI" — bounded by the host
Build low-codeprocess is unique but integration is standard → Copilot Studiobusiness-team velocity; heavy customization can force a re-platform to PaaS
Build pro-codecustom models, orchestration, isolation, multi-agent beyond low-code → Foundry / IaaScontrol + engineering skill; IaaS = maximum ops ownership
32The CAF question again: "Does a SaaS agent meet your functional requirements?" — yes → buy/adopt; no → build (Studio low-code / Foundry PaaS / GPUs-containers IaaS).
33Extend sits between: M365 Copilot extensibility (declarative agents, connectors, API actions), extending D365 first-party agents via Copilot Studio, and customizable agent templates.
34Adoption-model tradeoffs: SaaS = fastest value/least custom; SaaS dev platforms can hit limits and force a re-platform to PaaS; PaaS = control + skills; IaaS = maximum control + maximum ops.
35Partner-built agents from the Agent Store / Marketplace are "buy" options; the WAF SaaS AI table compares buy prebuilt / customize (fine-tune) / build your own.
36Practice: time-boxed prototypes comparing low-code vs pro-code before committing, with a stakeholder-reviewed recommendation.

10 · The model router — one endpoint, many brains

A deployable Foundry "model" that is really a trained routing model: it reads each prompt and forwards it to the best underlying chat model.

37Deployed like any Foundry model (model-router in the catalog): one deployment, one API, one rate limit — cutting cost and latency while keeping quality.
38Transparency & trust: the chosen model is disclosed in the response's model field; the router doesn't store prompts and routes only within your data-zone / compliance geography and access.
39Routing modes: Balanced (default, within ~1–2% of top quality) · Cost (~5–6% quality band, cheapest) · Quality (best regardless of cost); mode changes take up to 5 minutes.
40Model subsets control which models participate; new models are excluded by default until added; the router recalibrates automatically.
41GA version 2025-11-18: Global Standard + Data Zone Standard; underlying models include GPT-5 series, gpt-4o/mini, Anthropic Claude (must be deployed separately first), Grok, DeepSeek, gpt-oss-120b, Llama-4 — all others need no separate deployment.
42Constraints: effective context window = the smallest underlying model (use subsets for large-context needs); content filter and TPM limits are set once at the router; supports tools/agentic use (Foundry Agent Service, per-step model fit); reasoning_effort supported; temperature/top_p ignored when a reasoning model is selected.

11 · Router vs direct deployment — and the hybrid pattern

The closing judgment call in the Plan domain.

Choose the router when…

  • diverse prompt mixes · cost optimization
  • latency-sensitive high traffic · agents
  • single-endpoint simplicity · automatic failover
  • simplified lifecycle — deprecated models replaced transparently (auto-update)

Choose direct deployment when…

  • you need the same model on every request
  • parameter-sensitive or compliance-pinned workloads
  • hybrid pattern: router as default path + direct deployments for specialized workloads
43Router for mixed workloads, cost, latency, agents, failover, single endpoint; direct deployment for same-model-every-request, parameter-sensitive or compliance-pinned workloads.
44Hybrid pattern: router as the default path, direct deployments for specialized workloads.
45CAF cost governance quote: "for productivity and action agents, use the Model Router to dynamically optimize costs while maintaining quality."
46Trigger pair: "reduce cost across mixed workloads, one endpoint" → router (Balanced) · "audit-critical legal/medical" → Quality mode or a pinned direct deployment.
