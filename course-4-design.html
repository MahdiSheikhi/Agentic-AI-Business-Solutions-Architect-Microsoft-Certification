System components & autonomy

the anatomy of every agent
In brief: every agentic system — from a Studio chatbot to a Foundry multi-agent fleet — has six components: perception, reasoning, memory, tools, execution runtime, and guardrails, each with a specific Microsoft platform home. They run as a cycle, illustrated by a Teams order-status inquiry that remembers a repeat customer. The other core design choice is autonomy — copilot mode, semi-autonomous, or fully autonomous — decided by business risk, not technology, and shiftable per action within one agent.

1 · The six components — and their Microsoft homes

The exam describes a business need and expects you to name the relevant (or missing) component and its platform mapping.

👁️Perceptionthe front door — Teams messages, Dataverse events, Power Automate triggers, sensor events.
🧠Reasoningthe brain — Azure OpenAI models (GPT-4o) via generative orchestration; chain-of-thought, ReAct.
💾Memoryshort-term = conversation context; long-term = Dataverse / Azure AI Search vectors / SharePoint knowledge.
🔧Toolsthe hands — Power Platform connectors, MCP servers, Power Automate flows, custom APIs.
☁️Execution runtimeCopilot Studio (most agents) · Foundry Agent Service (multi-agent) · Container Apps (custom).
🛡️Guardrailscontent safety filters, DLP, Entra ID auth, escalation when confidence drops — non-negotiable.

The cycle in one story: a Teams message about an order (perception) → reasoning finds a repeat inquiry in memory → calls the Dynamics connector via the runtime (tool) → drafts a reply → guardrails verify authentication and data-sharing safety → responds, proactively mentioning the shipping delay it flagged last time.

1None of the six are optional — exam scenarios often test what's missing from a proposed design.
2Without long-term memory every conversation starts from scratch — the support agent that forgets yesterday's call is the canonical bad design.

2 · The three autonomy levels — a business-risk decision

How much freedom the agent gets is decided by the cost of an error, not by what's technically possible.

3Autonomy is a business-risk decision, not a technology decision — can the company afford the consequence of a mistake?
4Autonomy isn't all-or-nothing within one agent — design it to shift by the risk of each specific action.
5The exam rewards safe and auditable over clever — balance autonomy with strong governance in every answer.

Orchestration patterns

making agents work together
In brief: orchestration is the conductor: needed whenever a problem takes more than one step or agent. Five patterns — sequential, parallel, hierarchical, dynamic, event-driven — each with a signature example and trade-off; real solutions mix them, as in the manufacturing reorder scenario that layers an event trigger, a supervisor, three parallel specialists, and a human above $50K. Agents hand off via A2A (speed), MCP-wrapped tools (governance), or shared Dataverse tables (async + audit), and reliability patterns — circuit breakers, backoff, timeouts, fallbacks, observability — keep multi-agent systems from cascading failure.

Copilot Studio design

topics · tools · flows — and three methods by name
In brief: every Studio agent is topics (ears — what it understands), tools (hands — what it does), and flows (brain — the wiring, including every failure path). The exam tests three topic-routing methods by name: classic NLU (trigger phrases, brittle), CLU (trained intent model with entities), and generative orchestration (the LLM routes by your descriptions — now the default). Prompt actions turn the LLM itself into a tool, and the three agent types return with design considerations and a password-reset walkthrough.

6 · Ears, hands, brain

Keep them separate: one topic = one job, tools reusable across topics, flows owning branching and error paths.

10One topic, one job — balance, history, and settings crammed into one topic is an untestable mess.
11Tools are reusable: five topics needing a customer lookup share one "fetch customer" tool — fix a bug once, fixed everywhere.
12Every flow explicitly handles failure — API timeout, missing account — not just the happy path.

7 · The three orchestration methods — tested by name

How the agent decides which topic or action fires when a message arrives.

MethodHow it routesBest forTrade-off

13Generative orchestration is the default for new agents — and your topic descriptions become the routing logic, so write them well.
🧠 Hook: NLU matches phrases, CLU matches patterns, generative matches meaning.

8 · Prompt actions, the three agent types & the password-reset walkthrough

A prompt action uses the LLM itself as the tool — no API, no code. Then match the agent type to the task.

⚙️ Prompt actions — LLM as a tool

  • Define inputs → natural-language prompt template → expected output shape
  • Ticket triage: "one-sentence summary + severity low/medium/high" for hundreds of tickets daily
  • Sales recs: purchase history in → three likely next products with reasons out
  • Shines at transform, summarize, classify, generate structure — iterate by editing the prompt

🧩 The three agent types, with design duties

  • Prompt & response — knowledge-grounded Q&A (HR facts agent): current sources, clear system prompts, out-of-scope fallback
  • Task — multi-step workflows (IT password reset): clear action chains, every failure path, escalation to humans
  • Autonomous — scheduled/event-run (2 a.m. expense processor, auto-approve < $500): strongest guardrails, monitoring, spending limits, audit logs

Walkthrough: "password reset" topic (triggers: "I can't log in", "locked out") under generative orchestration → action 1 verifies identity via the Entra ID connector → action 2 triggers the reset through a Power Automate flow → guardrail: two failed verifications → stop and escalate to the help desk (anti-social-engineering) → task agent, because it acts on request but isn't scheduled.

14The exam penalizes over-engineering (autonomous agent for simple facts) and under-engineering (prompt-and-response for multi-step work) equally.
15Autonomous agents run with no human watching — the controls built into the agent are all you have.

Copilot for Sales & Service

prebuilt, then extended
In brief: Copilot for Sales lives in Dynamics 365 Sales and Outlook/Teams — a bidirectional CRM↔productivity bridge powering lead scoring, two-minute meeting prep, contextual email drafting, and automatic win-loss analysis. Copilot for Service pairs with Contact Center (the omnichannel infrastructure) to summarize cases, suggest resolutions, draft knowledge articles, and route intelligently. Know the business vocabulary the exam won't define, and extend progressively: configuration → Studio topics → connectors → Power Automate.

9 · What the two copilots actually do

Capabilities with concrete reference numbers attached.

💼 Copilot for Sales — D365 Sales + Outlook + Teams

  • Lead & opportunity scoring — focus on the 10 most likely of 50; pipeline velocity +20–40%
  • Meeting prep — deals, issues, emails, notes assembled: 20 min → 2 min
  • Email drafting — personalized from deal context: 15 min → 2 min
  • Win-loss analysis — automatic, feeds coaching & forecasting
  • The bridge: surfaces CRM insight in Outlook/Teams, pushes activity back to CRM — bidirectional

🎧 Copilot for Service — D365 Customer Service + Contact Center

  • Case summarization — the whole trail in one synthesis; pick up cases instantly
  • Resolution suggestions — ranked matches from resolved cases & KB articles; every rep gets the veterans' knowledge
  • KB authoring — novel fix detected → drafts the article from case notes
  • Intelligent routing — issue type, segment, urgency, sentiment: angry enterprise billing dispute → senior specialist
  • Division of labor: Contact Center = omnichannel infrastructure; Copilot = the AI layer on top
16Copilot for Sales is a connector between Dynamics 365 and Microsoft 365 — that bidirectional detail is exam bait.

Finance & Supply Chain AI

the operations side + cross-module
In brief: Copilot for Finance automates three-way invoice matching, anomaly detection, compliance checks, accruals, and variance analysis — always with audit trails. The Supply Chain copilot is more autonomous: stockout prediction, quality flagging, logistics optimization, inventory monitoring — event-driven agents governed by confidence thresholds and escalation rules. The real enterprise value is cross-module orchestration over the shared Dataverse backbone, and F&O agent chat can be extended with custom knowledge sources mapped to user roles.

12 · Finance vs SCM copilots — and the warranty-claim relay

Two operating personalities, one shared backbone.

🧾 Copilot for Finance — checks & balances

  • Three-way matching: invoice vs PO vs receiving — all match → auto-approve; mismatch → human review. Hours per batch → seconds
  • Anomaly detection: invoice 30% above vendor norm → flagged (price rise? typo? fraud?)
  • Compliance checks: blacklisted vendor, blown budget, exceeded authority → routed to the right approver
  • Month-end: accrual automation + variance analysis — investigate outliers, not everything

🚚 Copilot for Supply Chain — autonomous watchkeeper

  • Stockout prediction: demand forecast × inventory × lead times — "component X runs out Wednesday, three lines halt" → alert or auto-reorder
  • Quality flagging: defect rate 2% → 8% on a shipment → investigate before accepting
  • Logistics optimization: consolidate shipments, pick carriers by cost vs deadline
  • Inventory watch: slow movers → clearance; fast movers → order ahead of demand
18SCM agents are autonomous and event-driven — governance questions target confidence thresholds, escalation rules, and monitoring.
19F&O agent chat extends with custom knowledge sources (internal policies, supplier procedures) — mapped to roles: AP clerks get invoice policy, planners get procurement rules.

Microsoft 365 Copilot & Teams agents

four extensibility routes
In brief: M365 Copilot extends four ways — Copilot Studio plugins (multi-step logic), declarative agents (JSON-manifest, no-code, the exam's favorite new concept), Graph connectors (external data as native M365 content, with mandatory item-level ACLs), and message extensions (quick card actions in Teams). Governance rides on data classification: sensitivity labels first, then DLP, conditional access, and audit logging; admins scope agent availability through the Teams admin center. The HR worked example combines a Graph connector, SharePoint policies, and a declarative agent.

13 · Match the route to the scenario

Four options, one selection rule: choose by complexity.

🔌Copilot Studio plugincustom agent surfaced in Teams/Outlook — for multi-step workflows & external APIs (query monday.com from Teams).
📄Declarative agentdefined entirely by a JSON manifest — instructions + knowledge sources + capabilities. No code; deployed via Teams admin center. For simple scoped Q&A.
🕸️Graph connectorexternal data exposed through Microsoft Graph as if native M365 — searchable by Copilot like SharePoint. For external system data.
🃏Message extensioncard-based actions in the Teams compose box — quick lookups, approvals, data entry.
20Declarative agent manifests have three sections: instructions (scope) · knowledge sources (SharePoint/files/URLs) · capabilities (e.g., code interpreter on/off). Out-of-scope questions are declined back to normal Copilot.
21Graph connectors must carry item-level ACLs — without them Copilot can surface confidential data to unauthorized users. That's a governance failure the exam tests.
22Classification comes first, then governance: without sensitivity labels (public / internal / confidential / restricted), Copilot has no boundaries to enforce. DLP, conditional access, and audit logging build on top.
23Admins control discovery: integrated apps catalog + M365 admin center — enable, disable, or scope agents per user group; monitor adoption via usage analytics.

Grounding, knowledge sources & RAG

real data in, accurate answers out
In brief: an ungrounded agent answers "30 days" because that's common in training data, when your refund policy says 60. RAG — retrieve, augment, generate — is the fix and the most important enterprise pattern. Sources match data types (Dataverse for structured, SharePoint + Azure AI Search for documents, Graph for M365 context, MCP for legacy), Copilot Studio combines five knowledge-source types in one agent, and data processing (chunking, embeddings, hybrid search, cleaning, freshness) makes retrieval precise. Citations and verification are not optional.

25Copilot Studio's five knowledge-source types — public websites, SharePoint, Dataverse, uploaded files, Graph — can combine in one agent; the orchestrator searches across all of them.
26Processing that makes retrieval work: chunk by section (or one question returns a 50-page manual), embed for semantic match ("return a gift" finds "gift returns…30 days"), hybrid search, clean duplicates, manage freshness — last year's pricing is worse than no agent.

MCP, RPA & agent behaviors

advanced extensibility
In brief: MCP puts a governed middleman between agents and APIs — permission checks, input validation, rate limits, full logging, and no API keys in agent hands; REST API tools and prebuilt skills are the lighter-weight alternatives for single integrations. UI automation/RPA drives legacy screens when no API exists — always with whitelisted actions, confirmations, and timeouts. Configurable behaviors round it out: reasoning mode, voice mode, autonomous triggers, and grounded generative answers.

Power Platform Well-Architected

five pillars + responsible AI
In brief: the Power Platform Well-Architected Framework adapts Azure's five pillars — reliability, security, cost optimization, operational excellence, performance efficiency — for agentic AI, and layers responsible AI (fairness, transparency, accountability, safety) on top. WAF questions are scenario-based: "which principle is most at risk?" — and the best answers apply several pillars at once, as the health-insurance claims walkthrough shows. Generative pages, the AI Hub, and AI Builder each tie back to specific pillars.

17 · Six lenses on one design

The pillars with their signature patterns — then all six applied to the claims-processing agent.

🛠️Reliabilityretry with backoff · circuit breakers · graceful degradation — keep functioning when dependencies fail.
🔐Securityleast privilege (agent sees only the user's data) · MCP wrapping (no keys) · prompt-injection defense (validate inputs).
💰Cost optimizationmodel routing (80–90% of requests fit the cheap model) · caching · batch processing.
📟Operational excellencedashboards (latency, errors, escalation, tokens) · alerts on spikes · runbooks for common failures.
⚡Performance efficiencyparallelize independent steps · async notifications · retrieve 3–5 chunks, not 100.
🧭Responsible AIfairness (bias audits) · transparency (plain-language decisions + appeal path) · accountability (logs + human owners) · safety (filters + human review for high stakes).

Claims-agent walkthrough: queue-and-retry if the necessity model is down, auto-approve only low-risk (reliability) · claims via MCP, PII encrypted, role-based patient access (security) · cheap model triages the 90% of straightforward claims (cost) · claims/hour + escalation-rate dashboard with alerts (op-ex) · validation, coverage, and necessity checks in parallel, emails async (performance) · monthly bias audit of denial rates by demographic, every denial explained with appeal instructions (responsible AI).

29Address all six lenses in every design — a loan-approval system needs security and fairness; the keyed answer usually applies multiple pillars.
30Platform tie-ins: AI Hub = tenant-wide AI usage/cost visibility (op-ex + cost) · generative pages = performance + grounding · AI Builder for repeatable tasks vs Copilot control for free-form chat.
31Responsible AI is a foundation, not a feature — it applies to every agent that makes or influences decisions.

Custom models in Foundry

the end-to-end toolchain
In brief: the design-level decision framework: prebuilt models for standard tasks, fine-tuned for specialized language, custom-trained only for truly unique problems, SLMs for high-volume cost-sensitive work. Foundry covers the whole lifecycle — model catalog, prompt flow for prototyping, evaluation tools, fine-tuning, managed-endpoint deployment — and a seven-step workflow runs from task definition to integrated, drift-monitored production models connected to agents via MCP or Copilot Studio actions.
