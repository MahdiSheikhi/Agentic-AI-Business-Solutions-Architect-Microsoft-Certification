A · Multi-agent solutions

1 · Single agent or multi-agent?

CAF's default is humble: start single. Split only when a real boundary forces it.

Single agent

  • simpler, predictable, less overhead
  • one config, one lifecycle

Multi-agent

  • modularity, separation of concerns, scalability
  • but more coordination & orchestration cost
1Single-agent = simpler, predictable, less overhead; multi-agent = modularity and scalability at the price of coordination.
2Start single-agent unless the use case crosses security/compliance boundaries, involves multiple teams, or has known future growth.

2 · Copilot Studio — three ways to add other agents

Inside the same agent, inside the same environment, or across platform borders.

3Child agents live inside one agent (same config/lifecycle) to logically group tools, instructions and knowledge.
4Connected agents are other published Copilot Studio agents in the same environment, exposed via "Let other agents connect to and use this one" — with the option to pass or withhold conversation history.
5External agents connect over A2A, plus Foundry agents, Fabric data agents, and M365 Agents SDK agents (all preview; SDK via Activity Protocol).
6Routing: the orchestrator picks a connected agent from its name/description, the user message/context, and the primary agent's instructions — write distinct, specific descriptions.

3 · Child vs connected — the split decision

One number and four ownership signals decide it.

SignalStay with child agentsSplit into connected agents
Scaletool selection still accurate> 30–40 actions (tools/topics/agents) degrades selection
Ownershipsingle team, single use casedifferent teams own agents
Lifecycleno separate deploymentneed separate ALM, channels, model settings
Reusenot reused elsewherereused across multiple main agents
7Rule of thumb: break into connected agents when tool selection degrades — more than 30–40 actions.
8Also split for separate team ownership, separate ALM/channels/model settings, or reuse across main agents; child agents = one team, one use case, no separate deployment.

4 · Multi-agent gotchas — the fine print that becomes questions

Latency, topology and knowledge caveats.

⏱Latencyevery extra orchestration hop adds latency; testing & governance surface grows
⛓Topology limita main agent with connected agents cannot itself be a connected agent of another main
🧩Fabric limitsFabric data agents can't be redirected from topics and don't work when the main agent is deployed to M365 Copilot
📎Knowledge caveatscitations may be lost across agent boundaries; connected agents respect the main agent's general-knowledge setting
9Extra orchestration hops increase latency and grow the testing/governance surface.
10Topology: a main agent with connected agents can't be a connected agent of another main agent.
11Fabric data agents: not redirectable from topics; unavailable when the main agent is deployed to M365 Copilot; citations may be lost across agent boundaries.

5 · Protocol picker — MCP vs A2A vs plain HTTP

Tools speak MCP, foreign agents speak A2A, plain APIs just use connectors.

12MCP for tools/resources · A2A for foreign agents · HTTP/custom connectors for plain APIs.
13A2A is an open-standard contract for agent-to-agent communication: send tasks, structured metadata, predictable responses, multi-turn.

11 · Copilot Studio guardrails — the four layers

From instructions down to tenant admin controls. "Rules and constraints" = all four.

29Instructions define scope and behavior; child/connected-agent descriptions steer routing; prompt modification/custom instructions shape generative answers.
30Content moderation is configurable at three levels: agent (Generative AI settings), topic (generative-answers node — takes precedence at runtime), and prompt tool.
31Moderation runs Lowest → Highest; higher = fewer but safer answers; default is High.
32Knowledge constraints: Allow ungrounded responses off = block answers not derived from knowledge/tools that turn (fallback topic triggers); Use general knowledge off = configured knowledge only.
33Web search uses Grounding with Bing; tenant graph grounding with semantic search requires generative orchestration + "Authenticate with Microsoft" and bills +10 credits for non-Copilot-licensed users.
34Built-in: moderation on all generative requests, checked twice (input and pre-response), covering jailbreak, prompt injection, prompt exfiltration, copyright — plus RAG citation/provenance checks and a default AI disclosure message.
35Admins can disable publishing generative-answer agents tenant-wide; DLP policies govern which connectors/knowledge/tools agents may use.
36Agent Builder (declarative lite) can only prioritize specified sources — it can't fully block general knowledge; use Copilot Studio for stricter control.

12 · Foundry & Foundry Tools guardrails

Content Safety plus Agent Service controls — configured on deployments, enforced in code.

🛡Azure AI Content Safetyharm filters: hate · sexual · violence · self-harm with severity thresholds; Prompt Shields (jailbreak / indirect injection); groundedness detection; protected-material detection; blocklists
⚙Agent Service constraintsinstructions per agent · tool allow-lists (catalog, MCP, OpenAPI) · basic vs standard setup (network isolation, BYO storage) · Azure Policy for allowed models · Entra RBAC · evals & tracing
📐WAF guardrail patternsalways ground · hallucination filters · enforce tool use in code (don't trust instructions) · document refusal thresholds
37Content Safety filters four harm categories — hate, sexual, violence, self-harm — with severity thresholds, plus Prompt Shields, groundedness detection, protected-material detection and blocklists.
38Content filters are configured on model deployments; with model router, the filter and rate limits apply once at the router level for all underlying models.
39Agent Service: per-agent instructions, tool allow-lists, basic vs standard setup (standard = network isolation, BYO storage) for compliance, Azure Policy on allowed model deployments/publishers, Entra RBAC.
40WAF patterns: always ground, hallucination filters, enforce tool-use in code — don't rely on the model to follow instructions — and document refusal thresholds.

15 · Extend M365 Copilot or build custom — declarative vs custom engine

The classic extensibility fork. One inherits everything; the other owns everything.

Declarative agent — extend Copilot

  • uses Copilot's orchestrator + foundation models
  • instructions + knowledge + actions; no extra hosting
  • inherits M365 security / compliance / RAI
  • user-initiated only; M365 surfaces (Copilot Chat, Teams, Word, Outlook, SharePoint)
  • build with SharePoint agents · Agent Builder · Copilot Studio · Agents Toolkit

Custom engine agent — build your own

  • your orchestration & models (any model, incl. fine-tuned/industry)
  • proactive / autonomous triggers, complex workflows, agent-to-agent collaboration
  • deployable to M365 and external channels (web, custom apps)
  • you own hosting, security, RAI compliance
  • build with Copilot Studio · M365 Agents SDK · Teams SDK · Foundry (Semantic Kernel / LangChain)
48Declarative = Copilot's orchestration + models, no hosting, inherited compliance, user-initiated, M365-only surfaces.
49Custom engine = own orchestration/models, proactive triggers, external channels, own hosting/security/RAI.
50Decision drivers: stay declarative for quick low-code, M365-only, inherited compliance; go custom engine for model choice, autonomy, non-M365 channels, advanced integration. A declarative agent can be converted to custom engine (Agents Toolkit path).
51Licensing: declarative agents are covered for M365 Copilot-licensed users; unlicensed users use them in Copilot Chat via metered Copilot Credits; custom engine/Foundry consume Azure meters.
52CAF placement: "extensibility tools for M365 Copilot" (declarative agents, Agent Builder, Agents Toolkit, Copilot connectors) sit between ready-to-use Copilots and Copilot Studio on the adoption spectrum.

16 · When to create custom AI models — the customization ladder

Escalate only as needed. Each rung costs more than the one below it.

53The ladder: prompt engineering → RAG → fine-tuning → train your own; CAF says "buy and ground prebuilt models" covers most use cases.
54Fine-tune when you have hundreds-to-thousands of quality examples and need consistent style, specific output schemas, better tool-calling accuracy, or token/latency savings — costs = training (tokens × epochs × price) + hosting, plus re-tuning when base models change.
55Distillation: use a large model's outputs to fine-tune a smaller, cheaper model (e.g., o1 traffic → GPT-4o-mini).
56AI Builder: prebuilt models (receipts/invoices/ID, sentiment, language, entities, OCR, translation) → use immediately; custom models (document processing, object detection, prediction, classification) when data is unique — requires Dataverse and consumes Copilot Credits.
57WAF decision factors: data control, customization need, cost/upkeep, performance/scale, expertise — "usually, choosing a prebuilt model or managed service is a better option."

17 · Small language models — when small wins

SLMs (Phi family) trade breadth for speed, cost, privacy and deployability.

＋Advantagesfaster & cheaper to train/run · small footprint (edge/on-device) · less overfitting · more interpretable
🎯Use casesnarrow well-defined tasks · limited data/resources · high-precision tasks · sensitive/regulated needs · classification, summarization, entity extraction, simple Q&A
ΦPhi familyPhi-3 (mini 3.8B, small 7B) · Phi-4 incl. Phi-4-mini (Edge on-device Prompt API) · MIT-licensed · in the Foundry catalog, fine-tunable
🚀Deploymentlocal sidecar on App Service (privacy, sub-100ms, fixed cost) · ACA serverless GPU · edge/on-device (Win10/11 or macOS 13.3+, 20 GB storage, 5.5+ GB VRAM)
58SLM advantages: faster/cheaper, small footprint, less overfitting, more interpretable.
59SLM use cases: narrow tasks, limited data, high precision, sensitive/regulated scenarios — classification, summarization, entity extraction, simple Q&A.
60Phi = Microsoft's SLM family (Phi-3 mini 3.8B / small 7B; Phi-4-mini powers Edge's on-device Prompt API), MIT-licensed, fine-tunable in Foundry.
61Patterns: App Service local sidecar (complete privacy, zero external dependencies, predictable sub-100ms latency — healthcare/finance/legal/offline) · ACA serverless GPU · edge. Choose SLM for cost/latency-sensitive high volume or residency; choose LLM for broad reasoning and complex orchestration.

18 · The prompt library — gallery, builder, governance

A curated, shared, governed collection of reusable prompts — two products, one discipline.

M365 Copilot Prompt Gallery

  • Microsoft-curated + "Your Prompts" + team-shared prompts
  • save, like, share with teams
  • admins can track and export saved/liked/shared prompt data

Copilot Studio / Power Platform prompt builder

  • predesigned templates: document extraction, data transformation, content generation…
  • input variables (text, image/document) + Dataverse knowledge grounding
  • runs on Foundry-hosted models (GPT-4o/4o-mini; model + temperature configurable; JSON output)
62Prompt Gallery = curated + personal + team prompts inside M365 Copilot with save/like/share; admins export usage data for governance.
63Prompt builder prompts take input variables, ground on Dataverse knowledge, run on Foundry-hosted models with configurable model/temperature and JSON output — they consume Copilot Credits and require Dataverse.
64Sharing rules: new prompts are private by default; recipients can use but not edit or delete; views = My prompts / Shared with me / All prompts.
65Library guidelines: seed from templates, standardize structure (instruction + context + inputs + output format), test before sharing, version centrally (CoE-style), and update monthly from analytics themes.

19 · Prompt engineering — GCSE and the technique toolbox

Microsoft's four prompt ingredients, the shot ladder, and the best-practice list the exam quotes.

GGoalwhat output you want — "write a step-by-step cleaning procedure"
CContextwhy / for whom / background — "for SmartBrew machine owners"
SSourcewhich data, knowledge or plugins to use — "use the company knowledge base"
EExpectationsformat, tone, audience, length — "casual tone, numbered list"

The shot ladder & system message

  • zero-shot = no examples (cheap baseline)
  • one-/few-shot = example input→output pairs as user/assistant turns after the system message; vary examples, cover edge cases
  • system message defines role, boundaries, tone, output format, safety — influences but doesn't guarantee compliance; make fallback behavior explicit

Technique toolbox

  • Be specific · Be descriptive · Double down · Order matters · Give an "out"
  • cues (seed the output format) · primary/supporting content · break the task down step-by-step
  • recency bias → instructions before and after content
  • temperature/top_p tuning — reasoning models ignore them
66GCSE — Goal, Context, Source, Expectations: Microsoft's canonical four prompt elements; iterate and refine.
67The five best practices verbatim: Be specific. Be descriptive. Double down. Order matters. Give the model an "out" ("respond 'not found'" reduces fabrication).
68Studio-specific: descriptions are functional configuration (they drive orchestration); for flows use "When X happens, do Y" phrasing; prompt engineering is the recommended starting point before fine-tuning.
