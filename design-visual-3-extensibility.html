Custom models in Microsoft Foundry

1 · Catalog split, deployment types & the model router

Who supports the model, where inference runs, and the one endpoint that routes between models.

Sold directly by Azure (“Azure Direct”)

  • Microsoft-hosted · Azure SLAs · Microsoft Product Terms
  • All Azure OpenAI + selected DeepSeek / Meta / Mistral / xAI / Cohere
  • PTU fungible across Azure Direct families

Partner & community

  • Provider-supported · “Non-Microsoft Product” terms
  • Most of the catalog — incl. Anthropic Claude, Hugging Face
  • Serverless/standard or managed compute
Platform shapeFoundry projects (children of a Foundry resource, inherit governance) vs classic hub-based projects; a standalone Azure OpenAI resource upgrades in place (keeps endpoint/keys); the platform itself is free — pay per deployment/usage.
Deployment typesGlobal (any region, default, highest quota) · Data Zone (US/EU/APAC) · Regional (strictest residency, lower quota) · Developer (no SLA — cheap fine-tune evaluation); data at rest never moves — only processing location changes.
Billing flavorsStandard (tokens) · Provisioned (PTU, reserved throughput) · Batch ≈ 50% off within 24 h; spillover routes 429-overflow from provisioned → standard (Azure OpenAI models only).
Model selectioncriteria: task fit, modality, context window, tool-calling, latency, cost, region, fine-tunability, residency, safety, license; leaderboards rank by quality · safety · cost · throughput to shortlist before your own evals.
Model routerone endpoint routes each prompt to the best model; modes Balanced / Cost / Quality; restrict with a model subset (doubles as failover, include ≥2); context window = the smallest underlying model; Claude targets deploy separately first.

2 · Fine-tuning — techniques, tiers & the customization ladder

Prompt engineering first, RAG for knowledge, fine-tuning for behavior. Then know the four techniques.

SFTsupervised fine-tuning on labeled input/output pairs — broadest support; Azure uses LoRA under the hood.
DPOpreferred vs non-preferred response pairs, no reward model; stackable after SFT.
RFTgrader/reward-based for reasoning models (o4-mini) — preview.
Distillationtrain a small model from a big model's stored completions.
Training tiers & cost trapStandard (in-region) · Global (cheaper/faster, no residency) · Developer (preemptible, no SLA); costs = training + hourly hosting while deployed, even idle + inference → evaluate on a Developer deployment, delete unused deployments.
Fine-tuning is for behaviorformat, tone, terminology, tool-calling accuracy, shorter prompts — not fresh facts; combine RAG + fine-tuning when both apply.

3 · Six paths to land a custom model in a business solution

Direct exam bullet: “design AI solutions by using custom models in Microsoft Foundry.” Know every landing pad.

Path 1 — fine-tune & deployfine-tune in Foundry → Developer tier for evaluation → Standard/Global for production.
Path 2 — managed computefor open/custom-weight models that need dedicated hosting.
Path 3 — behind an agentback a Foundry agent with the fine-tuned deployment (or the model router).
Path 4 — into Power Platformthe Azure AI Foundry connector exposes a chat-completions deployment to prompts in Copilot Studio / Power Apps / Power Automate — DLP-governable; the bridge that gives low-code makers a fine-tuned model.
Path 5 — customized Foundry Toolscustom speech/voice, CLU, custom Document Intelligence, Custom Vision, Custom Translator — consumed via endpoints or as agent tools (“AI Service fine-tuning” in the portal).
Path 6 — edge & on-deviceFoundry Local · Phi Silica on Copilot+ PC NPUs with LoRA adapters · App Service SLM sidecar container (no external calls, fixed cost).
When to go custom at allonly when unique domain data gives competitive advantage, prebuilt models miss accuracy/compliance, and the org has Level 3–4 AI maturity — otherwise prebuilt or fine-tuned foundation models.
💡 Hook: “must run offline / on-device · data can't leave · fixed cost · narrow task” → fine-tuned SLM (Phi) · “wide-ranging reasoning over changing knowledge” → LLM + RAG.

5 · Agent builder (lite) vs full Copilot Studio

Same brand, different personas. One upgrade path connects them — and it only goes one way.

Copilot Studio agent builder (in M365 Copilot)

  • Information workers, natural-language authoring
  • Lightweight Q&A over SharePoint / Graph / web
  • Individual or small team scope
  • Governed in the M365 admin center

Full Copilot Studio

  • Makers; multi-step logic, 1,000+ connectors
  • Autonomous triggers, environments, ALM, RBAC, analytics
  • External publishing (websites, channels)
  • Governed in the Power Platform admin center
The upgrade path“Copy to Copilot Studio” — one-way, lite → full; there is no downgrade path back.
Governance splitagent builder agents are governed in the M365 admin center; full Studio agents in PPAC — different admin surfaces, different policies.

MCP & A2A in Copilot Studio

6 · MCP — architecture, rules & governance

Direct exam bullet: “agent extensibility with Model Context Protocol.” Host, client, server — then the Copilot Studio specifics.

Three rolesHost (the LLM app) · Client (1:1 connection inside the host) · Server (exposes capabilities) — JSON-RPC 2.0 underneath.
PrimitivesTools (model-invoked functions) · Resources (read-only context) · Prompts (templates). Copilot Studio supports tools + resources, not prompts — resources surface only as tool outputs.
Hard requirementsMCP needs generative orchestration; transport is Streamable HTTP only (SSE dead since Aug 2025).
The core valueone MCP server = one agent tool exposing many MCP tools; the server's tool list updates dynamically — centrally versioned tool sets shared by many agents whose upstream APIs change often.
Three ways inthe MCP onboarding wizard (auth: none, API key, OAuth 2.0 with dynamic discovery/DCR) · a custom connector · registering BYO servers via Agent 365 / M365 admin center.
Prebuilt Microsoft serversDataverse MCP (deletes require explicit user approval; PPAC allow-listing; honors user permissions) · Dynamics 365 ERP MCP · D365 Sales / Customer Service MCP · Power Apps MCP · Azure MCP · Microsoft Learn MCP.
GovernanceMCP rides on Power Platform connectors → DLP data policies apply; per-tool enable/disable via the Allow all toggle (off → new server tools default off); limits: can't enrich tool descriptions per agent, topics can't call MCP directly.
Security risks → mitigationstool poisoning, silent tool changes, injection via tool output, token theft → trusted providers only, Entra ID auth + network isolation, least-privilege RBAC, tool allow-lists, human-in-the-loop approval, audit logging.

7 · A2A & the who-talks-to-whom decision

Tools use MCP; foreign agents use A2A; teammates are child agents — no protocol at all.

A2A essentialsopen standard (Linux Foundation) for cross-framework agent-to-agent work: agent card at /.well-known/agent-card.json, messages, long-running tasks (with “input required”), artifacts, contextId for state.
Opacity is the pointthe remote agent's tools and reasoning are opaque — you delegate the outcome, it runs its own reasoning.
Microsoft supportCopilot Studio connects A2A agents (GA) as connected agents · Foundry publishes A2A endpoints (preview) · Agent Framework hosts/consumes natively · Entra Agent ID authorizes OAuth 2.0, MCP and A2A.
Integration mappingplain API → custom connector · shared evolving tool set → MCP · external framework agents → A2A · M365 Agents SDK agents → Activity Protocol · same team, co-located → child agent (less latency, no protocol).
💡 Hook: “partner company exposes their agent; ours must delegate” → A2A · “many of our agents need the same evolving tools” → MCP · “same team, same solution” → child agent — neither protocol.

Computer use

8 · Computer use — the loop, the machines, the guardrails

Direct exam bullet: “automate tasks in apps and websites by using Computer Use.” A last resort with a precise anatomy.

What it isa tool that drives websites and Windows desktop apps with a virtual mouse/keyboard, powered by computer-using agent (CUA) models; authored in natural language; requires generative orchestration.
Three places to runhosted browser (preview — prototyping only, not Entra-joined) · Cloud PC pool (preview — Entra-joined/Intune, autoscaling) · bring-your-own machine for production (Windows machine registered in Power Automate, “Enable for computer use”; concurrent runs queue).
ModelsOpenAI CUA (GA) and Anthropic Claude — Anthropic needs the admin's “external models” switch.
Controlsapp/website allow list · Enforce HTTPS · maker vs end-user credentials · human supervision (email a reviewer on suspicious instructions) · session replays and logs.
Billing5 credits/step (15 premium) — and it is excluded from M365 Copilot license inclusion; never free.
When to proposeonly when no API, connector or MCP integration exists (legacy/desktop-only UI). Stable UI + fixed rules + high volume → classic RPA instead; shifting UIs + judgment → computer use.

Agent behaviors — reasoning, voice & M365 surfaces

9 · Deep reasoning & the model pickers

Direct exam bullet: “design agent behaviors, including reasoning.” Know the keyword, the cost, and the switches.

Deep reasoning (preview)enables reasoning models (Azure OpenAI o3) for complex multistep analysis; requires generative orchestration + a settings toggle.
The reason keywordforce a reasoning step from instructions by writing the keyword reason.
Reasoning trade-offsbetter analysis but slower and pricier — premium AI-tools token meter plus the agent-action rate; US/EU regions, no residency commitments in preview.
Primary model pickerswap the agent's main model (tags Deep/Auto/General); optional external models (Anthropic, xAI, Mistral) sit behind an admin switch; prompts can separately BYO a Foundry model.

10 · Voice — basic IVR vs real-time speech-to-speech

Direct exam bullet: “including voice mode.” Two voice architectures, two billing tiers, one deployment door.

Basic voice agents

  • Classic orchestration + NLU/NLU+
  • Full IVR set: speech + DTMF, silence detection, barge-in, SSML, audio playback, latency messages
  • Deterministic, low-latency → regulated IVR flows

Real-time voice agents (preview)

  • Speech-to-speech on a real-time multimodal model
  • Natural interruption, mid-call language switching
  • Deployed through Dynamics 365 Contact Center telephony
  • Some triggers (unknown-intent fallback, disambiguation) unsupported
Basic voiceclassic orchestration + NLU(+) with the full IVR toolbox — pick for phone menus with DTMF and regulatory wording.
Real-time voicenatural speech-to-speech conversation, deployed via Contact Center telephony — pick for “natural conversation over the phone.”
Voice billing ladderper minute: classic 10 · generative 35 · premium 75 credits; bot-handled calls cap at 1 hour.

11 · Optimizing with agents in Microsoft 365 — Teams, SharePoint & publishing

Direct exam bullet: “optimize solution design by using agents in Microsoft 365, including Teams and SharePoint.”

SharePoint agentssite/library-scoped Q&A created by end users, permission-trimmed, one “approved” agent per site — the zero-build answer for “our team needs answers from this site.”
Teams agentsFacilitator (real-time collaborative meeting notes, in-meeting Q&A) and Channel Agent (deadlines, status summaries, task assignment) optimize collaboration without custom builds.
Publishing modelpublish once → all connected channels update; republish after every change; Teams + M365 Copilot channel requires Authenticate with Microsoft.
Channel catalogTeams/M365, SharePoint, demo website (stakeholder testing) vs custom website (embed), mobile SDK, WhatsApp/Facebook, D365 Customer Service / Contact Center, telephony (ACS), Azure Bot Service channels (Slack, Telegram, Twilio…).
Sharing & rolesco-author or view · Analytics Viewer for analytics-only · Bot Transcript Viewer for transcripts · Teams org-catalog publishing needs admin approval · PPAC sharing limits apply in managed environments.
