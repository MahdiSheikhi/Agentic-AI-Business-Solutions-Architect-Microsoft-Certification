Power Platform ALM foundations

1 · The solution highway — how anything reaches production

Solutions are the vehicle; environments are the road. Unmanaged only in dev — managed everywhere else.

  • Solutions are the ALM vehicle — containers for tables, apps, flows, agents, prompts and connectors moved dev → test → prod.
  • Every ALM environment needs Dataverse — no Dataverse, no solutions, no ALM.
  • Customize only in dev (unmanaged); deploy managed everywhere else.
  • You cannot export a managed solution — a classic trick answer.
  • Remove unmanaged layers in target environments so managed updates can flow.
  • Use a custom publisher with your own prefix — and in Copilot Studio set a preferred solution; never build in the Default solution.

2 · The two config carriers — variables and connection references

Nothing environment-specific is ever hardcoded. Two mechanisms carry configuration across the highway.

Environment variables

  • The solution ships the definition; each environment supplies its own value at deploy time — never hardcode secrets, URLs or config.

Connection references

  • Decouple connections (credentials) from flows; pre-populate them via a deployment settings file so pipelines run unattended.

3 · Pipeline rules — what moves, what doesn't, what's forbidden

Power Platform pipelines have five hard rules. Each one is a distractor in someone's exam question.

PayloadPipelines deploy solutions + configuration (connections, environment variables) — not table data.
Order is lawStages run in order — you can't bypass QA; deployed artifacts are tamper-proof.
Delegated deploymentsRun as service principals with approvals — makers don't need prod access.
No cross-tenantPipelines can't deploy across tenants — that's a job for Azure DevOps / GitHub.
Managed Environments onlyPipeline target environments must be Managed Environments.

4 · The tooling ladder — citizen to pro

Three rungs, increasing control. Pick by team maturity, not fashion.

  • The ladder: Power Platform pipelines (citizen-friendly) → Azure DevOps (Power Platform Build Tools, Dataverse Git integration) → GitHub Actions for Power Platform.

ALM for Copilot Studio agents, connectors & actions

5 · The six golden rules for agent ALM

Copilot Studio agents are solution-aware — so agent ALM is Power Platform ALM plus discipline.

Never customize outside devAll changes originate in the dev environment's unmanaged solution.
Always work in solutionsCopilot Studio embeds a solution explorer; set a preferred solution so new components land correctly.
Environment variables for settingsKnowledge-source URLs and configuration ride as variables — different values per environment.
Export / deploy managedManaged artifacts into test and prod, every time.
Automate source control + deploymentGit-backed, pipeline-driven — no manual exports as a process.
Minimum three environmentsDev / test / prod, each secured by Entra security groups.

6 · The NOT-solution-aware wall — reconfigure after every deployment

The single most-tested ALM fact set. These settings do not travel in the solution — someone must re-set them in the target.

  • App Insights settings — reconnect telemetry in each environment.
  • Manual authentication settings — re-enter OAuth config in the target.
  • Direct Line / web-channel security — re-enable secrets/tokens per environment.
  • Deployed channels — republish to Teams/web/etc. after each deployment.
  • Sharing (makers and viewers) — re-share in the target; permissions don't travel.
  • Also: exports drop topic comments.
  • A topic name containing a period blocks export — rename before shipping.
  • Don't delete agent subcomponents directly from a solution — remove them in the agent designer.

7 · Component collections & test gates

Reuse across agents, and let tests decide what ships.

Component collectionsReusable sets of topics / knowledge / tools shared across agents and exported between environments — modular multi-team development with independent release cadences.
Test gates in pipelinesRun Copilot Agent Kit test suites inside Power Platform pipelines — promotion is gated on passing tests, not on optimism.

ALM for Foundry agents & custom models

8 · Foundry Agent Service ALM — config vs containers

Two agent shapes, one CI/CD loop. The evaluate→gate→deploy→monitor cycle is the GenAIOps answer the exam wants.

  • Prompt agents are configuration — author them code-first via SDK/REST inside the deployment pipeline for version control, code review, automated rollout.
  • Hosted agents ship as container images with agent versions (name:version) and managed endpoints.
  • azd pipeline config bootstraps CI/CD — it generates GitHub Actions / Azure DevOps workflows (provision + deploy + smoke test).
  • The AI agent evaluation GitHub Action (preview) runs offline evaluation pre-release — test queries + catalog evaluators, comparing versions against a baseline with statistical significance.
  • The full loop: evaluate → gate → deploy → monitor, paired with continuous evaluation in production.

9 · The Ops stack — four methodologies, one solution

Custom-model ALM stacks four disciplines. GenAIOps is MLOps with the effort moved.

  • The stack: DevOps (app) + DataOps (pipelines) + MLOps (classic models) + GenAIOps (generative).
  • GenAIOps shifts the effort — from training to model selection, prompts, RAG, fine-tuning; it adds prompt lifecycle, retrieval augmentation, output safety, and token-cost governance.
  • MLOps maturity model: L0 (manual) → L4 (fully automated) — questions describe a team and ask you to place it.

10 · The custom-model lifecycle practices

Five practices turn a trained model into a governed asset.

Experiment trackingMLflow logs runs, parameters, metrics — experiments become comparable and reproducible.
Version everythingVersion the training code and the models — a model without its code lineage is unreproducible.
Automated pipelinesTraining and deployment run as pipelines, not notebooks-by-hand.
Progressive exposureCanary / blue-green deployments — new model versions earn traffic gradually.
Drift & decay monitoringWatch for data/model drift with automated retraining triggers.

ALM for data & Dynamics 365 AI

11 · ALM for data used by AI — version the grounding pipeline like code

A direct exam bullet with a memorable framing: your index deserves the same rigor as your code.

The five test typesIngestion tests · index schema backward-compatibility tests · freshness/staleness checks · load tests · security-trimming tests.
Side-by-side index deploysDeploy index changes side-by-side, coordinated with code — swap when both are proven, roll back cleanly.
Secure lower environmentsKeep dev/test data as secure as prod — grounding data is still sensitive data in a sandbox.
Parameterize sourcesCarry data-source parameters via environment variables, so each environment grounds on its own copies.

12 · D365 finance & supply chain — three switches, then Power Platform ALM

F&SCM Copilot has prerequisites at three levels; after that, agent customizations follow the standard solution highway.

  • The three prerequisites: Power Platform Integration per environment + feature-management flags + the tenant “Publish bots with AI features” switch.
  • Extensions run through Copilot Studio + Dataverse — so F&SCM agent customizations follow Power Platform solution ALM, not X++ ALM.
  • The ERP MCP server and in-app agent management are production-ready preview — current-state awareness the exam may probe.

13 · D365 customer experience & service — managed solutions and the linkage decision

CE-side agent ALM is standard solution ALM plus two service-specific facts and one billing nuance.

Move as managed solutionsCopilot Studio agents integrated with Customer Service travel the same dev→test→prod highway as anything else.
The known issue & fix“Can't determine Dynamics 365 capabilities” → re-export the latest solution and remove unmanaged layers in the target.
Copilot experience profilesAre solution-aware — profile config travels with the solution.
The billing nuanceSales/Service agent capabilities bill Copilot Studio credits on the linked environment — environment linkage is itself an ALM/environment design decision.
