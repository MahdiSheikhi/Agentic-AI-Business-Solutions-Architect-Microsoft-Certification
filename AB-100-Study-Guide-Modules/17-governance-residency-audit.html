Governance at Scale, Data Residency & Audit Trails

Governing agents — who controls what

Control plane Governs
Power Platform admin center (PPAC) Copilot Studio agents: DLP data policies (enforced for all tenants since early 2025), environments & groups, sharing limits, quarantine agents (work only in test canvas), tenant switch to disable publishing of generative-AI agents, generative-AI consent settings, environment routing, Power Platform inventory (all agents/apps/flows incl. drafts), credit capacity + per-agent caps
Microsoft 365 admin center — Copilot Control System (CCS) Agent inventory of everything available to users (first-party + ISV + shared): block/unblock tenant-wide, approve/reject Copilot Studio submissions, deploy/assign to users/groups, pin; agent sharing controls (all users / specific / none)
Microsoft Agent 365 Cross-origin control plane — Observe, Govern, Secure: unified agent registry (incl. shadow + third-party agents; registry sync pulls from Bedrock, Vertex AI, Salesforce, Databricks), Agent Map (agent↔person↔data connections), per-instance blocking, Agent Tools registry for MCP governance; integrates Entra Agent ID, Purview, Defender
Entra admin center Agent identities: Conditional Access, risky-agent detection, lifecycle workflows, access reviews

DLP mechanics (know cold): connectors classified Business / Non-business / Blocked; business + non-business can't combine in one resource; Microsoft-owned standard connectors can't be blocked; tenant + environment policies combine with the most restrictive winning. Granular: connector action control, endpoint filtering (HTTP, HTTP+Entra, Webhook, SQL Server, Azure Blob, SMTP), custom-connector URL patterns, and virtual connectors that switch Copilot Studio capabilities (channels, knowledge sources, auth requirements, App Insights) on/off. DLP violations block publish (DlpViolationError) and runtime (DataLossPreventionViolation).

Third-party AI consideration: block/isolate third-party AI connectors via DLP (especially in the default environment); Purview endpoint/browser DLP stops sensitive data being pasted into consumer AI apps; the governed alternative is bring-your-own-model (Foundry catalog models inside prompts / Copilot Studio model picker) so frontier models stay under tenant governance.

Data residency & movement (validate compliance — direct exam bullet)

  • Copilot Studio: data stored in the chosen environment geography; EU Data Boundary applies when the tenant billing address is EU/EFTA and environments are in EUDB geos. Caveat: M365-powered features store conversation history in the end user's Exchange mailbox region.
  • PPAC cross-geo toggles (per environment, Generative AI settings): "Move data across regions" (needed where Azure OpenAI isn't local; not shown for US/Gov; movement only on no-local-model, capacity overflow or reliability; irreversible for past data) · Bing search consent · Microsoft 365 services consent · EUDB "flex routing during peak load" (uncheck to keep all inference in-boundary).
  • Foundry/Azure OpenAI deployment types: at-rest data always stays in the resource's geography; Global = processing anywhere · Data Zone = processing within US / EU / APAC · Regional = processing in the resource's region. Fine-tuning training data stored in-region (Global training tier trades residency for cost).
  • M365 Copilot: interaction content + semantic index in the tenant's Local Region Geography; Advanced Data Residency / Multi-Geo extend commitments (storage follows Preferred Data Location).

Access controls on grounding data & tuning (direct exam bullet)

  • M365 Copilot/agents grant no new permissions — retrieval respects the current user's Graph/SharePoint/Teams ACLs; sensitivity labels add VIEW+EXTRACT gating; S/MIME and password-protected files excluded.
  • Dataverse: security roles (row + column) enforced for agents and the Dataverse MCP server; connector-file knowledge does live permission checks at query time; transcripts need Bot Transcript Viewer.
  • Azure AI Search security trimming: the GA pattern = a filterable group_ids-style field + search.in() security filter at query time; preview options: native ACL/RBAC scopes (ADLS Gen2/Blob), SharePoint ACL sync, Purview label enforcement.
  • Tuning: Foundry project RBAC separates who can see/train vs deploy; BYO storage + CMK for the strictest control; DLP knowledge-source and endpoint controls protect Copilot Studio grounding.

Audit trails (direct exam bullet)

  • Microsoft Purview Audit: CopilotInteraction events for M365 Copilot, Copilot Studio agents, Security Copilot, Fabric agents… Properties: AccessedResources (IDs, SiteUrl, SensitivityLabelId, policy blocks, XPIADetected), AgentId/Name/Version, AppHost. Copilot Studio admin + maker activity is logged by default and can't be disabled; full prompt/response text surfaces via DSPM for AI Activity Explorer.
  • DSPM for AI: posture dashboard — AI activity by category (Copilot experiences & agents / enterprise AI apps / other AI apps), one-click protection policies, weekly data risk assessments (top-100 SharePoint sites). Compliance companions: Communication Compliance, eDiscovery (AI interactions live in user mailboxes), retention policies, Compliance Manager AI assessments.
  • Dataverse auditing: table/column-level create/update/delete + access logging.
  • Model/data change tracking (Foundry): control-plane operations in Azure Activity Log; diagnostic settings + App Insights (tokens, latency, traces); evaluation runs and red-team scans logged in the portal; fine-tune jobs/files/checkpoints enumerable; deployments are versioned.

⚡ Exam focus: match the requirement to the right control plane — "block a rogue agent your admin found" → CCS block (M365 agents) or PPAC quarantine (Copilot Studio); "prove who the agent touched and whether a prompt-injection fired" → Purview CopilotInteraction (+XPIADetected); "EU processing only" → Data Zone EU + EUDB flex-routing unchecked; "stop makers calling ChatGPT connectors" → DLP block + endpoint filtering.
