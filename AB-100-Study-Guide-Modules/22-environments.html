Environment Strategy for AI-Powered Solutions

Environment types (Power Platform)

Type Notes
Default One per tenant; not for long-term solutions — route makers out of it
Production Business-critical work; premium governance via Managed Environments
Sandbox Copy/reset; test & ALM stages
Developer Personal (up to 3/maker); no tenant capacity; auto-cleanup after 90 days idle
Trial / Dataverse for Teams / Support Time-boxed / Teams-scoped / Microsoft-created

Managed Environments & environment groups

Managed Environments unlock the governance the exam assumes: sharing limits, usage insights, solution-checker enforcement, maker welcome content, IP firewall, CMK, Customer Lockbox, extended backups, pipelines (targets must be managed), environment routing. Users need premium licenses.

Environment groups apply published rules at scale (sharing controls, usage insights, solution checker, backup retention, AI-related settings); an environment must be managed to join, belongs to one group, groups can't nest but can span regions; group rules lock individual environment settings. Group by lifecycle (dev/test/prod), department, project or geography. Default environment routing sends new makers to personal managed developer environments automatically.

Zoned governance for Copilot Studio (the documented model)

  • Tenant level: data policies across all environments; block unauthenticated agents, channels, knowledge sources, connectors; tenant-wide switch to disable publishing of generative-AI agents; disable cross-geo data movement.
  • Environment level: scoped data policies; allow/block public data sources (e.g. web search); enable/disable gen-AI features; network isolation (VNet, IP firewall); credit allocation; transcript controls.
  • Agent level: orchestration mode, knowledge toggles, authentication choice, web-channel security, per-agent credit caps.

Recommended tiering: strict DLP in the default + personal dev environments → controlled managed dev/test → production-grade managed prod. Regions matter twice: data residency and generative-AI regional availability (where local Azure OpenAI capacity is missing, the admin must consent to cross-geo movement per environment — or the features stay off).

Third-party AI solutions in the environment strategy (exam bullet)

  • DLP first: classify connectors Business/Non-business/Blocked; block third-party AI connectors where policy demands; use endpoint filtering and custom-connector URL patterns to stop home-grown calls to external AI APIs; most-restrictive-wins when tenant + environment policies combine.
  • Purview DLP (endpoint/browser/network) blocks pasting/uploading sensitive data into consumer AI apps — governance beyond Power Platform.
  • The governed alternative — bring your own model: connect Foundry-catalog models (1,800+, incl. non-Microsoft frontier models) into Copilot Studio prompts / primary model via the Azure AI Foundry connector, which is itself DLP-governable. Third-party agents integrate via A2A (opaque delegation) under Entra Agent ID governance and appear in the Agent 365 registry (registry sync covers Bedrock, Vertex AI, Salesforce, Databricks).
  • Environment placement for third-party AI experiments: isolated sandbox environments with tight DLP, never the default environment.

Capacity & credits in the environment design

Prepaid Copilot Credit packs are allocated to environments in PPAC; PAYG links environments to Azure subscriptions (production/sandbox only); per-agent caps; agent flows block instantly on exhausted prepaid capacity — so environment-level capacity planning is part of the architecture, not an afterthought.

⚡ Exam focus: scenario says "makers keep building agents in the default environment with consumer connectors" → answer combines environment routing + managed environments + DLP (block/endpoint filtering) + environment groups rules. "Third-party LLM required" → BYO model via the Foundry connector under DLP, not an unmanaged consumer connector. "Feature unavailable in our region" → cross-geo consent toggle (or accept unavailability).
