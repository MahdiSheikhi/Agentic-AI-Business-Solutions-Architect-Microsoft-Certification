Microsoft Foundry II — Foundry Tools & Agent Service

Foundry Tools (formerly Azure AI services) — requirement → tool mapping

Direct exam bullet: "Propose Foundry Tools for a given requirement."

Requirement Propose
Transcribe calls, voice interfaces, brand voice (TTS) Speech (custom speech; custom neural voice is Limited Access)
Real-time / document translation, industry terminology Translator (+ Custom Translator)
Sentiment, NER, PII redaction, summarization, question answering Language (CLU for intents/entities — retires 2029; also exposed as an MCP server)
Extract fields from invoices/receipts/IDs/forms Document Intelligence (prebuilt + custom models)
Schema-defined extraction across documents + audio + video, RAG-ready output Content Understanding
Image tagging, OCR, objects/faces Vision (+ Custom Vision for your own labels)
Enterprise search / RAG index Azure AI Search (vectorization, hybrid search, security trimming)
Filter harmful content, detect jailbreaks Content Safety (Prompt Shields, groundedness detection…)

Retired — do not propose: Anomaly Detector, Content Moderator, LUIS, Metrics Advisor, Personalizer, QnA Maker. Most tools offer containers for on-premises use. Custom-model capable: custom speech/voice, CLU, custom Document Intelligence, Custom Vision, Custom Translator (surfaced as AI Service fine-tuning in the portal).

Foundry Agent Service

The managed agent runtime — now GA on the Responses API ("Agents v2"; Foundry classic agents retire March 31, 2027, and the older Azure OpenAI Assistants API retires August 26, 2026). New terminology: Conversations, Items, Responses, Agent Versions (classic: Threads/Messages/Runs).

Two agent types:

  • Prompt agents — configuration-only (model + instructions + tools); portal or code-first; fully managed, no container cost. Fast start, production-ready without custom orchestration.
  • Hosted agents — your code (Agent Framework, LangGraph, OpenAI/Anthropic SDKs…) packaged as a container/zip; Foundry runs it with managed endpoints, autoscaling, dedicated Entra agent identity, session state, observability, agent versions (name:version).

Tools: built-in — web search / Grounding with Bing (data leaves the Azure compliance boundary; separate terms), File Search (vector-store RAG), Code Interpreter (sandboxed Python, billed per session), Azure AI Search, Azure Functions, Fabric data agent, SharePoint, Deep Research (preview), browser automation/computer use (preview), image generation, Foundry IQ knowledge. Custom — MCP tools (with allowed_tools and require_approval defaulting to always), OpenAPI tools, A2A tool (preview). Auth to tools: key, Entra (agent identity / project managed identity), or OAuth on-behalf-of passthrough. tool_choice forces a specific tool deterministically.

Multi-agent: classic connected agents superseded by workflows — visual designer + YAML, versioned multi-agent orchestration; wire-compatible with Agent Framework. Publishing: versioned agent applications with stable endpoints, distributable to M365 Copilot, Teams, BizChat, and the Entra Agent Registry; A2A endpoints let other agents discover/call them.

Pricing: no separate service fee — model tokens + tool charges (Code Interpreter/session, File Search storage/GB, Bing per call) + container compute for hosted agents.

Custom models into business solutions

Paths to land a custom/fine-tuned model in the solution (exam bullet "design AI solutions by using custom models in Microsoft Foundry"):

  1. Fine-tune in Foundry → deploy (Developer tier for eval → Standard/Global for prod).
  2. Managed compute for open/custom-weight models.
  3. Back a Foundry agent with the fine-tuned deployment (or model router).
  4. Bring your own model into Power Platform: the Azure AI Foundry models for prompts feature connects a Foundry chat-completions deployment to prompts in Copilot Studio / Power Apps / Power Automate via the Azure AI Foundry connector (DLP-governable). GPT-5-family/Responses-endpoint models not supported there yet.
  5. Customized Foundry Tools models via their endpoints, or as agent tools.
  6. Edge/on-device: Foundry Local, Phi Silica (+LoRA), App Service SLM sidecar.

⚡ Exam focus: Grounding with Bing = the one built-in tool whose calls leave the Azure compliance boundary — flag it in compliance scenarios. Hosted agents = the answer when "our devs built the agent in LangGraph and we need managed hosting, identity and observability." The Azure AI Foundry connector = the bridge that makes a fine-tuned model usable by low-code makers under DLP governance.
