Domain 3 · Deploy

Security for Agents, Models & Data Workflows

Agent identity — Microsoft Entra Agent ID (preview)

First-class directory identities for agents:

  • Agent identity blueprint = the template/class; holds the credentials (federated identity credentials, certificates, secrets); can be multitenant.
  • Agent identity = a special service principal with no credentials of its own; single-tenant. Foundry auto-provisions a default blueprint + shared project identity; publishing an agent creates a dedicated identity.
  • Token patterns: attended agents act via OAuth 2.0 on-behalf-of (subject = user, actor = agent → agent limited to the user's permissions); unattended/autonomous agents use client credentials (agent's own RBAC).
  • Entra features extended to agents: Conditional Access (P1), Identity Protection/risky agents (P2), Identity Governance (ownership/sponsorship, lifecycle workflows, access reviews); all agent sign-ins logged. Inventory spans Foundry + Copilot Studio + third-party agents (Agent 365 registry; external agents onboard via Entra Auth SDK sidecar or workload identity federation).

Authentication layers (design every one)

  1. User → agent: No auth (public; DLP can forbid) · Authenticate with Microsoft (Teams/M365 channel requirement) · Manual OAuth2 (federated credentials recommended over secrets; exposes User.AccessToken).
  2. Agent → tools/data: delegated (user) connections / OBO when actions must respect each user's permissions; application/managed identity for autonomous agents — scoped to minimal RBAC. Foundry tool auth: key, Entra (agent/project managed identity), OAuth identity passthrough.
  3. Least privilege everywhere: tool allow-lists (allowed_tools), short-lived just-in-time privileges, Dataverse security roles, human-in-the-loop for consequential actions.

Network security

  • Copilot Studio: Virtual Network support for connectors, IP firewall, Continuous Access Evaluation, Direct Line channel security, customer-managed keys (CMK), Customer Lockbox.
  • Foundry: Basic setup (Microsoft-managed) · Standard (BYO Storage/AI Search/Cosmos DB — agent data at rest in your tenant; CMK) · Standard + private networking (BYO VNet injection, private endpoints, no public egress). Private MCP endpoints supported.

Model security

  • Keyless auth (Entra) preferred; disable local/key auth (disableLocalAuth=true) via policy; data-plane vs control-plane RBAC are separate (control-plane Owner ≠ inference rights).
  • Foundry RBAC roles (renamed): Foundry User / Foundry Owner / Foundry Account Owner / Foundry Project Manager. Fine-tuning needs both planes: Foundry User covers training (data plane), but deploying a fine-tuned model is a control-plane action — Foundry Owner covers both, or pair Foundry User with Foundry Account Owner.
  • Training data & tuned models: stored in-region, logically isolated, deletable; customer's fine-tuned models are exclusive to the customer and never train foundation models; AES-256 at rest.
  • CMK: Key Vault same region/tenant, soft-delete + purge protection; revoking CMK blocks new training/deployments — but already-deployed fine-tuned models keep serving until deleted (a classic exam nuance).
  • Content filters configured at the resource level, associated per deployment; disabling filters ("no filters"/annotate-only) requires Limited Access approval.

The threat → mitigation map (memorize)

Threat Mitigations
Direct prompt injection / jailbreak (UPIA) Prompt Shields for user prompts (on by default in content filtering), safety system messages, input validation, blocklists, red-team testing (PyRIT)
Indirect / cross-prompt injection (XPIA) — malicious instructions inside emails/docs/webpages the agent processes Prompt Shields for documents + Spotlighting (preview — tags untrusted content as low-trust), isolate untrusted content, least privilege on tools, human-in-the-loop for risky actions, monitoring; Purview audit exposes an XPIADetected flag
Data leakage / oversharing Purview sensitivity labels (encrypted content honored via VIEW + EXTRACT rights), DLP (M365 + Endpoint DLP blocks pasting into consumer AI), DSPM for AI oversharing assessments, SharePoint Advanced Management, Restricted Content Discovery; (Restricted SharePoint Search is retiring — new enablement blocked from July 31, 2026)
Hallucination / ungrounded output Groundedness detection (preview, with correction), groundedness evaluators, RAG grounding, ungrounded-attributes evaluator
Harmful content Content filters — 4 categories (hate, sexual, violence, self-harm) × severity safe/low/medium/high, default threshold medium, separately configurable for prompts vs completions
IP/copyright reproduction Protected material detection (text + code; on by default; ties to the Customer Copyright Commitment)
Runtime attacks on AI workloads (credential theft, wallet abuse, poisoning) Microsoft Defender for Cloud AI threat protection (GA) — runtime alerts using Prompt Shields signals + threat intel, Defender XDR integration; AI Security Posture Management discovers the AI bill-of-materials and attack paths
Excessive agency / tool misuse Task adherence checks, prohibited-actions & sensitive-data-leakage evaluators, tool allow-lists, approvals

Reference frameworks docs cite: OWASP Top 10 for LLM applications, MITRE ATLAS.

⚡ Exam focus: UPIA vs XPIA distinction (user-typed vs content-embedded) and their different shields; the OBO-vs-client-credentials identity split; "already-deployed model keeps serving after CMK revocation"; Foundry User trains but Foundry Owner deploys.
