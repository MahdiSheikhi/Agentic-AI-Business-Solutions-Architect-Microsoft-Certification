Microsoft Foundry I — Models, Deployments, Router, Fine-Tuning

Platform shape

Microsoft Foundry (formerly Azure AI Foundry / Azure AI Studio) is Azure's unified AI PaaS: models + agents + tools with built-in tracing, evaluations, RBAC, networking, policy. Foundry projects (new; child resources of a Foundry resource; inherit governance) vs hub-based projects (classic; some partner/community models still need them). Azure OpenAI is now "Azure OpenAI in Foundry Models"; a standalone Azure OpenAI resource can be upgraded in place to a Foundry resource (keeps endpoint/keys). Platform itself is free — you pay per deployment/usage.

Foundry Models — the catalog

1,900+ models from Microsoft (Phi), OpenAI, Anthropic (Claude), Meta (Llama), Mistral, xAI (Grok), DeepSeek, Cohere, NVIDIA, Hugging Face. Two categories:

Sold directly by Azure ("Azure Direct") Partner & community
Hosting/support Microsoft-hosted, Azure SLAs, Microsoft Product Terms Provider-supported, "Non-Microsoft Product" terms
Includes All Azure OpenAI models + selected DeepSeek/Meta/Mistral/xAI/Cohere Most of the catalog, incl. Anthropic Claude, Hugging Face
PTU Fungible provisioned throughput across Azure Direct families Serverless/standard or managed compute

Selection criteria: task fit, modality, context window, tool-calling, latency/throughput, cost, region availability, fine-tunability, data residency, safety, license/support. Model leaderboards (preview) rank by quality index, safety, cost, throughput with trade-off charts and scenario boards — the documented way to shortlist candidates before benchmarking on your own evals.

Deployment types (a favorite exam topic)

For all types, data at rest stays in the resource's geography — the deployment type changes only where inference is processed:

Type Processing location Billing Use when
Global Standard / Provisioned / Batch Any Azure region Tokens / PTU / Batch = ~50% off, 24 h Default start; highest quota
Data Zone Standard / Provisioned / Batch Within US / EU / APAC zone same trio Zone-level residency compliance
Regional (Standard / Provisioned) The resource's region only Tokens / PTU Strictest residency; lower quota
Developer Any Tokens, no SLA Evaluating fine-tuned models cheaply
Instant (preview) Any Tokens No-deployment prototyping

PTU = reserved capacity for predictable throughput; reservations shareable across Azure Direct model families. Spillover routes 429-overflow from a provisioned deployment to a standard one (Azure OpenAI models only). Bursty traffic → Standard; consistent high volume/low latency variance → Provisioned; non-urgent bulk → Batch.

Model router (direct exam bullet)

A deployable trained model that analyzes each prompt (complexity, reasoning need, tools, history) and routes to the most suitable underlying model behind one endpoint:

  • Modes: Balanced (default — cheapest within ~1–2% of best quality) · Cost · Quality.
  • Model subset: restrict routing to an approved list (compliance/cost; doubles as the automatic failover set — include ≥2 models). Claude targets must be deployed separately first.
  • Limits: effective context window = smallest underlying model; Global/Data Zone Standard; resource regions East US 2, Sweden Central; text+image in, routing decided on text.
  • Propose when: mixed simple/complex traffic, cost optimization, one endpoint simplifying lifecycle, heterogeneous agent turns (per-turn routing in Foundry Agent Service). Don't when you need the same deterministic model every request. Documented hybrid: router as default + direct deployments for specialized workloads.

Fine-tuning & customization

Technique What Status/notes
SFT (supervised) Labeled input/output pairs; Azure uses LoRA under the hood Broadest support
DPO Preferred vs non-preferred response pairs (no reward model) Stackable after SFT
RFT Grader/reward-based, for reasoning models (o4-mini) Preview
Distillation Train a small model from a big model's stored completions Stored completions preview

Training tiers: Standard (in-region residency) · Global (cheaper/faster, no residency) · Developer (preemptible, no SLA). Costs: training + hourly hosting while deployed (even idle) + inference → evaluate on a Developer deployment, delete unused deployments.

Prompt engineering vs RAG vs fine-tuning: prompt engineering first (cheapest); RAG for dynamic/proprietary knowledge with citations; fine-tuning for stable behavior (format, tone, terminology, tool-calling accuracy, shorter prompts → token/latency savings) — not for fresh facts. Combine RAG + fine-tuning when both apply.

⚡ Exam focus: "EU processing boundary" → Data Zone; "strict single-country residency" → Regional; "overnight bulk summarization at lowest cost" → Global Batch; "predictable high-volume latency-sensitive" → Provisioned + spillover; "mixed workloads, minimize cost per answer" → model router (Balanced/Cost); "answers must cite fresh policy docs" → RAG, not fine-tuning.
