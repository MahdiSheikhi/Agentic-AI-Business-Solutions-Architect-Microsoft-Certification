1A · Analyze requirements for AI-powered business solutions

1.1Assess the use of agents in task automation, data analytics, and decision-making▾

An agent uses a generative AI model as a reasoning engine to interpret input, decide, and act. Your first architectural decision is whether the use case needs an agent at all.

  • Five core agent components (CAF): model (reasoning engine) · instructions (scope/boundaries) · knowledge/retrieval (grounding) · tools/actions · memory.
  • Agent spectrum: Retrieval agents (find + synthesize information) → Task agents (perform specific actions in defined workflows) → Autonomous agents (run multi-step processes from triggers with minimal oversight — most governance needed).
  • Agent vs classic RAG: RAG is a deterministic retrieval pipeline; an agent dynamically decides which knowledge/tools to use each step — powerful but nondeterministic, so it needs testing and governance.
  • Use-case classification (CAF Strategy): business problem → use-case statement → individual productivity (→ Microsoft 365 Copilot) vs business automation (→ agents) → generative vs nongenerative AI (prediction/anomaly detection = nongenerative).
  • Benefits vocabulary for justifying agents: efficiency (automate repetitive work), speed (faster decisions), scalability (handle demand spikes).

Choose what, when

Structured, predictable, repeatable stepsPower Automate / code / nongenerative model — not an agent (cheaper, deterministic)

1.3Organize business solution data to be available for other AI systems▾

Make data “AI-ready” once, so M365 Copilot, Copilot Studio, Foundry agents, and Fabric data agents can all consume it with governance intact.

  • Unified data platform (CAF): Microsoft Fabric OneLake as central lake; domains publish governed data products; agents consume via Fabric IQ, Foundry IQ (shared knowledge layer in Foundry), and Copilot Studio.
  • Dataverse = structured business data platform under Dynamics 365/Power Platform; built-in RAG as a knowledge source (requires Dataverse search). Dataverse Link to Fabric exposes D365 data in OneLake in near real time with no ETL/copy.
  • Microsoft 365 Copilot connectors (formerly Graph connectors) put external/LOB content into the Graph semantic index with content + metadata + ACLs (permission-trimmed). Two models: synced (ingest + semantic indexing, continuous sync) vs federated GA Jun 2026 (MCP-based, query-time fetch, data stays in source — for dynamic/regulated data; read-only, no semantic index).
  • Copilot connectors vs Power Platform connectors as knowledge: Copilot connectors = indexed, searchable, citation-friendly; Power Platform connectors = live API calls for real-time facts/transactions.
  • OneLake shortcuts/mirroring unify ADLS, S3, Google Cloud Storage, Dataverse without copying; Fabric data agents give conversational Q&A over OneLake and plug into Foundry, Copilot Studio, and M365 Copilot.
  • Governance throughout: Microsoft Purview (sensitivity labels, lineage, DSPM for AI) + RBAC.
Exam tip: “external system data searchable in M365 Copilot with citations” → synced Copilot connector. “Data must stay in the source system / always real-time” → federated connector (MCP) or Power Platform connector knowledge.

1.5Design the strategy for building AI and agents in business solutions▾

The strategy ladder: adopt SaaS first, extend next, then build low-code, then pro-code — with metrics and governance from day one.

  • CAF agent decision tree: first ask “does a prebuilt SaaS agent meet the functional requirements?” Yes → adopt it. No → build with Copilot Studio (low/no-code SaaS; retrieval, task and autonomous agents; built-in responsible AI) or Microsoft Foundry (PaaS; pro-code + declarative agents, workflows, your choice of models) or GPUs & containers (IaaS; full control for compliance-sensitive/custom — ACA serverless GPU for SLMs, AKS for LLMs).
  • Hybrid pattern: Copilot Studio low-code front end + bring-your-own Foundry model for advanced capability.
  • Validate with time-boxed experiments (1–2 weeks per option); assess single- vs multi-agent before scaling.
  • Metrics discipline: baseline KPIs before building; use business metrics as go/no-go decision gates; evaluate post-deployment; retire underperforming agents.
  • Model selection strategy at scale: match model capability to task complexity; smaller models for routine tasks; use catalogs/leaderboards; use Model Router for productivity/action agents; enforce quota + cost governance.
  • Think agentic-first with open standards: MCP for tools, A2A for agent-to-agent.
1.6Design a multi-agent solution using Microsoft 365 Copilot, Copilot Studio, and Microsoft Foundry▾

Multi-agent systems split work across specialized agents. Start single-agent unless the use case crosses team/security boundaries or has known growth.

  • Copilot Studio options: Child agents — live inside one agent (same lifecycle/config), for logically grouping tools/instructions for a single team. Connected agents — separately published Copilot Studio agents in the same environment; routing driven by each agent’s name + description; can pass or withhold conversation history.
  • External agents connect via: A2A protocol, Microsoft Foundry agents Preview, Fabric data agents Preview, Microsoft 365 Agents SDK agents Preview.
  • When to split into connected agents: tool selection degrades above roughly 30–40 tools/topics; different owning teams; separate ALM/channels/model settings; reuse across multiple main agents.
  • Known limits: extra hops add latency; a main agent that has connected agents can’t itself be a connected agent; Fabric data agents don’t work when the main agent is deployed to M365 Copilot; citations may be lost across agent boundaries.
  • Foundry Agent Service: connected agents + multi-agent workflows (sequential logic, branching, state across agents); publishable to M365 and governed by Microsoft Agent 365 (cross-platform agent control plane).
  • Protocol choice: MCP = tools/resources; A2A = delegating tasks to external agents; plain connector/HTTP = simple APIs.
Exam tip: “single team, one use case, no separate deployment” → child agent. “Different owners, separate ALM, reused by several main agents” → connected agent.

1.8Define solution rules and constraints — Copilot Studio, Microsoft Foundry, Foundry Tools▾

“Rules and constraints” = instructions, guardrails, moderation levels, and admin policies bounding what an agent can say and do.

  • Copilot Studio content moderation is set at three levels: agent (Generative AI settings), topic (generative answers node — wins at runtime), and prompt tool. Scale Lowest → Highest; higher = fewer but safer answers; default is High.
  • Knowledge constraints: Allow ungrounded responses OFF = only answer from knowledge/tools (else fallback); Use general knowledge OFF = restrict to configured knowledge; web search toggle (Grounding with Bing); tenant graph grounding requires generative orchestration + Microsoft auth (10 credits for non-Copilot-licensed users).
  • Built-in protections: moderation checked twice (input + pre-response) covering jailbreak, prompt injection, exfiltration, copyright; citations/provenance checks; AI disclosure message; tenant admins can disable publishing generative-AI agents; DLP policies constrain connectors/knowledge/tools.
  • Foundry / Foundry Tools: Azure AI Content Safety = harm-category filters (hate, sexual, violence, self-harm) with severity thresholds, Prompt Shields, groundedness detection, protected material detection, blocklists. Filters are configured on model deployments; with model router the filter + rate limits apply once at router level.
  • Foundry Agent Service: per-agent instructions, tool allow-lists, standard setup for network isolation/BYO storage, Azure Policy for allowed models, Entra RBAC.
  • Agent Builder (lite) can only prioritize sources — it can’t fully block general knowledge; use Copilot Studio for strict control.

1.11Determine when custom AI models should be created▾

Escalate only as needed up the customization ladder — most needs stop at step 2.

  1. Prompt engineering — first, cheapest lever (in-context learning).
  2. RAG / grounding — add organizational knowledge at inference; covers most use cases per CAF (“buy and ground prebuilt models”).
  3. Fine-tuning (Foundry, LoRA-based) — needs hundreds–thousands of quality examples. Good for: consistent style/tone, exact output formats/schemas, better tool-calling, going beyond context-window limits, token savings/lower latency, and distillation (use a big model’s outputs to fine-tune a smaller, cheaper one). Costs: training + hourly hosting regardless of usage + re-tuning when base models change.
  4. Train your own model — Azure ML / Fabric for classic ML; AKS/ACA GPUs for BYO models.
  • AI Builder: prebuilt models (invoice/receipt/ID reading, sentiment, entity extraction, translation…) vs custom models (document processing, prediction, object detection, classification) when data is unique to the business; requires Dataverse; consumes Copilot Credits.
  • Well-Architected verdict: “usually, choosing a prebuilt model or managed service is a better option” — custom models demand data, skills, and ongoing MLOps funding.
Exam tip: “model answers are wrong about our products” → fix with RAG/grounding, not fine-tuning. “Model must always output our exact JSON schema / brand voice” → fine-tuning. “Predict a field from historical Dataverse data” → AI Builder prediction (nongenerative).
1.12Provide guidelines for creating a prompt library▾

A prompt library is a curated, shared, governed collection of reusable prompts — for consistency, quality, and faster adoption.

  • M365 Copilot Prompt Gallery: Microsoft-curated + user-saved (“Your Prompts”) + team-shared prompts; admins can track and export saved/liked/shared prompt data.
  • Copilot Studio / Power Platform prompt library (prompt builder): predesigned templates (extraction, transformation, generation) under Copilot Studio Tools → Prompt or Power Apps/Automate AI hub → Prompts; prompts support input variables (text, image, document), Dataverse grounding, model + temperature choice, JSON output.
  • Sharing & governance: new prompts are private by default; recipients of shared prompts can use but not edit/delete; prompts consume Copilot Credits and need Dataverse in the environment.
  • Library guidelines: standardize structure (instruction + context + inputs + output format); test before sharing; version and review centrally (CoE-style); update monthly from analytics themes (Microsoft’s agent-value guidance); include personalization tips.
1.13Develop the use cases for customized small language models (SLMs)▾

SLMs (e.g., Microsoft’s Phi family) trade breadth for speed, cost, privacy, and deployability — often fine-tuned for a narrow domain.

  • Advantages: cheaper/faster to train and run; small footprint (edge/on-device deploy); more interpretable; less prone to overfitting.
  • Use cases: narrow well-defined tasks (classification, summarization, entity extraction, simple Q&A); high-volume cost/latency-sensitive workloads; sensitive/regulated or offline scenarios needing local processing; edge devices.
  • Phi family: Phi-3 mini (3.8B)/small (7B), Phi-4 series (Phi-4-mini powers Edge’s on-device Prompt API); MIT-licensed; in the Foundry catalog and fine-tunable.
  • Deployment patterns: local SLM sidecar on App Service (full data privacy, predictable latency, fixed cost — healthcare/finance/legal); ACA serverless GPU; on-device.
  • Distillation transfers a large model’s task performance to an SLM for scale economics; CAF: “smaller optimized models handle routine tasks… reserve premium models for complex reasoning.”
Exam tip: “data can’t leave the device / offline clinic / sub-100ms latency / massive volume of one narrow task” → fine-tuned SLM. “Broad reasoning, creative multi-step orchestration” → LLM.
1.14Provide prompt engineering guidelines and techniques▾

Prompt engineering is the first optimization step, before RAG or fine-tuning. Know Microsoft’s named techniques.

  • Zero-shot (no examples) vs one-/few-shot (include input→output example pairs as extra conversation turns; pick varied examples; watch token cost).
  • System message / metaprompt: defines role, boundaries, tone, output format, safety rules; appears first; influences but doesn’t guarantee — layer with filters and evaluation; make fallback behavior explicit.
  • Named best practices (Foundry docs): Be specific · Be descriptive (use analogies) · Double down (repeat instructions before and after content) · Order matters (recency bias) · Give the model an “out” (e.g., respond “not found” when the answer isn’t present — reduces fabrication).
  • Other techniques: cues (seed the start of the desired output), primary/supporting content, chain-of-thought-style “break the task into steps,” temperature/top_p tuning (ignored by reasoning models).
  • M365 Copilot prompt “ingredients”: Goal, Context, Expectations, Source — the four elements taught across Copilot docs.
  • Copilot Studio specifics: agent/tool/topic descriptions drive orchestration — write them precisely; use prompt modification for generative answers; iterate in the test pane.
1.15Include the elements of the Microsoft AI Center of Excellence▾

An AI CoE is a cross-functional team driving AI strategy, standards, skills, and reusable assets — starting central, evolving to advisory.

  • Core responsibilities (CAF): define AI strategy (decision tree, responsible AI) · develop AI skills (assessments, learning pathways) · lead pilot projects · define + enforce AI standards (governance, security, data quality, model lifecycle, bias reviews, audits) · run intake & prioritization workflows (structured project requests, prioritized backlog) · develop reusable assets (templates, repos, checklists) · measure and report (adoption, compliance, cycle-time KPIs) · optionally manage AI services.

  • Skills the CoE builds: prompt engineering, agent evaluation/optimization, AI ethics & governance, data engineering for AI (RAG/vector indexes), AI security (injection/jailbreak defense).
  • Risk-tiered governance (agentic CoE): knowledge-only agents = light review; write-access agents = cross-disciplinary review; release gates + audit logs of who built/approved each agent.

1.18Create an ROI analysis for a proposed AI solution▾

A repeatable method: define value before building → baseline → project benefits and TCO → gate the pilot → measure actuals.

  1. Define value before you build: discovery questions, prioritization grid, baseline protocol (current handle time, volume, cost per transaction), licensing/consumption plan.
  2. Project benefits: hours saved × loaded cost, deflection %, revenue uplift. Worked example (docs): 14,400 weighted grounded sessions × 6 min ÷ 60 = 1,440 hrs/month × $72 = ~$103,680/month.
  3. Compute TCO: licenses + credits/consumption + build + run + change management.
  4. ROI / payback / NPV → set decision gates for the pilot.
  5. Measure post-deployment: Copilot Studio Analytics + Savings calculator + PPAC consumption reports; iterate or retire. Review at 90 days vs baseline, then scale.
  • Adoption levers that maximize ROI: role-based training (managers first), anchor each agent to a named high-volume workflow, visible executive usage, shared prompt libraries updated monthly, quarterly build→measure→scale rhythm.

1.20Implement a model router to intelligently route requests to the most suitable model▾

Model router (Microsoft Foundry) is a deployable “model” that is really a trained routing model: it analyzes each prompt and forwards it to the best underlying chat model — one endpoint, lower cost, preserved quality.

  • Single deployment, single API, single rate limit; the chosen underlying model is disclosed in the response’s model field; prompts aren’t stored for routing; honors data-zone boundaries.
  • Routing modes: Balanced (default; most cost-effective within ~1–2% of top quality) · Cost (cheapest, ~5–6% quality band) · Quality (best model regardless of cost). Mode changes take up to 5 minutes.
  • Model subsets: pick which models participate (cost/compliance control); newly released models are excluded by default; router recalibrates automatically.
  • GA version (2025-11-18) routes across GPT-5 series, gpt-4o/mini, Anthropic Claude (Claude models must be deployed separately first — others need no separate deployment), Grok, DeepSeek, Llama; supports tools/agents (usable in Foundry Agent Service).
  • Constraints: effective context window = the smallest underlying model’s; content filter + rate limits configured once at router level apply to all.

2.5Design task agents▾

Task (action) agents do things — they perform specific actions within defined workflows, beyond just answering.

  • Position on the spectrum: retrieval → task → autonomous. Task agents use knowledge tools plus action tools; moderate complexity; user-initiated (no event triggers).
  • Use cases: create a service ticket, update records, fill forms, trigger a process, system lookups with follow-up actions.
  • Anatomy in Copilot Studio: conversational agent + tools — connectors, agent flows, prompts, MCP servers, REST APIs, computer use.
  • Escalation logic: if it only answers questions → prompt-and-response agent (2.7); if it must run from events/schedules in the background → autonomous agent (2.6).
  • Platform fit: Copilot Studio (low-code) and Microsoft Foundry (pro-code) both support all three types.
2.6Design autonomous agents▾

Autonomous agents act without a user prompt: an event trigger fires, the agent reasons over instructions, and executes tools/topics in the background.

  • Event triggers: email arrives, SharePoint/OneDrive item created, Planner task completed, Dataverse row added/updated, Recurrence (schedule), Dynamics 365 F&O business events (e.g., “purchase order confirmed”). Trigger payload passes event data (@{triggerBody()}).
  • Requires generative orchestration. Triggers only fire after publishing. Available triggers are governed by DLP data policies.
  • Authentication: event triggers run with the agent maker’s (author) credentials — every tool must have working maker credentials. Consider data-exposure risk when sharing the agent; forcing end-user credentials breaks autonomous agents (no live user).
  • Monitoring: the Activity page / activity map shows what ran and why; analytics show run outcomes, trigger/tool/knowledge usage.
  • Billing: autonomous actions consume Copilot Credits (5 per action).
  • Guardrails: scoped permissions, explicit decision boundaries, auditable processes, careful trigger conditions (avoid over-firing), human-in-the-loop approvals via agent flows for risky actions; layer deep reasoning for complex decisions.
2.7Design prompt and response agents▾

2.10Design topics for Copilot Studio, including fallback▾
  • Topic anatomy: canvas of nodes (message, question, condition, tool/action, redirect, variables). Two kinds: system topics (built-in — can’t create or delete; can disable, customize, reset) and custom topics. Limit: 1,000 topics per agent.
  • System topics list: Conversation Start · Conversational boosting (generative answers when no topic matches) · End of Conversation · Escalate · Fallback · Multiple Topics Matched (“did you mean”) · On Error · Reset Conversation · Sign in.
  • Fallback: triggers On Unknown Intent; has no trigger phrases; default = ask to rephrase (max 2 attempts) then redirect to Escalate. Priority rule: Conversational boosting fires before Fallback. The UnrecognizedTriggerPhrase variable carries the raw utterance — pass it to flows/skills/external NLP.
  • Trigger phrases: 5–10 semantic examples per topic recommended (NLU matches meaning, not exact text).
  • Fallback design patterns: offload big single-turn Q&A sets to Azure AI Language question answering (incl. chitchat); mine fallback utterances for topic enrichment.
  • Cautions: generative orchestration + topics containing generative-answers nodes can produce duplicate answers; in Teams, Conversation Start runs once per user and there’s no default fallback (create one).

2.15Design agents and agent flows with Copilot Studio▾

2.16Design prompt actions in Copilot Studio▾

Prompts (AI Builder prompt builder) are reusable natural-language model instructions that agents call as tools, topics call as nodes, and flows call as actions.

2.20Design agent extensibility with Model Context Protocol (MCP) in Copilot Studio▾

MCP is an open protocol: define an integration once on an MCP server; every agent discovers its tools dynamically.

  • MCP servers expose Resources, Tools, Prompts — Copilot Studio supports tools and resources (resources only as tool outputs). Requires generative orchestration.
  • Server tool definitions are dynamically reflected — server updates propagate to all agents without republishing.
  • Transport: Streamable HTTP only — SSE is deprecated/unsupported (since Aug 2025). Auth: None, API key, OAuth 2.0.
  • Connect via the MCP onboarding wizard (recommended) or a custom connector; MCP rides Power Platform connector infrastructure → DLP policies apply. Tool selection: “Allow all” vs individually enabled tools.
  • Prebuilt Microsoft MCP servers: Dataverse · Dynamics 365 ERP MCP (F&O data + business logic under the agent’s security role) · D365 Customer Service MCP Preview · Work IQ (M365 mail/calendar/Teams — needs M365 Copilot license).
  • Limitations: can’t enrich tool descriptions with extra invocation context; topics can’t call MCP tools directly.
Exam tip: “multiple agents need the same integration” or “the upstream API changes frequently” → MCP server. Rapid one-off prototype or no MCP server exists → direct connector/REST tool.

3.7Create validation criteria of custom AI models▾
  • Classic (discriminative) models: split data into training/testing; evaluate on the unseen split. Precision = TP/(TP+FP); Recall = TP/(TP+FN); F1 = harmonic balance. Optimize precision when false positives are costly (wrong extracted value corrupts accounting); optimize recall when false negatives are costly (missing an important email); F1 for balance. AI Builder shows a 0–100 performance score.
  • Foundry built-in evaluator categories (know the taxonomy): General (Coherence, Fluency) · Similarity (Similarity, F1, BLEU, ROUGE, METEOR) · RAG (Retrieval, Groundedness 1–5, Groundedness Pro — binary pass/fail via Content Safety, Relevance, Response Completeness) · Agent (Intent Resolution, Task Adherence, Tool Call Accuracy…) · Risk & safety (Hate/Unfairness, Sexual, Violence, Self-Harm, Protected Materials, Indirect Attack (XPIA), Code Vulnerability, Sensitive Data Leakage). LLM-judge scores 1–5 → pass/fail thresholds; custom evaluators supported.
  • Recommended combos: RAG apps = Retrieval + Groundedness + Relevance + safety; agents = Tool Call Accuracy + Task Adherence + Intent Resolution + safety; translation = BLEU/METEOR + Fluency.
  • GenAIOps evaluation lifecycle: (1) model selection via benchmarks; (2) pre-production — own eval data, simulators (adversarial simulators for safety), AI Red Teaming Agent; (3) post-production — continuous evaluation of sampled traffic + scheduled evaluations (drift) + scheduled red teaming.

  • “Move data across regions” (PPAC, per environment): consents to generative AI processing in another region when capacity isn’t local — otherwise some AI features simply don’t work. Region mapping to know: US → in-region · Europe → within EUDB · Brazil/Canada/Japan/Korea/South Africa/UAE → United States · Asia/Australia/India → in-region or US · UK → in-region or EUDB. Bing Search data is always processed in the US. Customer data isn’t used to train foundation models.
  • Flex routing (EU/EFTA): optional bursting of LLM inferencing outside the EUDB at peak demand; on by default for tenants created after March 25, 2026.
  • Microsoft 365 Copilot: EU Data Boundary service; data-residency commitments in the Product Terms since March 1, 2024 (Advanced Data Residency, Multi-Geo); Anthropic-provided models are currently excluded from the EUDB; semantic index stored with the mailbox/SharePoint region.
  • Dynamics 365: per-app cross-geo consents (Business Central “Allow data movement,” F&O cross-region requirement — see 3.15).
  • Validation approach: inventory environment geos (Power Platform inventory shows region), confirm consent toggles match policy, verify EUDB eligibility, review ongoing-transfer disclosures.
3.23Design access controls on grounding data and model tuning▾
  • M365 Copilot semantic index: returns only data the current user is authorized to access — permission trimming applies to retrieval/search APIs too; honors sensitivity labels + IRM usage rights; indexing never changes permissions.
  • Oversharing reduction: SharePoint Advanced Management (Restricted Access Control, restrict org-wide links), Purview Information Protection (auto-labeling, default labels), Purview DLP for Copilot (exclude labeled items from Copilot processing).
  • Copilot Studio grounding: SharePoint/Dataverse/enterprise-data knowledge use the agent user’s Entra ID (trimmed responses); uploaded documents and public websites need no auth — any agent user can see them, so govern with DLP; Dataverse row-level security via security roles; DLP endpoint filtering restricts which SharePoint sites/URLs can be knowledge.
  • Foundry RBAC: project-level least privilege (Azure AI User for inference); fine-tuning/deployment rights granted narrowly via role assignments; approved-model policies; trace readers need Log Analytics Reader.
  • Tuning data: customer data isn’t used to train foundation models; store tuning datasets in access-controlled storage (managed identities + private endpoints).
3.24Design audit trails for changes to models and data▾
  • Microsoft Purview Audit for Copilot & AI: the unified audit log captures prompts/responses, interaction metadata, and which files were accessed for grounding (with sensitivity labels); events flow to DSPM for AI Activity Explorer. For Copilot Studio, Purview audits maker activities (create/edit/delete/publish) and user–agent interactions incl. knowledge sources accessed and transcripts.
  • License tiers: Foundational (E3) — Audit, retention/deletion of interactions, eDiscovery incl. legal hold of prompts/responses; Optimized (E5) — Communication Compliance, Compliance Manager, Insider Risk Management (“Risky AI usage”), Audit Premium retention. Caveat: audit data is for security/compliance, not usage reporting.
  • Dataverse auditing: per environment/table/column — who/when/old-new values for records incl. agent components and configuration; route to Microsoft Sentinel for SIEM.
  • Foundry/Azure: Activity Log for control-plane changes (deployments, key regenerations, RBAC); agent version history (automatic snapshots, SHA-256 content-addressable versions); model registry provenance + approval history; evaluation runs and red-team scans logged.
  • Design guidance: centralize logs (Log Analytics/Sentinel); align retention with compliance; capture three trails — data changes (Dataverse auditing + dataset registration), model changes (registry + activity logs + pipeline history), interactions (Purview).

ALM golden rules

  • Customize only in dev · always in a custom solution (custom publisher/prefix) · deploy managed · remove unmanaged layers when changes don’t appear in prod.
  • Environment variables for per-environment values · connection references for connections · dev/test = sandbox, prod = production type.
  • Not solution-aware (reconfigure post-deploy): App Insights settings, manual auth, web-channel security, deployed channels, sharing.
  • Pipelines: stages in order, same artifact promoted, targets must be Managed Environments (auto-enforced Feb 2026); alternatives = Azure DevOps, GitHub Actions, pac copilot CLI.
  • Foundry agents: automatic version snapshots, content-addressable versions (SHA-256), azd deploy, OIDC CI/CD, evaluations + red teaming as release gates.
  • F&O: deployable packages via LCS (sandbox → prod, manual prod request) + Copilot apps in the linked Dataverse environment; pause max one One Version update; RSAT/BPM regression.
  • Data ALM: knowledge is per-environment; ingested files reindex ~4–6 h; conditional dataset registration; pair prompts with model versions.

KPI definitions (exact conditions)

  • Engaged = custom topic OR Boosting/Escalate/Fallback triggered. Rates (resolution/escalation/abandon) computed over engaged sessions.
  • Resolved = confirmed (End of Conversation + user confirms, CSAT collected) or implied. Abandoned = 60 min inactivity.
  • Deflection = resolved by self-service instead of human. Autonomous agents: run outcomes, trigger/tool/knowledge usage.
  • ROI: Agent Assisted Hours = weighted sessions × minutes saved ÷ 60 × loaded rate; baseline before build; 90-day decision gates.
