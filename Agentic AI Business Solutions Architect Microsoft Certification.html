Domain 1 · Plan

AI Strategy & the Cloud Adoption Framework

The Plan domain leans heavily on the Cloud Adoption Framework (CAF) for Azure — AI adoption guidance. Know the phases, the order, and what artifact each produces.

The six CAF AI adoption phases

Strategy → Plan → Ready are sequential; Govern, Secure, Manage are continuous and iterative.

Phase Key activities & artifacts
Strategy Identify use cases (start from business problems); classify each as individual work (→ Microsoft 365 Copilot class) or business automation (→ agents/integration); pick generative (non-deterministic, unstructured input) vs non-generative (deterministic — prediction, anomaly detection) AI; choose a technology adoption model; define responsible AI and data strategy
Plan AI maturity assessment (Levels 1–4): L1 = ready-made Copilots/quickstarts, L3 = RAG apps in Foundry, L4 = training custom models on AKS/VMs; skilling plan; prioritized use-case roadmap; proof of concept (internal, non-customer-facing first; Copilots reach ROI in days–weeks, custom Azure AI in weeks–months; add a 20–30% time buffer)
Ready Resource organization: separate management groups for internet-facing ("online") vs internal ("corporate") AI workloads; AI-specific Azure Policy; workload landing-zone subscriptions
Govern Follows NIST AI Risk Management Framework: assess risks against the six Responsible AI principles; document policies (model onboarding — sandbox first, then production catalog; a "golden dataset" as the quality benchmark; retraining frequency by risk tier); enforce with Azure Policy + Microsoft Purview; review quarterly (high-risk) / annually (low-risk)
Secure Discover AI risks, protect AI resources and data, detect AI threats — continuously
Manage Establish the AI Center of Excellence; choose MLOps (classic ML) vs GenAIOps (generative); keep a sandbox separate from dev/test/prod; manage deployment, model drift/decay, cost, and business continuity

The four technology adoption models

Trading customization against simplicity — a favorite exam framing:

  1. Ready-to-use SaaS Copilots/agents — Microsoft 365 Copilot, prebuilt D365 agents. Fastest ROI, least control.
  2. Low-code SaaS development — Copilot Studio, M365 Copilot extensibility. Business-process transformation on managed rails.
  3. Managed PaaS — Microsoft Foundry (models, agents, tools). Model choice, custom orchestration, deep integration.
  4. Azure infrastructure (IaaS) — bring-your-own models, full stack control, strictest isolation/compliance.

⚡ Exam focus: "Fastest time-to-value, minimal customization" → SaaS Copilot/prebuilt agent. "Department workflow automation, low-code, SaaS governance" → Copilot Studio. "Model choice / pro-code orchestration / evaluations / CI-CD" → Foundry. "BYO model, strict isolation" → IaaS.

The agent decision tree

Apply it in order for every use case:

  1. Do you need an agent at all? Static Q&A or one-shot content generation without reasoning/action → plain code or a non-generative model is cheaper and more reliable.
  2. Does a ready-to-use agent meet the need? (M365 Copilot agents — Researcher, Analyst; D365 first-party agents; Fabric data agents; GitHub Copilot; Security Copilot.) If yes — configure, don't build.
  3. Build low-code in Copilot Studio (retrieval, task, and autonomous agents) when speed + SaaS security fit.
  4. Build pro-code in Foundry (Agent Service / Agent Framework) for strategic transformation, custom logic, model choice.
  5. Start single-agent. Move to multi-agent only when the use case crosses security/compliance boundaries, spans multiple teams, or has known growth. Validate with 1–2-week time-boxed prototypes.

Assessing where agents help: automation, analytics, decisions

  • Task automation: agents suit variable, judgment-laden tasks; use deterministic automation (Power Automate / RPA) for stable rules. RPA vs computer-using agent: RPA when UI is stable, rules are clear, volume/speed matter; a computer-use agent when UIs shift, decisions are fuzzy, or logic depends on what's on screen.
  • Data analytics: agents (e.g., Fabric data agents, Analyst) let users interrogate governed data conversationally; they complement — not replace — BI.
  • Decision-making: agents recommend and prepare decisions; humans stay accountable for consequential ones (human-in-the-loop is both a Responsible AI and security control).

Agentic-first roadmap: transformation patterns

Microsoft's agentic transformation patterns guidance (learn.microsoft.com/agents) defines six patterns: Employee AI enablement · Business expert empowerment · Workplace & IT services · Core business process transformation · External engagement · AI-first capabilities.

  • Patterns are design choices, not sequential stages; run 2–3 in parallel at most.
  • The assist → execute line (from AI helping people to AI acting) is the biggest readiness jump.
  • Each pattern has a scale-breaker mapped to the agentic AI adoption maturity model's five capability drivers: AI strategy & experience, business strategy, governance & security, technology & data, organization & culture.
  • Method: define the outcome → assess readiness → align ambition with readiness (close the largest gap first) → sequence and reassess quarterly. Every pattern needs a named outcome and a named owner.

Domain 1 · Plan

Agent Types & Core Design Decisions

The agent taxonomy (exam vocabulary)

The exam's design bullets name three agent types you must be able to design. They map to Microsoft's complexity/autonomy spectrum:

Exam term CAF label What it does Design concerns
Prompt-and-response agent Retrieval agent ("productivity") Answers questions by retrieving and synthesizing knowledge Grounding quality, permissions, citation behavior
Task agent Task agent ("action") Takes defined actions inside a conversation (knowledge + action tools) Tool descriptions, authentication model, error/fallback paths
Autonomous agent Autonomous ("automation") Runs multi-step processes on event triggers with minimal oversight Triggers, guardrails, human-in-the-loop approvals, audit, cost

Autonomous agent design rules (straight from Microsoft guidance): narrow scope and explicit authority boundaries; high-quality instructions and data; sandbox testing and gradual rollout; human approval for critical actions; least-privileged access; validate trigger authenticity (e.g., sender checks on email triggers); guardrails and fail-safes; audit logging and monitoring. Common pitfalls: triggers missing context (record ID, identity), over-firing triggers burning Copilot Credits, ignoring platform quotas.

Build-or-extend decisions you must make instantly

Declarative agent vs custom engine agent (Microsoft 365 Copilot extensibility):

Declarative agent (DA) Custom engine agent (CEA)
Orchestrator & models M365 Copilot's own Yours (Foundry models, Agent Framework, LangChain…)
Hosting Microsoft-hosted, none to manage You host (unless built in Copilot Studio)
Compliance Inherits Microsoft 365 security/compliance/RAI Your responsibility
Behavior User-initiated only Proactive, autonomous, group collaboration
Channels M365 Copilot & M365 apps Any channel, external surfaces

Copilot Studio agent builder (lite, in M365 Copilot) vs full Copilot Studio: Agent builder = information workers, natural-language authoring, lightweight Q&A over SharePoint/Graph/web, individual/small team, governed in the M365 admin center. Full Copilot Studio = makers; multi-step logic, 1,000+ connectors, autonomous triggers, environments/ALM/RBAC/analytics, external publishing; governed in the Power Platform admin center. One-way "Copy to Copilot Studio" upgrade path.

Integration mechanism decision (how an agent reaches systems):

Need Choose
Simple/one-off API, fastest prototype Power Platform connector or REST/custom connector
Standardized, centrally versioned tool set shared by many agents; upstream APIs change often MCP server (update once, all agents inherit; requires generative orchestration)
No API exists — legacy web/desktop UI only Computer use (last resort — costlier, less deterministic)
Delegate work to an external/opaque agent (cross-team, cross-org) A2A protocol
Same-team, co-located capability Child agent / agent-as-tool (less latency & ops overhead than A2A)

When to create custom AI models

Create custom models only when unique domain data provides competitive advantage, prebuilt models miss accuracy/compliance needs, and the org has Level 3–4 AI maturity (data + skills). Otherwise: prebuilt Foundry Tools models, or fine-tune a foundation model. Path mapping: build agents → Foundry Agent Service; RAG & fine-tuning → Foundry; train classic ML → Azure Machine Learning (or Fabric); consume prebuilt AI → Foundry Tools.

Small language model (SLM) use cases

SLMs (≈ under 10B parameters — Microsoft's Phi family: Phi-3, Phi-3.5, Phi-4 variants) vs LLMs:

  • Choose a customized SLM for: well-defined narrow tasks; high-precision domain tasks; cost- and latency-critical paths; on-device/edge/offline (Foundry Local; Phi Silica on Copilot+ PC NPUs with LoRA adapters); data-privacy/residency (App Service SLM sidecar container — no external calls, fixed cost, no per-token fees); regulated tasks needing interpretability.
  • Choose an LLM for: open-ended generation, broad world knowledge, complex multi-step reasoning across domains.

⚡ Exam focus: trigger phrases → answer: "must run offline / on-device" · "data cannot leave the environment" · "predictable fixed cost" · "narrow repetitive task" = fine-tuned SLM (Phi). "Wide-ranging reasoning over changing knowledge" = LLM + RAG.

Build vs buy vs extend

  • Buy when a SaaS Copilot/prebuilt agent meets functional requirements (always evaluate first — exam expects you to check prebuilt agents before proposing custom).
  • Extend (declarative agents, Copilot Studio on top of M365/D365) when you need company data/processes on managed rails.
  • Build (Foundry) when you need model choice, custom orchestration, advanced RAG, fine-tuning, or external channels. Cost drivers: tokens in/out, compute, storage, grounding connections.

Domain 1 · Plan

Data Readiness & Grounding Design

The exam bullet: "Review data for grounding, including accuracy, relevance, timeliness, cleanliness, and availability" — plus "organize business solution data to be available for other AI systems."

The five grounding-data quality dimensions

From the Well-Architected grounding data design guidance:

Dimension What the architect checks Design response
Accuracy Is the content correct and consistent? Fix inconsistencies/duplicates before indexing; standardize schemas
Relevance Does the index match the questions users will actually ask? Anticipate user queries; make fields searchable/filterable/sortable only where needed (index bloat hurts relevance and cost)
Timeliness How stale can an answer be? Measure freshness = window between source change and index availability, tracked against SLOs; "an index should only be as fresh as required"
Cleanliness Noise, boilerplate, junk data Remove noisy/irrelevant data; curate before you index
Availability Can retrieval survive source-system outages/load? Externalize grounding data into a search index rather than querying source systems live — protects sources, keeps answers available

Core patterns: chunking vs full-document grounding (accuracy vs token cost/latency trade-off); embeddings + hybrid search (vector + keyword + semantic ranking); security trimming via filterable identity fields so users only ever retrieve authorized content; side-by-side index deployment for safe updates and emergency content removal.

Organizing business data for other AI systems

CAF's agent data architecture prescribes:

  • Unify on Microsoft Fabric OneLake; data domains publish governed data products through the medallion model (bronze → silver → gold).
  • Agents consume via Fabric IQ (ontology/semantic layer), Foundry IQ (knowledge retrieval in Foundry), and Copilot Studio knowledge.
  • Default to built-in retrieval (Foundry IQ, Fabric IQ, Fabric data agents, MCP servers, SharePoint tool, Azure AI Search, OneLake indexers) before building custom retrieval pipelines.
  • Dataverse Link to Fabric exposes Dataverse/D365 data in OneLake via shortcuts in near real time — no ETL, no data copy.
  • Fabric data agents (GA): conversational Q&A over lakehouses, warehouses, semantic models, KQL databases — and they can plug into Copilot Studio as connected agents.

Dataverse as the AI data backbone

  • Dataverse search must be enabled for Dataverse knowledge in agents; indexes: relevance search, structured, and unstructured (the latter two power Copilot/generative experiences and the Dataverse MCP server).
  • Dataverse knowledge: up to 15 tables per knowledge source; requires "Authenticate with Microsoft"; generative orchestration supports unlimited Dataverse sources (classic: 2).
  • Improve answer quality with per-column synonyms and a glossary (e.g., define "closed leads", map "opportunity revenue" to a custom column) — the same mechanism behind D365 Sales "business terms."
  • Security is honored end-to-end: agents answer only from rows/columns the querying user's security roles allow; connector-sourced files get live permission checks at query time.
  • Bring ERP (finance & operations) data via virtual entities (mserp_ tables) or dual-write; model-driven D365 apps inherit AI because they run on Dataverse.
  • Uploaded files in Copilot Studio are stored in Dataverse and auto-processed into semantic indexes + vector embeddings.

⚡ Exam focus: "Garbage in, garbage out" is the top reliability factor for agents. When a scenario shows wrong/insufficient agent answers, check — in order — (1) is the data accurate/clean, (2) is the right knowledge source connected & authenticated, (3) do users have permission to the source, (4) are synonyms/glossary needed for org-specific language?

AI Center of Excellence, Prompt Library & Prompt Engineering

Elements of the Microsoft AI Center of Excellence

CAF's "Establish an AI CoE" sequence: secure executive sponsorship (steering committee, monthly reviews) → appoint an AI CoE leader → assemble a multidisciplinary team (business leaders + data scientists, ML engineers, AI governance, AI security, AI ops) → integrate into an existing Cloud CoE rather than standalone → start centralized, evolve to advisory as maturity grows.

Responsibilities: define AI strategy and use-case intake/prioritization; responsible AI; skilling; pilot projects; define & enforce standards; build reusable assets; measure & report outcomes (adoption rates, compliance, cycle times). Triggers to shift to advisory mode: approval delays, knowledge bottlenecks, CoE-vs-team friction.

The agentic CoE guidance adds four functions — Govern (release gates, audit logs), Enable (golden paths, templates, training, community), Optimize (quality/cost/value monitoring), Scale (repeatable pipeline) — and three anti-patterns to avoid: the gatekeeper, the ghost, the perfectionist. It layers on top of (not replacing) security/compliance, cloud/IT governance, the Power Platform (low-code) CoE, M365 governance and responsible AI. In the operating model: platform teams enforce guardrails, workload teams own agent lifecycles, the AI CoE advises and sets standards. The Power Platform CoE Starter Kit (includes the Business Value Toolkit) remains the tenant-governance toolkit.

Prompt engineering guidelines (Microsoft's documented structure)

Microsoft's canonical prompt anatomy has four elements — remember GCSE:

Element Meaning Example
Goal What output you want "Write a step-by-step cleaning procedure"
Context Why / for whom / background "for SmartBrew machine owners"
Source Which data/knowledge/plugins to use "use the company knowledge base"
Expectations Format, tone, audience, length "casual tone, numbered list, ≤10 steps"

General best practices to teach and validate: be clear and specific; use action verbs and correct terminology; provide context; specify format and audience; break complex asks into steps; iterate ("adjust and resubmit"); English currently yields the best results; ground with knowledge sources rather than model memory; use few-shot examples and structured system instructions.

Copilot Studio prompt builder: a prompt = instructions + context (input variables + grounding data); good prompts are clear, concise, specific, contextual, relevant; the prompt assistant (preview) drafts prompts applying best practices; prompts run on Azure OpenAI models and are region-limited.

Prompt library — the four documented mechanisms

The exam bullet "provide guidelines for creating a prompt library" maps to these:

  1. Copilot Prompt Gallery (M365 Copilot / Copilot Chat) — Microsoft-curated + user-saved + team-shared prompts; stored in the Microsoft 365 Substrate inside the tenant boundary; admins can export prompt analytics.
  2. Organizational prompts (M365 admin center → Copilot → Prompts; currently gated to the Frontier preview program) — publish up to 1,000 prompts/tenant; bulk CSV import (≤100 prompts, 5 MB/file); pin up to 4 suggested prompts; ~3 hours to propagate; per-language entries (no auto-translation).
  3. Copilot Studio / AI Builder prompt sharing — prompts are private by default; owners share to users/teams (use, not edit); prompts are solution objects → versionable, deployable through ALM. This is the library mechanism for maker/agent prompts.
  4. Security Copilot promptbooks — reusable multi-step prompt sequences shared org-wide (know it exists as the pattern for sequenced prompts).

Architect guidelines for a prompt library: curate against the GCSE structure; assign owners and review cadence; version prompts in solutions; test with real sample data before publishing (prompt builder "Test prompt"; prompt accuracy scoring / prompt evaluations, preview); localize deliberately; measure usage and prune.

⚡ Exam focus: distinguish the mechanisms by audience — end users → Prompt Gallery & organizational prompts; makers/agents → AI Builder/Copilot Studio solution-aware prompts; sequenced investigations → promptbooks.

ROI, TCO & the Cost Model

Two exam bullets live here: "Select ROI criteria … including total cost of ownership" and "Create an ROI analysis for the proposed AI solution." You also need the licensing/cost mechanics that feed TCO.

ROI criteria & framework

Value measures (Power Platform business-value guidance): efficiency improvement (time before/after), revenue increase, employee productivity, TCO, employee/customer satisfaction. Business-value drivers: performance improvement, direct/indirect cost savings, risk mitigation. Classify benefits as tangible vs intangible.

TCO components: hardware, software/licensing, maintenance, energy/cooling, personnel, downtime, replacement — compared against platform cost.

Framework: centralize objectives → identify aligned metrics (time / rate / number / %) → collect & analyze (Power BI) → report to stakeholders. Standard math the exam may expect you to apply: ROI % = (benefit − cost) ÷ cost × 100; payback period = cost ÷ monthly net benefit.

Measurement discipline: baseline before rollout; keep a non-user comparison group; treat time-saved as a leading indicator and track where reclaimed time goes; avoid vanity metrics (seats purchased ≠ value). The Business Value Toolkit (CoE Starter Kit) structures the value story.

The cost model you must know (feeds every TCO question)

Microsoft 365 Copilot — per-user add-on license.

Copilot Studio — Copilot Credits (renamed from "messages" Sept 1, 2025):

  • Obtain via prepaid capacity packs = 25,000 credits/month (tenant-pooled, allocated to environments in PPAC) and/or pay-as-you-go on an Azure subscription. No monthly carryover.
  • Billing rates (memorize):
Consumption Credits

AI Builder credits — separate capacity for AI Builder models/prompts consumed from Power Apps/Automate.

Build vs buy vs extend (cost lens)

  • Buy/configure prebuilt (M365 Copilot, D365 first-party agents): license cost, near-zero build cost, days-to-value.
  • Extend (Copilot Studio, declarative agents): credits + maker time; weeks-to-value.
  • Build (Foundry): Azure consumption + engineering + ongoing GenAIOps; months-to-value but maximal fit. WAF note: fine-tuning's upfront cost can be less than an ongoing heavyweight RAG pipeline in some scenarios — evaluate both.

⚡ Exam focus: given a scenario, be ready to (1) pick ROI criteria incl. TCO, (2) compute simple ROI/payback, (3) spot the cheaper architecture (e.g., M365-Copilot-licensed employees hitting a Copilot Studio agent = no credit burn; batch jobs → Global Batch; mixed simple/complex traffic → model router; high-volume deterministic steps → agent flows not per-message generative answers).

The Platform Decision Framework

The single most-tested judgment on this exam: which platform builds which agent. The exam bullet: "Design a multi-agent solution by using platforms such as Microsoft 365 Copilot, Copilot Studio, and Microsoft Foundry."

The master comparison

Audience Individual / small team Department / org / external Developers, enterprise apps Developers

Interop points to remember: Copilot Studio can consume Foundry agents (preview), Fabric data agents (preview), M365 Agents SDK agents (preview), and any A2A-protocol agent (GA) as connected agents. Foundry agents publish to M365 Copilot, Teams, BizChat and the Entra Agent Registry; the Microsoft 365 Agents Toolkit/SDK carries pro-code agents into M365 channels. Azure AI Foundry is explicitly not the tool for extending M365 Copilot's own orchestrator — that's declarative agents / Copilot Studio.

Multi-agent orchestration patterns (Agent Framework)

Microsoft Agent Framework = the merger/successor of Semantic Kernel + AutoGen (enterprise features + simple abstractions + graph-based workflows with type-safe routing, checkpointing, human-in-the-loop).

Pattern Shape Use when
Sequential Pipeline A→B→C Staged processing (draft → review → format)
Concurrent Broadcast, parallel Independent analyses aggregated (ensembles)
Group chat Shared conversation + manager Brainstorming, consensus, debate
Handoff Dynamic transfer of control Triage/escalation by context or rules
Magentic Manager plans dynamically over specialists Open-ended problems with unknown solution path

Rules of thumb from docs: agent when the task is open-ended/conversational; workflow when steps are well-defined; plain code when no LLM judgment is needed. In Foundry Agent Service, classic connected agents are superseded by workflows (visual designer + YAML, versioned, multi-agent).

Multi-agent in Copilot Studio: child vs connected

  • Child agents — live inside the parent; one team owns everything; logical grouping of tools/instructions/knowledge; no separate deployment.
  • Connected agents — separately published agents (same environment, shared to you); use when tool selection degrades (rule of thumb: >30–40 tools/topics/agents of action choices), separate teams own components, independent ALM/publishing cadence, per-agent model settings, or reuse across parents.
  • Costs of splitting: extra orchestration hops (latency) and a bigger testing/governance surface. Limits: an agent with connected agents can't itself be a connected agent; Fabric data agents can't be redirect targets and don't work when the main agent is deployed to M365 Copilot.

Designing the cross-platform multi-agent solution

Standard AB-100 scenario shape: employee-facing knowledge + a business-process automation + a pro-code specialized capability. The expected answer usually composes:

  1. M365 Copilot / declarative agent for in-flow-of-work knowledge (inherits M365 trust boundary),
  2. Copilot Studio for the orchestrating business agent (connectors, approvals via agent flows, autonomous triggers), and
  3. Foundry for the deep capability (custom model, advanced RAG, evaluations), surfaced to Copilot Studio as a connected agent / MCP tool / A2A agent.

Governance completes the answer: Entra Agent ID for identities, PPAC DLP + M365 admin center (Copilot Control System) + Agent 365 registry for inventory and control.

⚡ Exam focus: eliminate options by governance home and model control. If the scenario demands "your own fine-tuned model" → Foundry. "No-code, org-wide, DLP-governed" → Copilot Studio. "Just answers from a SharePoint site for one team" → agent builder / SharePoint agent. "Coordinate specialists dynamically on open-ended tasks" → Magentic workflow in Agent Framework.

Domain 2 · Design

Copilot Studio I — Anatomy, Orchestration, Topics & Knowledge

Agent anatomy

An agent = instructions + knowledge + tools + topics + triggers coordinated by an orchestrator and language models. The orchestrator uses instructions plus the names and descriptions of tools/topics/agents/knowledge to decide what to invoke — descriptions are functional configuration, not documentation. Instructions can reference objects with /; an agent can't act on instructions that reference resources it doesn't have. Agents live in Power Platform environments (Dataverse-backed) → ALM, DLP, credit allocation all happen at environment level.

The language-understanding decision (direct exam bullet)

"Determine when to use standard NLP, conversational language understanding, or generative AI orchestration":

  1. Standard NLU (built-in, classic) — trigger phrases (5–20 per topic), RegEx/list entities; single intent; zero extra infrastructure.
  2. NLU+ — enhanced precision at enterprise scale; precompiled model requiring training; also optimizes speech recognition for voice; requires a Dynamics 365 Contact Center license.
  3. CLU (Azure AI Language) — bring a trained CLU model: intents map to topic triggers, CLU entities appear in the agent. Requires classic orchestration, an Azure subscription, matching region, intents for all system topics; enabling deletes existing trigger phrases. Choose for industry vocabularies, more languages, advanced entity extraction. (Note: CLU retires March 31, 2029; LUIS/QnA Maker are already retired.)
  4. Generative orchestration — the LLM plans; no trigger-phrase maintenance; multi-intent; context carryover.

Topics & the fallback design

  • Topic = trigger + nodes (Message, Question, Condition, tool calls, Redirect). Topics accept input/output variables; redirects map values and return to the caller. With generative orchestration, topic inputs auto-fill from context and outputs can feed the generated response.
  • System topics (can disable, can't delete): Conversation Start · Conversational boosting (generative answers — fires before Fallback) · End of Conversation (resolution confirm + CSAT) · Escalate · Fallback (unknown intent) · Multiple Topics Matched ("did you mean") · On Error · Reset Conversation · Sign in.
  • Fallback flow: asks the user to rephrase up to two times, then redirects to Escalate. Use UnrecognizedTriggerPhrase to pass the raw utterance to a flow/skill/external NLP. Mine Fallback + Conversational boosting hits to grow topics — that's the documented topic-enrichment loop.

Model router (direct exam bullet)

A deployable trained model that analyzes each prompt (complexity, reasoning need, tools, history) and routes to the most suitable underlying model behind one endpoint:

  • Modes: Balanced (default — cheapest within ~1–2% of best quality) · Cost · Quality.
  • Model subset: restrict routing to an approved list (compliance/cost; doubles as the automatic failover set — include ≥2 models). Claude targets must be deployed separately first.
  • Limits: effective context window = smallest underlying model; Global/Data Zone Standard; resource regions East US 2, Sweden Central; text+image in, routing decided on text.
  • Propose when: mixed simple/complex traffic, cost optimization, one endpoint simplifying lifecycle, heterogeneous agent turns (per-turn routing in Foundry Agent Service). Don't when you need the same deterministic model every request. Documented hybrid: router as default + direct deployments for specialized workloads.

Open Protocols — MCP & Agent2Agent (A2A)

Model Context Protocol (MCP)

An open standard (JSON-RPC 2.0) for connecting AI apps/agents to tools, data and context consistently — "reducing per-client custom integration."

Architecture — three roles:

  • Host — the LLM application embedding MCP clients (VS Code, Copilot Studio, Foundry Agent Service).
  • Client — the component inside the host holding a 1:1 connection to a server.
  • Server — the program exposing capabilities; local (stdio transport) or remote (Streamable HTTP). (SSE transport is deprecated.)

Primitives: Tools (functions the model invokes, with user approval) · Resources (read-only, file-like context) · Prompts (predefined templates). Clients discover dynamically (ListTools/CallTool, etc.). Copilot Studio supports tools + resources, not prompts; Foundry supports MCP tools with allowed_tools + approval controls.

Microsoft MCP surface (know these exist): Copilot Studio MCP tools · Foundry Agent Service MCP tool (require_approval defaults always) · Dataverse MCP server (https://<orgUrl>/api/mcp — search/create/update/delete tools; deletes require explicit user approval; PPAC setting "Allow MCP clients…" + per-client allow-listing; honors user permissions) · Dynamics 365 ERP MCP · D365 Sales/Customer Service MCP · Power Apps MCP server (agent feed) · Azure MCP Server · Microsoft Learn MCP · Agent Tools registry in Agent 365 for BYO MCP governance · hosting options on Azure Functions/Container Apps/API Management · Azure API Center as an organizational tool catalog.

MCP security risks → mitigations: prompt injection via tool output; tool poisoning (malicious tool descriptions); silent tool changes; credential/token theft; untrusted third-party servers. Mitigate: trusted providers only (no proxies); review/track servers; Entra ID auth + network isolation; least-privilege RBAC; tool allow-lists; human-in-the-loop approval on invocation; log all data shared for audit; DLP policies (Copilot Studio).

Agent2Agent (A2A)

An open standard (now under the Linux Foundation) for agent-to-agent communication across frameworks, vendors and organizations.

Key concepts: Agent Card — capability metadata discoverable at https://{domain}/.well-known/agent-card.json (or via registry/direct config) · messages (sync or streaming) · tasks (long-running work with task objects/continuation and an "input required" state) · artifacts (outputs) · contextId (conversation state across agents). Transports: HTTP+JSON and JSON-RPC. Crucially, the remote agent's tools and reasoning are opaque to the caller.

Microsoft support: Agent Framework hosts/consumes A2A agents natively (C#, Python, Go); Copilot Studio connects A2A agents (GA) as connected agents; Foundry Agent Service exposes A2A endpoints (preview) on published agents; Entra Agent ID supports OAuth 2.0, MCP and A2A for discovery/authorization.

MCP vs A2A — the comparison the exam loves

Dimension MCP A2A
Connects Agent ↔ tools/data/context Agent ↔ agent
Who orchestrates The host selects tools and synthesizes output The remote agent runs its own reasoning; caller sees results only
Discovery Server publishes tools/resources/prompts Agent card at a well-known URI
State Context stays with the host contextId carries context across agents; task lifecycle incl. "input required"
Trust shape You govern the tools your agent calls You delegate to an opaque, possibly external, agent
Choose when One orchestrator must control selection/invocation/synthesis Crossing service/team/org boundaries; dynamic delegation to specialized agents

Copilot Studio integration mapping (one-liner answers): plain API/HTTP → custom connector · standardized tool sets → MCP server · external framework agents → A2A · Microsoft 365 Agents SDK agents → Activity Protocol.

⚡ Exam focus: "Partner company exposes their agent; ours must delegate work to it" → A2A. "Many of our agents need the same evolving set of enterprise tools" → MCP. "Same team, same solution, low latency" → child agent/agent-as-tool — neither protocol. Security answer for both: Entra-based auth, least privilege, allow-lists, approvals, audit.

Authentication layers (design every one)

  1. User → agent: No auth (public; DLP can forbid) · Authenticate with Microsoft (Teams/M365 channel requirement) · Manual OAuth2 (federated credentials recommended over secrets; exposes User.AccessToken).
  2. Agent → tools/data: delegated (user) connections / OBO when actions must respect each user's permissions; application/managed identity for autonomous agents — scoped to minimal RBAC. Foundry tool auth: key, Entra (agent/project managed identity), OAuth identity passthrough.
  3. Least privilege everywhere: tool allow-lists (allowed_tools), short-lived just-in-time privileges, Dataverse security roles, human-in-the-loop for consequential actions.

Network security

  • Copilot Studio: Virtual Network support for connectors, IP firewall, Continuous Access Evaluation, Direct Line channel security, customer-managed keys (CMK), Customer Lockbox.
  • Foundry: Basic setup (Microsoft-managed) · Standard (BYO Storage/AI Search/Cosmos DB — agent data at rest in your tenant; CMK) · Standard + private networking (BYO VNet injection, private endpoints, no public egress). Private MCP endpoints supported.

Model security

  • Keyless auth (Entra) preferred; disable local/key auth (disableLocalAuth=true) via policy; data-plane vs control-plane RBAC are separate (control-plane Owner ≠ inference rights).
  • Foundry RBAC roles (renamed): Foundry User / Foundry Owner / Foundry Account Owner / Foundry Project Manager. Fine-tuning needs both planes: Foundry User covers training (data plane), but deploying a fine-tuned model is a control-plane action — Foundry Owner covers both, or pair Foundry User with Foundry Account Owner.
  • Training data & tuned models: stored in-region, logically isolated, deletable; customer's fine-tuned models are exclusive to the customer and never train foundation models; AES-256 at rest.
  • CMK: Key Vault same region/tenant, soft-delete + purge protection; revoking CMK blocks new training/deployments — but already-deployed fine-tuned models keep serving until deleted (a classic exam nuance).
  • Content filters configured at the resource level, associated per deployment; disabling filters ("no filters"/annotate-only) requires Limited Access approval.

Responsible AI — Principles, Standard & Tooling

The six principles (memorize verbatim)

  1. Fairness — comparable outcomes for similarly situated groups.
  2. Reliability & Safety — operates as designed; responds safely to the unanticipated; resists harmful manipulation.
  3. Privacy & Security — protect data and models; preserve privacy rights.
  4. Inclusiveness — empower everyone; accessibility.
  5. Transparency — explainable systems; disclose capabilities and limitations.
  6. Accountability — humans remain answerable for system operation and impact.

The Microsoft Responsible AI Standard v2

Operationalizes the principles: the six domains contain 14 goals, each decomposed into concrete requirements engineering teams implement across the lifecycle. Anchored practices: Responsible AI Impact Assessments (template published; required for sensitive uses), Transparency Notes for platform services and application cards for apps (capabilities, intended uses, limitations, mitigations), annual transparency reporting. Governance chain: Board committee → Responsible AI Council → Office of Responsible AI (policy, sensitive-use review).

AI shared responsibility model: the mitigation split moves with the layer — SaaS (M365 Copilot: Microsoft runs the safety systems) → PaaS (Foundry/Azure OpenAI: customer owns the system message, grounding data, filter configuration, app mitigations) → IaaS/custom models (customer owns training data and model safety).

The RAI tooling map (Foundry)

GenAIOps evaluation stages: (1) model-selection evals → (2) pre-production app evals (own data or simulators, incl. adversarial) → (3) production monitoring with tracing + continuous evaluation.

Built-in evaluators:

  • Quality/general: Coherence, Fluency (+ QA, similarity, NLP metrics).
  • RAG: Groundedness, Groundedness Pro, Relevance, Retrieval, Document Retrieval, Response Completeness.
  • Agent-specific: Intent Resolution (understood the user?), Task Adherence (followed instructions?), Tool Call Accuracy (right tools, right params?) — plus tool-selection/input/output/success and task-completion variants.
  • Risk & safety (output = defect rate; content-harm severity scale 0–7): hate/unfairness, sexual, violence, self-harm, protected materials, indirect attack (XPIA), code vulnerability, ungrounded attributes; agent-only previews: prohibited actions, sensitive-data leakage. LLM-judged quality scores run 1–5 with pass thresholds (default 3).

AI Red Teaming Agent (preview): automates adversarial probing with Microsoft's open-source PyRIT + safety evaluators — seed attack objectives per risk category → attack strategies (Base64/ROT13/leetspeak/morse encodings, character manipulation, jailbreak, XPIA via mock tool outputs, multi-turn/crescendo, compositions) → scores pairs → Attack Success Rate (ASR) scorecard by category × complexity. Local runs = single-turn text; cloud runs add agent targets + scheduled/continuous scans. Use alongside human red teaming; the security benchmark expects continuous red teaming.

Azure AI Content Safety (the service behind guardrails): Prompt Shields (user prompts + documents; GA — spotlighting preview), groundedness detection (preview, with correction), protected material (text + code, on by default), custom categories, harm-category analysis, blocklists, task-adherence API (preview). In the new portal these surface as Guardrails + controls with intervention points (user input, tool response) and actions (annotate vs block).
