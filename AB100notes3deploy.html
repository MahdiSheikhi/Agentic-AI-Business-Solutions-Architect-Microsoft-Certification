A. Analyze, monitor, and tune AI-powered business solutions

1. Recommend the process and tools required for monitoring agents

Plain English: After deployment, you monitor agents at several layers: the maker layer (Copilot Studio Analytics per agent), the developer/ops layer (Azure Application Insights + Kusto; Foundry observability for Foundry agents), the tenant/admin layer (Microsoft 365 admin center reports, PPAC), and the business-impact layer (Viva Insights Copilot Dashboard / Copilot Analytics).

Key exam facts:

  • Copilot Studio Analytics (Analytics page per agent): analytics for last 360 days; session details and transcripts for last 28 days; timestamps in UTC; test-panel conversations are excluded. Tailored views for conversational agents (Summary/Overview KPIs, Custom metrics, Effectiveness, Use sections) and autonomous agents (run outcomes, trigger use, tool use, knowledge source use). A Savings area estimates time/cost savings. The Summary area is an AI-generated (Copilot) digest of insights, including a customer-comments summary (preview).
  • Access control for analytics: share the agent with the Analytics Viewer role for view-only analytics; the Bot Transcript Viewer security role is additionally required to view transcripts. Analytics Viewer can only be granted to individuals, not groups.
  • Application Insights integration: configured in agent Settings > Advanced with a connection string (requires an Azure subscription). Options: Enable logging (incoming/outgoing messages/events), Log conversation details (user ID/name/message text; also controls tool inputs/outputs in OpenTelemetry spans), Log sensitive Activity properties, Node execution events. Makers can emit custom events with the Log custom telemetry event node. In unified authoring, App Insights integration is present by default.
  • Microsoft Foundry observability = three capabilities: Evaluation (built-in evaluators), Monitoring (Azure Monitor Application Insights dashboards: operational metrics, token consumption, latency, error rates, quality scores, alerting), Tracing (distributed tracing on OpenTelemetry GenAI semantic conventions; supports LangChain, LangGraph, OpenAI Agents SDK, Microsoft Agent Framework). Tracing is off by default and must be enabled by a Foundry Account/Foundry Owner; traces go to a connected Application Insights resource; viewers need Log Analytics Reader (or Privileged Monitoring Data Reader for protected tables). External (non-Foundry) agents can be registered for observability and matched by gen_ai.agent.id.
  • Microsoft 365 admin center: Microsoft 365 Copilot Agent usage report (Reports > Usage > Microsoft 365 Copilot > Agents) — data available within 72 hours, org-built agents (Copilot Studio, Teams Toolkit, agent builder), periods 7/30/90/180 days, metrics like active agents/active users/last activity. A new Agents usage report (preview) adds ~1-hour latency, both licensed and unlicensed users, all creator types (org, Microsoft, partners), and per-user/per-agent detail. The admin center's Agent workload (Microsoft Agent 365 as agent "control plane") shows agent inventory, usage trends, governance gaps, ownerless agents.
  • Microsoft names four admin reporting sources for Copilot: M365 admin center reports; Viva Insights Copilot Analytics (Copilot Dashboard + Advanced Insights analyst tools/Power BI templates); Microsoft Purview audit logs; Power Platform & Copilot Studio analytics.
  • Viva Insights Copilot Dashboard: categories Readiness, Adoption, Impact, Sentiment; available to Microsoft 365/Office 365 business customers with Exchange Online — no paid Viva Insights license required for the base dashboard; tenants with 50+ Copilot (or Viva Insights) licenses get agent insights, benchmarks (internal cohorts + external companies), delegation, group-level data. Impact metrics include Copilot actions taken, Copilot assisted hours, Copilot assisted value, satisfaction rate. Default minimum group size 10 (configurable, min 5). Viva Pulse integration captures sentiment surveys.
  • Copilot Agent Kit — Agent Insights Hub (Power CAT): centralized dashboards aggregating App Insights telemetry + transcripts; KPI cards: Conversations, Average DAU, Responses, Average response time, Duration, Tool success %, Errors, with period-over-period trends.
  • New Copilot Studio agent experience adds a Monitor tab (production-ready preview): analytics dashboard, session tracking, error monitoring; available only after publishing.

Choose X when: Copilot Studio Analytics for maker-level tuning of topics/knowledge/outcomes; Application Insights when you need custom telemetry, retention beyond 360/28 days, KQL, SIEM export, or alerting; Foundry observability/tracing for pro-code agents and step-level debugging/evaluation; M365 admin center for tenant-wide adoption/inventory; Viva Insights when leadership needs business impact/ROI and sentiment; Agent Insights Hub for a fleet view across many Copilot Studio agents.

2. Analyze backlog and user feedback of AI and agent usage

Plain English: Feedback signals (thumbs up/down, CSAT, comments) plus failure signals (unrecognized utterances, fallback hits, low-quality generative answers) feed a continuous-improvement backlog: enrich topics, fix knowledge gaps, redesign low-satisfaction flows.

Key exam facts:

  • Reactions: users give thumbs-up/down on any agent message and can add comments; visible in the Analytics tab for the past 28 days. Best practice: prompt for a comment after a reaction.
  • CSAT: collected via the End of Conversation system topic survey (1–5). Design conversations to end with End of Conversation so outcomes and CSAT are captured.
  • Unrecognized utterances: phrases that trigger no custom topic land in Conversational boosting (triggers first) or Fallback system topics. Topic enrichment analysis (offline exercise) buckets fallback queries into: (1) should have triggered existing topics (add trigger phrases), (2) candidates for new topics, (3) irrelevant/unmapped, (4) other (e.g., "Multiple Topics Matched"/did-you-mean, unclear queries). Topic confusion analysis finds semantically overlapping trigger phrases (measured by how often Multiple Topics Matched fires) — consolidate or differentiate topics.
  • Themes: Copilot Studio AI-clusters user questions that triggered generative answers into themes with response-quality metrics; you can create an evaluation test set directly from a theme with one selection. Question and reaction lists can be filtered and downloaded to CSV; downloaded transcripts include session-level CSAT (with Bot Transcript Viewer privilege).
  • Monthly optimization review items: deflection rate, resolution rate, engagement rate, topics with low resolution, unrecognized utterances, per-channel analysis. The improve checklist covers themes/backlog, conversation outcomes, generated answer rate and quality (completeness, groundedness, relevance), tool/action usage, knowledge-source error rates, satisfaction trends.
  • Generated answer rate/quality and daily AI-generated insights surface unanswered generative questions with specific coverage fixes.
  • Copilot Studio analytics vs Dynamics 365 Omnichannel analytics: Copilot Studio analytics covers agent/topic performance only; Omnichannel analytics gives end-to-end insight including human-representative conversations.

Choose X when: Add trigger phrases when fallback queries map to existing topics; create new topics when clusters are unmapped; consolidate topics when Multiple Topics Matched spikes; fix knowledge sources when generated answer quality is low; escalate to backlog items when themes recur with negative reactions.

3. Apply AI-based tools to analyze and identify issues and perform tuning

Plain English: Microsoft increasingly uses AI to analyze AI: AI-written analytics summaries, AI-grouped themes, natural-language custom metrics, LLM-as-judge evaluations, and AI-driven sampling of production traces.

Key exam facts:

  • Copilot Studio Summary area: Copilot-generated bulleted insights per reporting period (engagement, sentiment, trends); customer comments summary (preview) groups and prioritizes free-text feedback, filtering low-value comments.
  • Custom metrics (conversational agents): define up to three business-specific metrics in natural language; AI classifies sampled sessions; drill down to sessions with reasoning and transcript (needs Bot Transcript Viewer).
  • AI-generated insights: daily recommendations on unanswered generative questions with concrete fixes.
  • Agent evaluations (GA in Copilot Studio): AI can generate test cases from the agent's instructions, capabilities, and knowledge; quality is judged by AI methods (relevance, groundedness, completeness, abstention).
  • Foundry evaluations: AI-assisted (LLM-as-judge) evaluators score quality/safety; trace evaluation (preview) runs evaluators over production traces already in Application Insights — with intelligent sampling (MinHash farthest-first diversity selection: dedupe, hard filters, aggregation, diversity pick) to evaluate a representative, diverse subset at lower cost. Continuous evaluation of sampled production traffic plus scheduled evaluations detect drift.
  • Copilot Agent Kit: user-defined rubrics for grading generative answers; Agent Debugger for step-by-step diagnostics of recorded conversations; Conversation KPIs parsing transcripts.
  • Security Copilot has Autotune instruction optimization (scores current vs recommended instructions) — an example of AI-assisted tuning of agent instructions.
  • Guidance set: "Optimize prompts and instructions" — refine instructions, tool descriptions to guide the orchestrator.

4. Monitor agent performance and metrics

Plain English: Know the KPI definitions cold — the exam loves the exact conditions that flip a session between engaged/resolved/escalated/abandoned.

Key exam facts (definitions):

  • Engagement rate: % of sessions classified engaged. A session starts unengaged; it becomes engaged when a custom topic triggers or the conversation enters Conversational Boosting, Escalate, or Fallback system topics. Low engagement = topic coverage gaps or unclear agent purpose.
  • Session outcomes (engaged sessions only): Resolved — confirmed (End of Conversation topic runs and user confirms; may collect CSAT) or implied (agent logic determines success without user confirmation). Escalated — Escalate topic or Transfer conversation node. Abandoned — ended without resolution/escalation after 60 minutes of inactivity.
  • Resolution rate = resolved/engaged; Escalation rate = escalated/engaged; Abandon rate = abandoned/engaged; CSAT = average 1–5 survey score; Deflection rate = % of incoming requests resolved via self-service instead of human escalation (the key tier-1 support metric); First-contact resolution (FCR) = resolved on first interaction with no return contact within 7 days.
  • Autonomous agent KPIs: run outcomes (successful/unsuccessful with average duration — the autonomous analogue of resolution rate), trigger use, tool use (frequency + success), knowledge source use (frequency + errors).
  • Total sessions: one conversation can produce multiple analytics sessions as the user raises new questions.
  • Latency / token usage: Foundry monitoring dashboards track token consumption, latency, error rates, quality scores; tracing spans carry timestamps, latency, token usage, errors. Copilot Agent Kit tracks average response time and duration; Copilot Studio test activity map shows per-step execution times.
  • Savings: Copilot Studio Savings area estimates time and cost savings attributable to the agent/tools.

5. Interpret telemetry data for performance and model tuning

Plain English: Raw telemetry lives in Application Insights (query with Kusto/KQL) and, for Foundry, in OpenTelemetry traces that can also be evaluated after the fact.

Key exam facts:

  • Copilot Studio App Insights events land in the customEvents table. Event names include BotMessageReceived, BotMessageSend, TopicStart, TopicEnd, Action, GenerativeAnswers, custom telemetry. Rich data sits in customDimensions: type (message/event/invoke), channelId (msteams/directline/webchat), text, fromId/fromName, locale, TopicName, errorCodeText, and designMode (True when from the test canvas — filter it out with where isDesignMode == "False" for production analysis).
  • Example KQL patterns to recognize: summarize uc=dcount(user_Id) by bin(timestamp, 1d) | render timechart (distinct users/day — only accurate with authenticated, consistent user IDs); top-errors query parsing errorCodeText; GenerativeAnswers telemetry extraction.
  • App Insights does not expose full internal reasoning (intermediate search queries, discarded knowledge). For deeper visibility use conversation snapshots/transcripts, custom telemetry, or Foundry architectures.
  • Foundry tracing: spans follow OpenTelemetry GenAI semantic conventions; trace evaluation reads invoke_agent spans with attributes gen_ai.operation.name, gen_ai.agent.id (format agent-name:version), gen_ai.input.messages, gen_ai.output.messages, gen_ai.tool.definitions, gen_ai.conversation.id. Traces record user inputs, agent/model inputs and outputs, tool calls, timestamps, latency, token usage, errors.
  • Trace evaluation modes: by trace IDs (operation_Id values) or by agent filter (auto-discovery over a lookback window). Project managed identity needs Log Analytics Reader on both the App Insights resource and its Log Analytics workspace.
  • Agents playground thread logs: view thread details, run steps, tool calls, inputs/outputs; optional Metrics enable automatic AI quality + Risk & safety evaluations (billed; results expire after 24 hours; unavailable in some regions).
  • Interpreting for tuning: high latency in a tool span → optimize the tool; low groundedness scores → fix knowledge/RAG; frequent tool failure spans → repair connectors; token spikes → prompt/context optimization.
  • Privacy: trace data can contain personal data/customer content — restrict access, redact sensitive content, treat as production telemetry; retention follows Application Insights configuration.

B. Manage the testing of AI-powered business solutions

6. Recommend the process and metrics to test agents

Plain English: Treat agent testing as continuous: interactive testing in the test pane during authoring, automated evaluation test sets pre-deployment, bulk/regression testing with the Copilot Agent Kit, adversarial red teaming, and load testing.

Key exam facts:

  • Test pane ("Test your agent"): walk conversations, click a response to jump to the originating node (fired nodes show colored checkmarks), Track between topics, real-time activity map for generative orchestration (shows the orchestrator's plan, tool selection, inputs/outputs, per-step durations, errors), Variables panel, Chain of Thought view (only for selected models such as GPT-5 Reasoning, Claude Sonnet/Opus). Test panel activity is excluded from analytics.
  • Testing strategy types (Microsoft guidance table): development-time (unit) testing, core "happy path" scenario testing, knowledge testing, regression testing, adversarial testing (edge cases, unknown intent, false info), performance/load testing, security & compliance testing (RBAC, conditional access, sensitivity label enforcement, no data leaks), accessibility & UX testing. Principles: shift-left, iterate test cases, always validate in staging before production, automate in CI/CD pipelines.
  • Agent evaluations (Copilot Studio, GA): test sets of up to 100 test cases; create manually, import CSV, generate with AI from instructions/knowledge, or from past test-chat conversations/themes. Test methods: Text match (exact/partial/keywords), Similarity (semantic similarity to expected answer), Quality (AI judges relevance, groundedness, completeness, abstention — no expected answer needed). Multi-turn conversation tests evaluate realistic dialogs. Run under selectable user profiles (different permissions). Results retained 89 days; export to CSV. Automatable via REST APIs and Power Platform connectors for CI/CD gates.
  • Copilot Agent Kit test automation: test types include response match, topic match, generative answers (with AI-judge/rubrics), attachments (Adaptive Cards incl. invoke actions), multi-turn, and Plan Validation (deterministic check that the generative-orchestration plan contains expected tools, with a pass-threshold %). Results capture latency and pass/fail. Requires allowed connectors (Direct Line channels, Dataverse, etc.) under DLP.
  • Red teaming: use the AI Red Teaming Agent (see #7) plus human-in-the-loop probing before deployment and on a schedule after.
  • Batch testing for prompts (Test hub, preview): for AI Builder-style prompts in Copilot Studio/Power Apps/Power Automate — test cases via upload (CSV), AI-generate, activity data, or manual; prebuilt criteria Response quality, Response matches, JSON correctness plus a passing score; run history tracks accuracy progression/regressions.
  • Pre-deployment security checks: validate data policies and RBAC in production, Azure app registrations/VNets/keys, and that production knowledge sources are referenced (not dev/test copies).

Choose X when: test pane for authoring-time debugging; agent evaluations for repeatable pre-release quality gates; Copilot Agent Kit for bulk regression and plan validation at scale; AI Red Teaming Agent for safety/security probing; performance testing guidance for high-volume readiness.

7. Create validation criteria of custom AI models

Plain English: Classic (discriminative) models are validated with precision/recall/F1 against a labeled test split; generative models/agents are validated with AI-assisted evaluators (groundedness, relevance, coherence, fluency, agent/tool metrics) and safety evaluators, plus red teaming.

Key exam facts (classic ML / AI Builder / Language):

  • Dataset is split into training and testing sets; evaluation runs automatically after training against the unseen test set.
  • Precision = TP/(TP+FP) — of predicted positives, how many are right. Recall = TP/(TP+FN) — of actual positives, how many were found. F1 = 2·P·R/(P+R) — balance of both. Computed at entity/class level and model level. AI Builder shows a composite performance score (0–100) plus Quick Test.
  • Decision criteria: optimize precision when false positives are costly (e.g., extracting an interest rate from a contract — wrong value corrupts accounting; ticket misrouting); optimize recall when false negatives are costly (e.g., search indexing enrichment, missing an important email); use F1 for balance. For single-label classification, FP count = FN count so precision = recall = F1.
  • Foundry built-in evaluators (know the categories):
  • General purpose: Coherence, Fluency.
  • Textual similarity: Similarity (AI-assisted), F1 score, BLEU, GLEU, ROUGE, METEOR.
  • RAG: Retrieval, Document Retrieval, Groundedness (1–5 model-judged), Groundedness Pro (binary pass/fail via Azure AI Content Safety, no judge-model deployment needed), Relevance, Response Completeness.
  • Agent: Intent Resolution, Task Adherence, Task Completion, Tool Call Accuracy, Tool Selection, Tool Input Accuracy, Tool Output Utilization, Tool Call Success, Task Navigation Efficiency.
  • Risk & safety: Hate/Unfairness, Sexual, Violence, Self-Harm, Protected Materials, Indirect Attack (XPIA), Code Vulnerability, Ungrounded Attributes, Prohibited Actions, Sensitive Data Leakage.
  • Azure OpenAI graders: Model Labeler, String Checker, Text Similarity, Model Scorer. Custom evaluators (preview) supported.
  • LLM-judge evaluators output 1–5 scores converted to pass/fail via thresholds (4.0–5.0 strong, 3.0–3.9 acceptable, <3 weak/poor); agent/tool evaluators are binary pass/fail. Evaluation levels: turn vs conversation — all evaluators in a run must support the chosen level.
  • Recommended combos: RAG apps = Retrieval + Groundedness + Relevance + content safety; agents = Tool Call Accuracy + Task Adherence + Intent Resolution + safety; translation = BLEU/METEOR + Fluency.
  • Lifecycle stages of evaluation (GenAIOps): (1) base model selection via Foundry model benchmarks/own data; (2) pre-production: bring-your-own evaluation data, simulators (context-appropriate for quality; adversarial simulators for safety), and the AI Red Teaming Agent; (3) post-production monitoring: continuous evaluation of sampled traffic, scheduled evaluations against test datasets (drift detection), scheduled red teaming.
  • AI Red Teaming Agent (preview): built on Microsoft's open-source PyRIT + Foundry Risk & Safety Evaluations. Automates: adversarial scans of model/app endpoints, scoring of attack-response pairs, key metric Attack Success Rate (ASR), scorecards logged in Foundry. Supports seed prompts per risk category (default: violence, sexual, hate/unfairness, self-harm), attack strategies (Base64, Caesar/ROT13, leetspeak, character flip, suffix append, etc. from PyRIT); single-turn, text-only; limited regions (e.g., East US 2, France Central, Sweden Central, Switzerland West, North Central US). Targets: model configs, callbacks, PyRIT prompt targets. Recommended with human-in-the-loop, mapped to OWASP LLM Top 10 and MITRE ATLAS practices.

8. Validate effective Copilot prompt best practices

Plain English: Microsoft teaches a four-element prompt anatomy and a set of best practices; validating prompts means checking they contain these elements and testing/iterating them.

Key exam facts:

  • The four elements of an effective prompt: Goal (what output you want), Context (why/for whom/situation), Source (which data/files/plugins to use), Expectations (format, tone, audience). Memorize this — it appears across M365 Copilot, Security Copilot, Dynamics 365, and Fabric docs.
  • Best practices: be clear and specific (avoid vague asks); use natural language, full sentences, good grammar/punctuation; use explicit verbs and correct product/domain terminology; provide context and details; break complex requests into smaller steps; set expectations (format, expertise level); iterate — refine and resubmit; avoid interrupting/topic-mixing (say "new topic"); don't rely on document location for context; English generally performs best.
  • Validation/testing aids: Copilot Prompt Gallery (discover/save/share prompts); Prompt Coach agent template (M365 Copilot agent that critiques prompts for structure and Responsible AI compliance); Copilot Studio Prompt assistant in Prompt builder (GPT-powered drafting suggestions); batch testing for prompts (Test hub) to score prompt outputs against criteria at scale (see #6); Copilot Studio guidance "Optimize prompts and instructions" for agent instructions and tool descriptions.
  • For agent instructions, validate with agent evaluations: if tool selection accuracy is low → refine tool-usage guidance in instructions; if quality is low → add examples; compare models.

9. Design end-to-end test scenarios of AI solutions that use multiple Dynamics 365 apps

Plain English: Cross-app AI solutions (e.g., an agent that summarizes Sales opportunities inside Customer Service, or Copilot in F&O + a Copilot Studio agent) need E2E scenarios that traverse app boundaries, environments, and handoffs — tested in production-like staged environments.

Key exam facts:

  • Environment progression for D365 CE testing: DEV > TEST > PRE-PROD > PROD with managed solutions promoted upward; Copilot in Customer Service configuration moves through the same solution ALM process — include it in E2E test scope.
  • Handoff testing: for Copilot Studio → Dynamics 365 Customer Service (Omnichannel) escalation, testing on a custom website must use the embed code from the D365 chat widget, not the Copilot Studio embed code, or handoff fails. Agent and D365 Customer Service must be in the same environment; each agent needs a unique Application ID (reuse causes error 1004 DuplicateBotAppId).
  • Cross-environment/cross-app scenarios: reference architecture uses a custom connector + Azure Function calling out-of-box plugins (e.g., msdyn_SalesOpportunitySummary) so Copilot in Customer Service can summarize Sales data — E2E tests must validate connector auth, plugin enablement (admin/agent auth, security-role access), and the returned summary.
  • F&O testing: unit/component tests with the SysTest framework; automated regression with RSAT-style automation testing solutions; Business process modeler (BPM) generates test cases/scripts from business processes and syncs to Azure DevOps; regression test every One Version service update in a sandbox/UAT before production.
  • E2E scenario design should include: identity flow (SSO, per-user permissions), data flow across apps (Dataverse ↔ F&O via Power Platform integration/dual-write), AI-specific checks (grounding correctness per app, summary field mappings in Customer Service), channel behavior, escalation/handoff, and load/performance for conversational agents.
  • Use agent evaluation user profiles and Copilot Agent Kit multi-turn tests to simulate different roles across a scenario; validate security/compliance (RBAC, sensitivity labels, no cross-app data leakage).
  • Analytics differences matter for verifying outcomes: Copilot Studio analytics stops at the agent; Omnichannel/Customer Service analytics covers the human side after escalation.

10. Build the strategy for creating test cases by using Copilot

Plain English: Use AI itself to generate, maintain, and scale test cases — in Copilot Studio evaluations, prompt Test hub, and code-based testing of Power Platform apps.

Key exam facts:

17. Design security for agents

Plain English: Secure agents along four axes: who can talk to the agent (user authentication), whose identity accesses data (user vs author credentials), what the agent may connect to (DLP/data policies), and who can build/share it (sharing controls).

Key exam facts:

  • Foundry Agent Service: dedicated Entra agent identities, RBAC, private networking (BYO VNet for hosted agents), integrated content filters incl. cross-prompt-injection mitigation.

20. Analyze solution and AI vulnerabilities and mitigations, including prompt manipulation

Plain English: The two headline attack classes are direct prompt injection (UPIA/jailbreak — the user attacks) and indirect/cross-prompt injection (XPIA — attacker hides instructions in content the AI reads). Defense is layered: shields, spotlighting, isolation, monitoring, least privilege, human-in-the-loop.

Key exam facts:

  • Direct (UPIA / jailbreak): user deliberately subverts the system. Prompt Shields recognizes four subtypes: attempts to change system rules, embedded conversation mockups, role-play/persona replacement ("you are now Yendys with no limits"), encoding attacks (Base64, ciphers, character transforms).
  • Indirect (XPIA / cross-prompt injection): malicious instructions embedded in third-party content (emails, documents, websites, plugin/tool outputs) that the model treats as commands — can cause unauthorized actions and data exfiltration. Traditional input validation is insufficient because the model can't distinguish user intent from external content.
  • Azure AI Content Safety Prompt Shields (text:shieldPrompt API; formerly "Jailbreak risk detection"): analyzes userPrompt and documents arrays, returns attackDetected per element. In Foundry: configured as guardrails with intervention points (user input, tool response) and actions (annotate/block), assignable to deployments/agents.
  • Spotlighting (preview sub-feature of Prompt Shields): transforms/tags document content (base64 encoding) so the model treats it as lower-trust than user/system prompts; off by default; no direct cost but increases tokens and can hit input limits; Chat Completions models only; side effect: model may mention base64.
  • Defense-in-depth pattern (Zero Trust guidance): assume XPIA will happen; combine probabilistic + deterministic controls — prompt shields, spotlighting, plan drift detection, critic agents, tool-chain analysis, information flow control (IFC) to isolate untrusted content, least privilege + short-lived privileges, human-in-the-loop for risky actions; monitor with Defender for Cloud + Sentinel mapped to MITRE ATLAS (e.g., AML.T0051 LLM Prompt Injection, AML.T0057 LLM Data Leakage) and OWASP Top 10 for LLM applications.
  • Related detections/evaluators: Content Safety Groundedness detection (ungrounded/hallucinated output); Foundry safety evaluators Indirect Attack (XPIA), Code Vulnerability, Ungrounded Attributes, Sensitive Data Leakage, Prohibited Actions; AI Red Teaming Agent to probe continuously (integrate in CI/CD and schedule in production).
  • Other AI vulnerability themes Microsoft covers: adversarial examples, data poisoning, model inversion/membership inference (data leakage), insecure plugin/tool chains, oversharing via maker credentials (see #17). Insider Risk Management "Risky AI usage" template.

21. Review solution for adherence to responsible AI principles

Plain English: Microsoft's six RAI principles are the review vocabulary; the Responsible AI Standard operationalizes them; impact assessments and transparency notes are the artifacts; reviews act as release gates that scale with risk.

Key exam facts:

  • Six principles: Fairness (treat similarly situated groups the same), Reliability & safety (operate as designed, fail safely, resist harmful manipulation), Privacy & security (protect data/models, honor data boundaries), Inclusiveness (empower everyone, accessibility), Transparency (explainable, capabilities/limitations communicated), Accountability (named human owners answerable for behavior and outcomes).
  • Microsoft Responsible AI Standard v2: framework of 6 domains and 14 goals with concrete requirements; governance bodies: Office of Responsible AI (ORA), Responsible AI Council, Aether committee; board oversight.
  • Microsoft's AI development lifecycle aligns with NIST AI RMF: Govern → Map → Measure → Manage. Map includes the Responsible AI Impact Assessment (template available; assess harms, affected stakeholders, mitigations) plus threat modeling and AI red teaming. Measure uses evaluations (groundedness, relevance, content safety). Partners/customers are advised to run the same process: assess with the Impact Assessment Template, review with diverse stakeholders, mitigate, measure mitigation success.
  • Transparency notes: Microsoft documents per-service capabilities, limitations, intended uses, and best practices (e.g., Transparency note for Azure OpenAI); use them when evaluating/integrating; produce your own transparency documentation (transparency note, limitations statement, safety FAQ) for published AI solutions.
  • Tooling: Azure Machine Learning Responsible AI dashboard (fairness assessment across sensitive groups, model interpretability/global-local explanations, counterfactual what-if, error analysis) and RAI scorecard PDF; Foundry risk & safety evaluators; content filtering; for agents: design-guideline table mapping each principle to agent design decisions; review checklist as a release gate and post-launch recurring review.
  • Power Platform/Dynamics copilots follow the Microsoft Responsible AI Standard; each Copilot feature ships Responsible AI FAQs.

22. Validate data residency and movement compliance

Plain English: Know where each layer stores and processes data: environment geo for data at rest, Azure OpenAI hosting region for inferencing, the EU Data Boundary commitments, and the admin consent toggles that permit cross-region AI calls.

Key exam facts:

  • Copilot Studio: environments are created in a chosen geographic region; customer data at rest stays in that geo's Azure datacenters. EU Data Boundary (EUDB) compliant when the tenant's billing address is EU/EFTA and all environments are created in EUDB regions. Supports GDPR (DSRs, DPIAs) and broad certifications.
  • "Move data across regions" (PPAC → generative AI settings, per environment): when generative AI capacity isn't hosted in-region, prompts/outputs may be processed in another region only if this consent is on; otherwise some AI features simply don't work. Region mapping to know: US → in-region; Europe + FR/DE/NO/SE/CH → within EU Data Boundary; Brazil/Canada/Japan/Korea/South Africa/UAE → United States; Asia/Singapore/Australia/India → in-region or US; UK → in-region or EUDB; GCC/GCC-High in-region. Bing Search data is always processed in the US. Data isn't used to train foundation models.
  • Flex routing (EU/EFTA): optional setting letting LLM inferencing burst outside the EUDB (US, Canada, Australia) during peak demand; data encrypted in transit/at rest; data at rest stays in EUDB except limited pseudonymized IDs; on by default for eligible tenants created after March 25, 2026.
  • Microsoft 365 Copilot: an EU Data Boundary service for EU users; covered by data residency commitments in the Product Terms since March 1, 2024, including Advanced Data Residency and Multi-Geo; LLM calls route to nearest capacity (worldwide traffic can use EU and other regions); Anthropic-provided models are currently excluded from the EUDB; abuse-monitoring prompt storage up to 24 hours (Azure OpenAI); semantic index stored with mailbox/SharePoint region, honoring EUDB and Multi-Geo.
  • Dynamics 365: per-app cross-geo consent (e.g., Business Central "Allow data movement" toggle with a region table; Sales agent consent rules; F&O copilots need cross-region data movement in some regions — see #15).
  • Validation approach: inventory environment geos (Power Platform inventory shows region — "find agents running in non-approved geographies"), confirm consent toggles match policy, verify EUDB eligibility, check ongoing-transfer disclosures (flex routing, OpenAI-operated models storing pseudonymized user IDs in the US).

23. Design access controls on grounding data and model tuning

Plain English: Grounding must respect source permissions: M365 Copilot's semantic index is permission-trimmed per user; Copilot Studio knowledge should use user (not maker) auth; Dataverse enforces row-level security; Foundry uses Azure RBAC to control who can tune/deploy models.

Key exam facts:

  • M365 Copilot / semantic index: only returns data the current user is authorized to access — identity-based access boundary honored during grounding; permission trimming applies to Copilot Retrieval/Search APIs too; honors sensitivity labels + IRM usage rights (encrypted content restricted; programmatic-access exclusion can block agents); Conditional Access applies. Indexing does not change permissions; tenant-level index only surfaces content the user can already access.
  • Oversharing reduction: SharePoint Advanced Management (Restricted Access Control by default for business-critical sites, restrict Anyone links/org-wide links), Purview Information Protection (site labels at provisioning, auto-labeling, default labels), Purview DLP for Copilot (exclude labeled items from Copilot processing; block prompts containing specified sensitive info).
  • Copilot Studio grounding: SharePoint, Dataverse, and enterprise-data (Graph/Copilot connector) knowledge use the agent user's Microsoft Entra ID authentication — responses trimmed to that user; uploaded documents and public websites need no auth (thus visible to any agent user — govern with DLP). Sensitivity labels surfaced for SharePoint sources (highest label shown). Dataverse row-level security via security roles governs Dataverse knowledge. DLP endpoint filtering limits which SharePoint sites/URLs may be used as knowledge. Enforce end-user credentials for tools (see #17).
  • Foundry RBAC: least-privilege built-in roles assigned at project level (e.g., Azure AI User for using models, project managers/owners for configuration); who can fine-tune/tune models is controlled through role assignments on the Foundry/Azure OpenAI resource (fine-tuning and deployment need contributor-class data actions — grant narrowly); Entra Agent ID for agent access; trace data readers need Log Analytics Reader (Privileged Monitoring Data Reader for protected tables); approved-model policies gate what can be deployed.
  • Fine-tuning/training data protection: customer data not used to train foundation models; store tuning datasets in access-controlled storage (managed identities + private endpoints per #19).

24. Design audit trails for changes to models and data

Plain English: Auditability comes from Purview's unified audit log (AI interactions + maker/admin activities), Dataverse auditing (data changes), Application Insights/Foundry traces (runtime), and version histories (agents, models, solutions).

Key exam facts:

  • Microsoft Purview Audit for Copilot & AI applications: prompts/responses and interaction metadata captured in the unified audit log — when/how users interacted, which M365 service, which files were accessed for grounding and their sensitivity labels; events flow to DSPM for AI Activity Explorer ("Copilot Interaction" records). Copilot Studio: Purview audits maker activities (agent create/edit/delete/publish) and user–agent interactions including generative-AI logs, knowledge sources accessed, and conversation transcripts (GA Jan 2025; authenticated users). Foundational (E3): Audit, Data Lifecycle Management retention/deletion of Copilot interactions, eDiscovery incl. legal hold of prompts/responses. Optimized (E5): Communication Compliance (detect violations in AI interactions), Compliance Manager, Insider Risk Management (risky AI usage), Audit Premium retention.
  • Caveat: audit data is for security/compliance, not usage reporting — use usage reports/Copilot Dashboard for metrics.
  • Dataverse auditing: enable per environment/table/column to log create/update/delete of records (including agent components, AI Builder models, configuration tables) with who/when/old-new values; route to Microsoft Sentinel for SIEM; Application Insights for runtime telemetry.
  • Foundry/Azure: Azure Activity Log records control-plane changes (deployments, key regenerations, RBAC changes); agent version history (automatic snapshots; content-addressable versions with SHA-256) provides change trails for agents/prompts; model registry tracks model provenance and approval history; evaluation runs and red-team scans logged in Foundry for compliance tracking; tracing/App Insights hold runtime records (respect privacy controls).
  • Dynamics 365 example (Business Central pattern): Purview audit events like "Agent configuration changed," "Agent state changed," "User-agent interaction," "Agent action executed" — shows the direction of agent-specific auditing across D365.
  • Design guidance: centralize logs (Log Analytics/Sentinel), align retention with compliance obligations, capture data-change trails (Dataverse auditing + dataset version registration), model-change trails (registry + activity logs + pipeline history in DevOps/GitHub), and interaction trails (Purview).
