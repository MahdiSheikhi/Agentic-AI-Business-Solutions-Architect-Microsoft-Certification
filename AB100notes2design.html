Section A: Design AI and agents for business solutions

1. Design business terms for Copilot in Dynamics 365 apps for customer experience and service

Plain English: Copilot in D365 Sales/Customer Service answers sellers'/reps' natural-language questions against CRM data. Out of the box it only knows standard table/column names. You configure business terminology (glossary + synonyms) so Copilot correctly interprets org-specific language ("top deals," "custom revenue," "we").

Key exam facts:

  • Copilot in Dynamics 365 Sales is itself a Copilot Studio agent named "Copilot in Dynamics 365 Sales" in the environment. Customization path: Copilot Studio → select the D365 Sales environment → Agents > Copilot in Dynamics 365 Sales → Knowledge > SalesSpecificQnA → Glossary and Synonyms sections → Publish.
  • Glossary = maps business terms + logic/descriptions so Copilot interprets queries (e.g., define what "top," "best," "we" mean; ownership models; fine-tune CRM queries). Glossary applies across all skills, including custom skills.
  • Synonyms = alternative names for table columns (e.g., add "opportunity revenue" as a synonym for a custom column "custom revenue" on the opportunity table) so sellers can ask questions in their own words.
  • Prerequisite: the maker needs the Copilot Studio Author security role (assigned in Power Platform admin center).
  • The same glossary/synonym configuration also drives which tables the Sales agent in Microsoft 365 Copilot can query (tables with synonyms configured are used; otherwise a default table list applies).
  • In Customer Service, terminology/behavior tailoring is done through the Copilot Service admin center (Copilot settings) and by extending the "Customer Service Copilot Bot" agent in Copilot Studio (see #2).

Choose when: Add glossary/synonyms when users report Copilot misunderstanding org jargon or custom columns; use glossary for business concepts/logic, synonyms for column-name variants.

2. Design customizations of Copilot in Dynamics 365 apps for customer experience and service

Plain English: Beyond terminology, architects decide how far to tailor the embedded Copilots — which features to enable, which fields feed summaries, and how to extend answers with custom logic.

Key exam facts — Sales:

  • Customizable aspects of Copilot chat in D365 Sales: (1) add topics so Copilot learns from org data/content/processes, (2) customize the prompt guide (suggested prompts), (3) add glossary/synonyms.
  • Copilot in Sales capability set: record summarization (lead/opportunity/account), record catch-up/recent changes, meeting preparation, email assistance, news updates, information assistance from SharePoint docs, embedded opportunity summary widget and post-qualification lead summary.
  • Email assist ("Contextual email drafting with AI"): enabled per model-driven app in Power Apps → app Settings > Features → toggle Contextual email drafting with AI. Supported only for lead and opportunity records through Dynamics 365 email. Does not require a Microsoft 365 Copilot license.
  • Copilot in other model-driven apps can be customized via "Customize Copilot chat using Copilot Studio (preview)".

Key exam facts — Customer Service:

  • Admin surface = Copilot Service admin center (Productivity > Copilot settings). Configurable: turn on Copilot features (ask a question, email drafts, case summaries, conversation summaries), manage the fields used in case summaries, customize how conversation summaries are delivered, and display Copilot case summary on custom case forms.
  • Extend agent with Copilot Studio (preview): check Extend agent with Copilot Studio under Agent within Copilot > Customer Support > Settings. This exposes the underlying "Customer Service Copilot Bot" in Copilot Studio where you add tools, knowledge, agents, and topics. This supersedes the older "plugins for generative AI" model — existing plugins keep working, but changes should be made as Copilot Studio tools.
  • Context passing: default context (current record, user role, session metadata) is sent automatically; custom context (entity + data field + description) is configured in Copilot Service admin center on the Extend agent tab — the description is used by the orchestrator to decide how to use the context.
  • Older plugin management: Customer Service admin center > Agent experience > Productivity > Manage Plugins for Generative AI (preview), with Admin or Agent auth and access via all-Copilot-users or security roles.

Choose when: Prefer admin-center configuration for feature toggles/summary fields; extend via Copilot Studio when you need custom answers, external data (connectors/tools), or new conversational topics.

3. Design connectors for Copilot in Dynamics 365 Sales

Plain English: Extend Copilot for sellers (Sales agent in Microsoft 365 Copilot, formerly Copilot for Sales) with data from external sales applications by building Power Platform custom connectors plus connector actions in Copilot Studio.

Key exam facts:

  • Pattern: (1) build a Power Platform custom connector over the partner/ISV application API → (2) create a connector action in Copilot Studio referencing the connector → (3) a Sales administrator enables the action for users (Copilot Studio "Manage extensions for Sales in Microsoft 365 Copilot") → (4) optionally get the connector + action certified to make it available beyond your tenant.
  • Enabled actions can take up to 7 days to be ingested into Sales experiences; users can sign out/in to force refresh.
  • Governance caution: enabling a connector action for Sales may let users send/receive external data even if the same connector action is disallowed for Microsoft 365 directly — validate against org policy.
  • If "Sales" doesn't appear in Copilot Studio, the Sales enhanced Teams app must be deployed by an admin.
  • Cross-environment collaboration pattern (Dynamics 365 guidance): custom connector + Copilot plugin/tool lets Copilot in Customer Service summarize Sales data from another environment (e.g., "Summarize Deal" plugin), configured through the admin center wizard (auth = Admin or Agent; access = all Copilot users or security roles).

4. Design agents for integration with Dynamics 365 Contact Center channels

Plain English: Dynamics 365 Contact Center is Microsoft's Copilot-first CCaaS. Architects design where Copilot Studio agents plug into voice/chat/messaging channels, how work routes to humans, and which CRM sits underneath.

Key exam facts:

  • Contact Center capabilities: self-service with AI/autonomous service agents, IVR + AI agents, Customer Intent Agent, sentiment analysis, live transcription/translation, conversation summaries, unified routing across voice and digital channels, quality evaluations, Teams collaboration, supervisor analytics/whisper/barge-in.
  • Channels: voice (PSTN/SMS via Azure Communication Services), Live Chat, digital messaging (SMS, WhatsApp, Apple Messages for Business — requires Apple onboarding + Microsoft support step, etc.), and custom channels (e.g., Direct Line; requires a Bot Framework bot registered with Azure Bot Service as multitenant).
  • Copilot Studio is the low-code environment for building the self-service IVR and messaging agents used in Contact Center; agents hand off to representatives with full context (conversation history).
  • Unified routing: AI-infused classification and assignment using skills, sentiment, predicted effort, presence, capacity, customer criteria; percent-based routing, overflow, preferred-representative routing.
  • Deployment models (reference architectures): (a) full Microsoft stack (Customer Service Premium), (b) retain existing CRM + Dynamics 365 Contact Center (standalone) for routing/channels/AI, (c) retain existing CCaaS + Dynamics 365 Customer Service Enterprise as CRM, (d) existing CRM + existing CCaaS + Copilot for Service and Copilot Studio.
  • Voice caveats: the only customer engagement hub supporting voice agents is Dynamics 365 (Customer Service/Contact Center); Genesys, LivePerson, Salesforce, ServiceNow integrations support chat-only agents. For voice agents, enable the Telephony channel first, then connect to Dynamics 365. Multilingual voice agents published to Contact Center require the No authentication setting.
  • NLU+ language understanding option is available when channels are managed with a Dynamics 365 Contact Center license (processing occurs in Dynamics 365, vs plain NLU processed in Copilot Studio).

5. Design task agents

Plain English: Microsoft's agent taxonomy (Cloud Adoption Framework) describes a spectrum of three agent types: retrieval → task → autonomous, increasing in complexity and autonomy. Task agents do things — they perform specific actions within defined workflows rather than just answering questions.

Key exam facts:

  • Retrieval agents (a.k.a. productivity agents): access and synthesize information; knowledge tools only. Use cases: knowledge management, customer-service Q&A.
  • Task agents (a.k.a. action agents): perform specific actions within defined workflows — update records, trigger processes, create tickets — using knowledge tools + action tools. Moderate complexity; user-initiated. Use cases: service ticket creation, record updates, system monitoring, form filling.
  • Autonomous agents (automation agents): manage multi-step processes with minimal oversight; knowledge + action tools plus triggers that determine when to run/stop/escalate; require the most governance.
  • Platform fit: Copilot Studio (low-code SaaS) supports retrieval, task, and autonomous agents; Microsoft Foundry (PaaS, pro-code + low-code) supports all three with deep integration/custom logic; ready-to-use SaaS agents cover retrieval/task for personal productivity.
  • Design implication: a task agent = conversational agent + tools (connectors, flows, MCP, prompts) that execute actions; no event triggers needed. If the scenario needs background/event-driven operation → autonomous agent (see #6). If it only needs answers → prompt-and-response/retrieval (see #7).

Sources:

  • https://learn.microsoft.com/azure/cloud-adoption-framework/ai-agents/
  • https://learn.microsoft.com/azure/cloud-adoption-framework/ai-agents/technology-solutions-plan-strategy

6. Design autonomous agents

Plain English: Autonomous agents in Copilot Studio act without a user prompt — an event fires, the agent reasons over its instructions, and it executes tools/topics in the background.

Key exam facts:

  • Event triggers make an agent autonomous. Examples: email arrives (Outlook), item created in SharePoint, file created in OneDrive, Planner task completed, Dataverse row added/updated, Recurrence (schedule), and Dynamics 365 F&O business events (e.g., "purchase order confirmed").
  • Requires generative orchestration turned on. Trigger fires → trigger payload (event data + optional extra instructions, e.g., Use content from @{triggerBody()}) is sent to the agent → agent follows its instructions to choose tools/topics.
  • Authentication: event triggers can only use the agent maker's credentials (author authentication); each tool must be configured with working maker credentials for fully autonomous runs ("Maker-provided credentials"). Consider data-exposure implications when sharing the agent.
  • Available triggers are governed by Power Platform data policies (DLP).
  • Agents only react to triggers after publishing; monitor behavior in the Activity page/activity map (records topics/tools called and why).
  • Billing: autonomous operation consumes Copilot Credits ("autonomous actions" billing rate); enabling triggers changes billing calculation.
  • Design guardrails (guidance): scoped permissions, explicit decision boundaries, auditable processes; define trigger conditions carefully (avoid over-firing, missing context, quota exhaustion).
  • Deep reasoning can be layered on autonomous agents for complex decision steps (see #22).
  • Copilot Studio agents report / Viva Insights defines autonomous agents as "agents built in Copilot Studio with defined event triggers."

Choose when: the process is event-driven or scheduled, humans shouldn't have to initiate it, and actions are safe to automate with audit + (optionally) human-in-the-loop approvals via agent flows.

7. Design prompt and response agents

Plain English: The simplest agent pattern — a user asks, the agent answers (single-turn or lightly multi-turn Q&A), typically grounded on knowledge sources, with no actions taken in other systems. Equivalent to the "retrieval" end of the agent spectrum.

Key exam facts:

  • Implementation options: Copilot Studio agent with knowledge sources + generative answers only (no tools/triggers); declarative agent built with Agent Builder or SharePoint scoped to files/sites; Copilot answer control in canvas apps for a fixed Q&A.
  • Generative answers summarize retrieved content with citations; place at agent level or in the Conversational boosting topic; responses validated/moderated.
  • Governing settings: Allow ungrounded responses off → agent blocks answers not backed by a knowledge source/tool; Use information from the web adds Bing-grounded web knowledge.
  • Design considerations: data quality drives accuracy; plan for follow-up question behavior; use fallback for unmatched queries; cheapest/simplest pattern — no action tooling, no trigger governance.
  • Escalation path: pair with Escalate/fallback topics to hand off to humans when answers aren't found.

Choose when: requirement = "answer questions about X" with no system updates; choose task agents when actions are needed; autonomous when event-driven.

8. Propose Foundry Tools for a given requirement

Plain English: Foundry Tools is the 2026 name for the suite formerly called Azure AI Services (Cognitive Services) — prebuilt, customizable AI APIs surfaced through the Microsoft Foundry portal. Architects match tools to requirements.

Key exam facts — the catalog:

  • Speech — speech-to-text, text-to-speech (270+ neural voices, 119+ languages), speech translation, speaker recognition. Requirement cues: call transcription, voice bots, read-aloud.
  • Translator — text translation across 100+ languages/dialects.
  • Language — NLU: sentiment, key phrases, NER, PII detection, summarization, conversational language understanding (CLU), question answering. (Question answering can back Copilot Studio fallback/chitchat scenarios.)
  • Content Understanding — multimodal analysis of documents, audio, video, images into structured outputs (newer service; requirement cue: "analyze mixed media content").
  • Document Intelligence — extract text, key-value pairs, tables, structure from documents; prebuilt models (invoices, receipts, contracts) + custom trained models. Requirement cue: invoice/forms processing at scale.
  • Vision — image analysis (objects, faces, OCR/Read, tags, captions); Custom Vision for custom image classification/detection.
  • Azure AI Search — AI-powered retrieval with vector/semantic search; the standard grounding/RAG index (also a Copilot Studio knowledge source).
  • Content Safety — detect harmful/unwanted content (text + image); guardrails for gen-AI apps.
  • Immersive Reader — reading comprehension aid.
  • Also in Foundry portal: Azure OpenAI in Microsoft Foundry Models for generative models.
  • Retirements to know: Anomaly Detector retires October 1, 2026 (no new resources since Sept 2023); LUIS and QnA Maker superseded by Language (CLU / question answering). Don't propose retired services for new solutions.
  • Deployment traits: REST APIs/SDKs; many available as on-premises containers for compliance; regional availability varies; some support custom training (bring your own data).

Choose X when: speech interface → Speech; multilingual text → Translator; extract fields from invoices/forms → Document Intelligence; classify/understand video+audio+docs uniformly → Content Understanding; image inspection → Vision/Custom Vision; enterprise search/RAG grounding → Azure AI Search; moderate user content → Content Safety.

9. Propose code-first generative pages and the use of an agent feed for apps

Plain English: Generative pages let makers/developers create fully functional React pages in model-driven apps by describing them in natural language; a code-first path lets pro developers build them in external AI coding tools. Agent feed surfaces agent-generated tasks inside model-driven apps so humans can supervise agent work.

Key exam facts — generative pages:

  • AI generates React/TypeScript code covering UX + business logic against specified Dataverse tables; iterative conversational refinement; can attach an image of the desired layout.
  • Two development approaches: (1) Power Apps (make.powerapps.com) conversational designer — available in United States, Great Britain, Australia, Singapore regions; GPT-4.1 is the model in the maker experience; (2) code-first with AI code generation tools (e.g., GitHub Copilot CLI, Claude Code) — local development, direct TypeScript/React access, CLI deployment, available worldwide on public clouds; GA April 2, 2026; pages pushed back integrate fully with the designer (restore, undo, code compare, code editing) and get AI-generated names.
  • Designer features: thought streaming → code generation → transpilation → final rendering; Code tab view/edit; Compare iterations (diff); attach screenshots; Accessibility assistant with Auto fix; Save and Publish. Maker remains responsible for validating generated code.
  • Code-first pages use the documented dataApi object for Dataverse data operations.

Key exam facts — agent feed (preview):

  • Surfaces agent-generated, actionable tasks in model-driven apps for users to review/validate/complete — the app becomes a human–agent collaboration surface.
  • Prerequisite: at least one agent uses the Power Apps MCP server and is supervised in the app (agents added via "Add agents to an app"). From May 1, 2026, only agents using the Power Apps MCP server are supported (replaces older Copilot Studio activity-based feed).
  • Current limits: preview; English only; all users with access to the Agent Task table see feed items — avoid logging user-targeted tasks with sensitive data.

Choose when: propose generative pages for bespoke UX beyond standard forms/views with low-code speed (code-first variant when devs want frontier models, source control, local tooling); propose agent feed when autonomous agents produce work items requiring human review inside the app.

10. Design topics for Copilot Studio, including fallback

Plain English: Topics are authored conversation paths. Architects design custom topics, tune trigger phrases, and configure system topics — especially Fallback — so unrecognized queries are handled gracefully.

Key exam facts:

  • Topic anatomy: authoring canvas of nodes (message, question, condition, action/tool, redirect, variable management); two types — system topics (built-in, can't create or delete, can disable/customize/reset to default) and custom topics.
  • Trigger phrases: 5–10 sample phrases recommended per topic; NLU matches semantically, not exactly. Behavior varies by conversation state (start of conversation, after "did you mean," topic switching during slot filling).
  • System topics list: Conversation Start, Conversational boosting (generative answers when no topic matches), End of Conversation, Escalate (triggers on "talk to agent" or Escalate event), Fallback, Multiple Topics Matched ("did you mean"), On Error, Reset Conversation, Sign in.
  • Fallback: triggers On Unknown Intent; has no trigger phrases; default flow asks the user to rephrase (max two attempts) then redirects to Escalate. Priority: Conversational boosting triggers before Fallback. The UnrecognizedTriggerPhrase variable carries the raw utterance — pass it to a flow/skill/external system or a generative AI model. Reset to default is possible.
  • Fallback design patterns (guidance): offload large single-turn Q&A sets externally (agent limit 1,000 topics); call Azure AI Language question answering (incl. chitchat personality); mine fallback utterances for topic enrichment (add trigger phrases / new topics).
  • Generative orchestration caution: mixing generative orchestration with topics containing generative answers nodes can produce duplicate answers (orchestrator's Universal Search Tool + topic both answering).
  • Teams channel notes: Conversation Start runs only once per user; no default fallback system topic in Teams (create one).
  • Topic management: On/Off status (Off topics still publish but never trigger), copy topics, redirect between topics.

11. Design data processing for AI models and grounding

Plain English: Grounding = giving the model trusted enterprise data at answer time (RAG) instead of relying on trained knowledge. Architects design which knowledge sources to use, how data is indexed/embedded, and how retrieval + response generation is validated.

Key exam facts — RAG in Copilot Studio (per knowledge source):

  • Dataverse tables: max 15 tables per agent; support synonyms + glossary; natural language transformed into analytical queries over structured data; Entra ID delegated auth.
  • SharePoint/OneDrive & files: document grounding with citations.
  • Microsoft Graph connectors: enterprise sources indexed into the Graph (ServiceNow KB, Confluence, custom sites); requires Entra-authenticated delegated calls; "Enhanced search results" uses tenant Graph grounding.
  • Real-time connectors: live data from Salesforce, ServiceNow, Zendesk, Azure SQL; logged-in user creates connections.
  • Azure AI Search: returns results from a linked vectorized index; connection is not delegated — no security trimming; add via a formal data connection (auth: access key, client certificate, service principal, Entra Integrated); one vector index per connection.
  • Custom data: query any system via flows/HTTP, pass a table (Content, optional ContentLocation, Title) into a generative answers node.
  • Pipeline stages: query understanding → retrieval → summarization/response generation (instructions, citations, personalization) → safety/governance validation (moderation, grounding validation); customer data never trains the models.
  • Node-level knowledge (generative answers node) takes priority over agent-level knowledge (agent-level acts as fallback).
  • For custom index design (Azure Well-Architected AI workloads): dedicated grounding data design area — chunking, enrichment, embeddings, index freshness/refresh, security filtering; plus training data design, data platform selection.
  • Bing options: public websites (Grounding with Bing Search) and Bing Custom Search at agent or node level.

Choose when: Dataverse for structured business records; Graph connectors for enterprise content with security trimming; Azure AI Search for large custom corpora/vector semantics (accepting no per-user trimming); custom data for anything else.

14. Determine when to use standard NLP, conversational language understanding, or generative AI orchestration in Copilot Studio

Plain English: Copilot Studio agents interpret user input through either generative orchestration (LLM plans across topics/tools/knowledge) or classic orchestration with one of three NLU models: built-in NLU, NLU+, or Azure CLU.

Key exam facts:

  • Generative orchestration (default for new agents): selects topics by description, chooses tools by name/description, proactively searches knowledge, handles multiple intents per utterance, auto-generates slot-filling questions and final responses. Limits: 5 messages per topic/action chain, 128 topics/actions for triggering. Required for MCP, event triggers, computer use, deep reasoning. Admin can disable per environment.
  • Classic orchestration: deterministic — topic triggered by trigger-phrase match; knowledge is fallback; tools only called from within topics; single topic per response.
  • NLU (built-in): default classic model; 5–20 phrases/topic; RegEx/closed-list custom entities; no annotation needed; single intent per query; can't be extended; too much training data increases latency.
  • NLU+: high accuracy for large enterprise agents (many topics/entities/training samples); precompiled model must be trained before production; entity annotations in trigger phrases boost accuracy; training data also optimizes speech recognition for voice agents; requires Dynamics 365 Contact Center license (runtime processing in Dynamics 365).
  • Azure CLU: bring your own CLU model (Azure subscription + costs); more languages via native models, industry-specific vocabulary, advanced entity extraction (multiple entities of the same type, silent extraction); CLU intents must be manually kept in sync with Copilot Studio topics; single intent per query.
  • Decision criteria: need repeatable/regulated, fully controlled dialogs → classic; complex multi-intent conversations spanning knowledge + actions → generative; enterprise IVR/voice accuracy at scale + Contact Center licensing → NLU+; non-English/industry NLU with existing Azure investment → CLU.
  • Combination: with generative orchestration you can still pick NLU or NLU+ for voice agents in Contact Center IVR setups.

20. Design agent extensibility with Model Context Protocol in Copilot Studio

Plain English: MCP is an open protocol that lets an agent consume tools/resources published by an MCP server — integrations defined once on the server, discovered dynamically by every agent.

Key exam facts:

  • MCP servers expose Resources (file-like data/context), Tools (callable functions), Prompts (templates). Copilot Studio supports tools and resources (resources must be returned as outputs of an MCP tool).
  • Requires generative orchestration. Server-published tool definitions (name, description, inputs, outputs) are dynamically reflected — updates/removals propagate to agents without republishing.
  • Connection options: MCP onboarding wizard (recommended; server name/description/URL) or a custom connector via Power Apps; also register a server through Agents 365 (M365 admin center BYO MCP). Auth: None, API key (header or query), OAuth 2.0 (dynamic discovery with DCR, dynamic, or manual).

  • Tool selection control: "Allow all" toggle vs enabling individual MCP tools (new server tools default off when Allow all is off).
  • Governance: MCP connectivity rides Power Platform connectors → DLP policies apply.
  • Prebuilt Microsoft MCP servers: Dataverse, Dynamics 365 ERP MCP (F&O data + business logic under agent's security role; replaces retired 13-static-tool version), Dynamics 365 Customer Service MCP Server (preview), Work IQ MCP (Mail, Calendar, Teams, SharePoint, Word, etc. — requires M365 Copilot license), plus third-party (Salesforce, JIRA, GitHub...).
  • When MCP vs connectors/REST: choose MCP for standardized, centrally managed tools shared by multiple agents and for frequently changing upstream APIs (update server once, all agents follow). Choose direct connector/REST for rapid prototyping or when no MCP server exists. Limitations: can't enrich tool descriptions with extra invocation context; topics can't call MCP tools directly.

27. Orchestrate the configuration of Microsoft 365 Copilot for Sales and Microsoft 365 Copilot for Service

Plain English: These bring CRM context into Outlook/Teams/M365 Copilot. Naming note: Copilot for Sales now largely surfaces as the "Sales agent"/"Sales in Microsoft 365 Copilot"; Copilot for Service similarly surfaces as an agent experience.

Key exam facts — Sales:

  • Supported CRMs: Dynamics 365 Sales and Salesforce Sales Cloud. Users must sign in to the CRM from the Sales app in Outlook or Teams.
  • Setup: admin installs the Sales agent/app in Outlook and Teams; Sales Chat must be turned on in Access settings; open Sales Chat settings once to initialize the CRM connection; configure CRM entities (Dynamics 365 tables / Salesforce objects) the agent can query — default D365 tables include Lead, Opportunity, Account, Contact, Product, Quote, Invoice, Email, Competitor... (tables with Copilot Studio synonyms take precedence); Forms settings control record types/fields for save/edit experiences; privileges assigned in CRM (Salesforce admins assign the security role in the msdyn_viva Power Platform environment).
  • Capabilities: CRM-enriched email summaries in Outlook (requires >1,000 characters, supported language, not encrypted; new Outlook/web for combined experience; save summary to CRM record — once only; AI-ranked opportunity suggestions), save Outlook activities to CRM, create contacts, meeting prep, Teams meeting summaries; conversational Sales agent in M365 Copilot chat (also embedded in Word/Excel/PowerPoint/Outlook/Dynamics 365 — preview, chat only).

Key exam facts — Service:

  • Copilot for Service embeds Copilot into existing CRM/contact center solutions (Salesforce, ServiceNow, Zendesk) and Outlook/Teams; reference architecture: existing CRM + existing CCaaS + Copilot for Service + Copilot Studio for IVR/self-service.
  • Extend with Copilot Studio: in Copilot for Service select "Extend with Copilot Studio" — edit topics, add Power Automate connectors/flows, publish; anything Copilot Studio supports can extend the agent.

28. Propose Microsoft Power Platform AI features, including AI hub

Plain English: AI hub is the home for AI Builder capabilities inside Power Apps and Power Automate — prebuilt models, custom models, and prompts — used to infuse AI into apps and flows.

29. Design interoperability of the finance and operations agent chats to use additional knowledge sources

Plain English: The F&O in-app Copilot chat (sidecar) and related agents can be extended with additional structured knowledge so users can Q&A over ERP data — and that same data can feed custom agents and Microsoft 365 Copilot.

Key exam facts:

  • Three Copilot surfaces in F&O apps: (1) app sidecar chat, (2) embedded AI in workflows/pages, (3) Microsoft 365 chat integration (ask about F&O data from Teams/Outlook).
  • Chat with finance and operations data (preview): add F&O structured data as knowledge: use virtual entities for finance and operations apps as knowledge sources, or native Dataverse tables synchronized from F&O via Dual-write; add these as Dataverse knowledge sources to (a) Copilot for finance and operations apps agent (extends the in-app sidecar Q&A), (b) custom Copilot Studio agents, or (c) Microsoft 365 Copilot via a declarative agent (add knowledge + actions, publish to M365 Copilot).
  • Actions via MCP: add the Dynamics 365 ERP MCP server to an agent (Copilot Studio > Tools > Add a tool > Model Context Protocol) for data + business-logic operations respecting the agent's security role and environment context; the older 13-static-tool ERP MCP server is retired.
  • Autonomous triggers: F&O business events (e.g., purchase order confirmed) can trigger Copilot Studio agents ("When a Business Event occurs Fin & Ops Apps" trigger; category/business event/legal entity selection; @{triggerBody()} context).
  • Design considerations: virtual entities avoid data duplication (live queries); dual-write gives Dataverse-local copies for richer knowledge behavior; declarative agent = the interoperability vehicle into M365 Copilot.

30. Recommend the process of adding knowledge sources to in-app help and guidance for Dynamics 365 Finance or Supply Chain Management apps

Plain English: "Generative help and guidance with Copilot" is the in-app help sidecar in F&O apps, grounded by default on public product documentation; architects recommend how to add company-specific knowledge.

Key exam facts:

  • What it is: conversational sidecar (Copilot button on every page) answering how-to questions; powered by a Copilot Studio agent deployed into the Dataverse environment linked to the F&O environment; pipeline = rephrase question → search indexed public docs → synthesize answer; thumbs up/down feedback (+ optional written feedback feature).
  • Enablement: typically available by default when Power Platform Integration is enabled (Lifecycle Services); otherwise follow "Enable Copilot capabilities in finance and operations apps"; default knowledge = published product documentation for F&O apps, honoring user language preferences.
  • Adding knowledge — the recommended process: the help experience uses the out-of-box Conversational boosting topic (generative answers) in the Copilot for finance and operations apps agent in Copilot Studio; (1) for file-based knowledge, upload files (PDF, RTF, Word) to Copilot Studio as knowledge on that agent; (2) for other source types (e.g., SharePoint or other data sources), add your own topic to the agent; (3) publish.
  • Caution: the conversational boosting topic assumes all knowledge relates to product functionality — adding unrelated knowledge can produce answers that wrongly imply relevance to F&O usage. General knowledge can also be enabled.
  • This is distinct from #29: in-app help/guidance knowledge (docs, files) vs data Q&A knowledge (virtual entities/Dataverse tables).
