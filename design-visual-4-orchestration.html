Dynamics 365 Finance & Supply Chain

1 · The three Copilot experience types & the plumbing

Every F&O AI question starts with which experience type is in play — and whether the plumbing prerequisites are met.

Three experience typessidecar (in-app chat panel) · embedded (AI inside pages) · outside (external agents over F&O data).
The plumbingAI chat control + Copilot Studio as the orchestration hub + Dataverse — F&O Copilot is a Copilot Studio agent under the hood.
Prerequisite switchesPower Platform Integration enabled per environment · tenant setting “Publish bots with AI features” · per-feature enablement in feature management.
Newer surfacesImmersive Home (work with agents + prioritized work items) · in-app Agent management · Dynamics 365 ERP MCP server (data ops + business logic without custom code).

2 · Adding knowledge to in-app help — the decision tree

Direct exam bullet: “recommend the process of adding knowledge sources to in-app help for D365 Finance / SCM.” The source type decides the path.

Default groundingthe sidecar Q&A is grounded in public Microsoft Learn docs out of the box.
File pathCopilot Studio → the environment's “Copilot for finance and operations apps” agent → Knowledge → upload files (PDF/RTF/Word) → test → Publish; answers flow through the built-in conversational boosting topic.
SharePoint pathSharePoint or other source types require creating your own topic in that agent — they don't ride the default boosting topic.
The virtual-entity trapDataverse virtual entities are NOT supported as knowledge for generative help — they power chat with data, a different feature.
AI general knowledgeoptional model + Bing answers are used only after custom sources are exhausted.

3 · Agent-chat interoperability — one dataset, three surfaces

Direct exam bullet: “design interoperability of the finance and operations agent chats to use additional knowledge sources.”

The interop patternexpose F&O data via virtual entities (mserp_*) or dual-write tables → add as Dataverse knowledge.
Three consuming surfaces(a) the F&O in-app agent, (b) custom Copilot Studio agents, (c) declarative agents in M365 Copilot — same knowledge, three chats.
Security constantevery surface answers only within the querying user's data access — interop never bypasses permissions.

F&O-side agents deploy through a wizard in the PPAC Copilot hub — the steps are testable, and so is the roster.

Customer experience & service orchestration

5 · Orchestrating AI features in CE & service apps

The admin-orchestration view of what page 1 covered in design depth — switches, admin centers, agent prerequisites.

Master switchesPPAC “Enable AI-powered Copilot features” per environment (service) · PPAC Copilot hub “turn on AI for D365 Sales” (sales) — always the first checkbox in an orchestration sequence.
Admin surfacesSales: Sales Hub App Settings → Copilot · Service: Copilot Service admin center → Productivity · rollout to rep groups via experience profiles.
Cross-geo consentregions without a local Azure OpenAI endpoint need cross-region data-movement consent before Copilot features light up.
Agent prerequisites recapCustomer Service agents need pay-as-you-go billing; Customer Intent Agent needs the Intent Manager role; Knowledge Management needs configured knowledge + connection references + flows + published agent.
Sales Qualification recapAzure app registration → Dataverse application user with AISalesPerson; Research and engage needs a shared mailbox + server-side sync; one mode per org; BANT handoff.

Microsoft 365 Copilot for Sales & Service

6 · Copilot for Sales — configuration chain

Direct exam bullet: “orchestrate the configuration of Microsoft 365 Copilot for Sales.” Seller assistant in Outlook/Teams, CRM behind it.

What & wherethe Sales agent lives in Outlook, Teams, and M365 Copilot Chat, connected to Dynamics 365 Sales or Salesforce.
Deployment chainM365 admin center integrated apps (fixed/auto-deploy) → Teams setup policy to pin → enable transcription for meeting summaries → each user needs an M365 Copilot license.
Configuration reuseglossary is shared with Copilot in D365 Sales; configure summary fields once for both.
ExtensibilityCopilot Studio connector actions + partner insights + enrichment APIs extend seller experiences.

7 · Copilot for Service — AI without rip-and-replace

The pitch is the design: keep the incumbent CRM, layer generative AI on top of existing knowledge.

Core promisegenerative AI for service reps without rip-and-replace — grounds on existing websites, SharePoint, KBs and files.
Out-of-the-box CRM integrationsSalesforce, ServiceNow, Zendesk + Dynamics 365 — the differentiator vs rebuilding on a new platform.
Where reps feel itOutlook/Teams experiences: case summaries, drafted replies, updating CRM records in the flow of work.
Extending it“Extend with Copilot Studio” adds custom topics and actions on top.
Related: Service Agentan M365 Copilot agent for D365 Customer Service reps — prioritize cases, summarize, retrieve knowledge, case write actions.

8 · Propose the right Microsoft 365 agent — the matching wall

Direct exam bullet: pure matching questions. Scenario keyword → agent, eight times.

Researcherdeep multi-step research over web + work content with cited, shareable reports; first-party under Copilot Chat “Tools”, auto-available to licensed users.
Analystadvanced data analysis over datasets and spreadsheets — the “analyze this data” keyword.
Facilitatorreal-time collaborative meeting notes + in-meeting Q&A in Teams; notes visible to all participants.
Channel AgentTeams channel collaboration: deadlines, status summaries, task assignment.
SharePoint agentssite/library-scoped Q&A, created by end users, permission-trimmed, one approved agent per site.
Sales agent / Sales ChatCRM-connected selling in the flow of work (see §6).
Planner agent / Project Managerplan creation and task execution in Planner (premium).
Skill coachesPrompt Coach, Idea Coach, Writing Coach… — upskilling scenarios.
Governance layerM365 admin center (enable/disable/assign/block) · Agent Store for distribution · Microsoft Agent 365 as the cross-origin control plane.
💡 Hook: research → Researcher · data analysis → Analyst · meetings → Facilitator · site Q&A → SharePoint agent. Don't over-engineer: if a first-party agent matches, never propose a custom build.

Power Platform AI features

9 · AI hub, AI Builder & the plan designer

Direct exam bullet: “propose Microsoft Power Platform AI features, including AI hub.”

AI hub — wherein the Power Apps and Power Automate maker portals (“… More → AI hub”) — the entry point to AI models and Prompts.
AI Builder prebuilt modelsreceipt/invoice/business-card processing, text recognition, sentiment… — consume instantly from apps, flows and agents.
AI Builder custom modelsdocument processing, prediction, category classification, entity extraction, object detection — train on your data; requires Dataverse, enabled per environment.
AI Builder governancelicensed via AI Builder credits; admins can disable preview models per environment.
Prompts are solution-awaregenerative prompts grounded in Dataverse, multimodal inputs, reusable across copilots, apps (Power Fx), flows and Dataverse.
Plan designer (GA)“copilot-first development”: describe the business problem (+ process images) → generates roles, user stories, data model and the solution set (tables, apps, Power Pages, flows, Copilot Studio agents, Power BI); can also generate a plan from an existing solution; the plan doubles as documentation.
When to propose plan designergreenfield envisioning and requirements-to-solution acceleration — the answer to “turn a described business problem into a scaffolded Power Platform solution.”
