The Microsoft AI landscape

five platforms, one ecosystem
In brief: agentic solutions are almost never built on one tool — the exam repeatedly hands you a scenario and asks for the right combination of platforms. Know the five platforms cold, see how a single customer request can flow through all of them in one afternoon, and internalize the trade-offs (speed vs customization, cost vs capability, simplicity vs scale) that scenario answers hinge on. Business alignment beats deep tool expertise.

1 · The five-platform map

Know each platform's one-line identity and the moment it's the right choice — that's the core of what the exam tests.

⚡Copilot Studiothe no-code agent builder — quick deployment, easy to use, builds all three agent types. Idea → deployed agent in hours, not months.
🏭Azure AI Foundrythe heavy machinery — custom tools & models, fine-tuning, enterprise-scale orchestration. Go here when off-the-shelf stops cutting it.
🔌Power Platformthe integration layer / glue — hundreds of connectors and automated flows bridging agents to back-end systems. How agents actually get work done.
🏢Dynamics 365domain-specific apps — sales, service, finance, supply chain — each with embedded, purpose-built AI that understands CRM/ERP out of the box.
📎Microsoft 365 CopilotAI where users already work — Teams, Outlook, Word, PowerPoint. Meet people in their natural workflow.

The three agent types

organized by autonomy
In brief: Microsoft defines three agent types by how much autonomy they have — prompt-and-response (waits to be asked), task (executes bounded actions with tools, usually human-initiated), and autonomous (plans, reasons, and acts independently across steps). Higher autonomy means more power and more risk, so governance scales with it. A three-question sorter classifies any scenario, and each type maps onto specific platforms.

4 · The autonomy ladder

Same ladder the exam climbs in its questions — with the risk curve riding alongside.

7Prompt-and-response = low autonomy: processes input, maybe reasons, answers — never acts on its own.
8Task agents = medium autonomy: execute bounded tasks with tools (escalate, look up, trigger a workflow) but typically need a human to start them.
9Autonomous agents = high autonomy: plan, reason, and act independently over multiple steps — e.g., proactively watch inventory, flag issues, adjust orders.
10Higher autonomy = more power = more risk. An autonomous agent in a scenario with no guardrails mentioned is a red flag answer.

5 · The three-question sorter

Ask these in order and any scenario classifies itself.

🧠 Hook: the classic exam scenario — "proactive supply-chain agent that monitors inventory and flags issues before they escalate" → autonomous (works without user input). Platform: Copilot Studio if simple, Foundry if custom models or complex reasoning are mentioned.

Copilot Studio core concepts

topics · tools · connectors · knowledge
In brief: Copilot Studio structures no-code agent building around four components — topics (which conversations it handles), tools/actions (what it can do), connectors (how it reaches data — 1,000+ prebuilt), and knowledge (what grounds its answers). In a worked customer-support build, the instructions panel acts as the guardrail/system prompt (the user can never see or override it), knowledge comes from an uploaded product catalog, and two Dataverse-backed workflows check order status and create support tickets. Exam answers that mention grounding + DLP are usually the keyed ones.

Each block answers one question about the agent — and a single support request touches all four in order.

8 · Built-in guardrails — and how they look in practice

Copilot Studio ships with governance features the exam loves to see in answers — here's where they live in a real build.

🚫Content filtersblock harmful outputs before they reach users.
🔒DLP policiescontrol what data can flow where.
🧪Environment isolationseparate dev / test / production environments.
👤Role-based accessRBAC over agents and environments.

🛠 In practice — the instructions panel IS the guardrail

  • The agent's instructions work like a system prompt: purpose, allowed and forbidden behavior.
  • Users can't see or override it — your defense against prompt-injection / hijacking.
  • Example configuration: no internet, no general knowledge — only the uploaded product catalog.

🛠 In practice — tools are just workflows

  • Two Dataverse-backed workflows: get order status and create support ticket — the agent parses parameters from the user's message.
  • "Order 2004 was delivered but I didn't get it" → status lookup plus automatic ticket creation.
  • Activity traces show each step — debugging matters; expect to rework auto-generated scaffolding by hand, especially on the database side.
15Four built-in protections: content filters · DLP · environment isolation · RBAC.
16Exam tip: answers pairing knowledge grounding + DLP are the "safer, smarter choice" — the exam rewards governance thinking.
17Use Copilot Studio when speed and simplicity are the priorities — don't overcomplicate what it can already handle.

Azure AI Foundry

custom models & enterprise scale
In brief: Foundry is Microsoft's centralized platform for model discovery, hosting, and customization — a catalog of ~11,000 models (Azure OpenAI, Anthropic Claude, Meta Llama, Mistral and more), fine-tuning tools, and an Agent Service that hosts custom agents (built with frameworks like LangChain or Semantic Kernel) with scaling, observability, and cost tracking built in. In the model catalog you can filter by capability, compare models, and see which ones expose fine-tuning.

MCP & Agent2Agent

the open protocols
In brief: two open standards prevent vendor lock-in and make enterprise agentic systems possible. MCP (Model Context Protocol, originated by Anthropic, adopted by Microsoft) is how agents securely reach tools, APIs, and context through a governed intermediary server instead of hard-coded endpoints. A2A (Agent2Agent, originated by Google, supported by Microsoft) is how agents communicate, delegate, and collaborate across platforms. Different problems, different protocols — and complex exam answers usually need both.

11 · MCP — the governed doorway to tools

Instead of fragile hard-coded API calls, every tool request goes through an MCP server you control, log, and reuse.

21MCP is an open industry standard — originally developed by Anthropic, adopted by Microsoft across its AI platforms. No vendor lock-in.
22Four wins over direct API calls: secure central control · auditability · reusability · interoperability (works across Studio, Foundry, custom code).
23Exam pattern: "Foundry agent needs secure access to a private database?" → register an MCP server that owns the connection, credentials, and access control.

When to build custom models

build vs buy vs extend
In brief: the pragmatic hierarchy: off-the-shelf model + prompt engineering + RAG solves most business problems — escalate to fine-tuning only when that provably fails for a specific reason (domain accuracy, privacy control, cost at scale, genuinely unique data patterns). When custom is justified, Foundry's six-step pipeline runs from base-model selection through rigorous evaluation to MCP-exposed deployment. The exam punishes answers that jump straight to custom; its mantra is "start simple, escalate thoughtfully, with strong governance and evaluation."

13 · The escalation ladder — and the four legitimate reasons to climb it

Stay on step one until a named requirement forces you up.

⚖️Domain accuracy criticallegal, financial, medical vocabulary that generic models consistently get wrong.
🔐Privacy / compliance controlmodel stays on your infrastructure, your audit trails, no data leaving.
💰Cost at scalethousands of queries/day — a small fine-tuned model can beat per-token premium pricing (if the math works).
🧬Proprietary patternsinternal jargon and business logic no public model saw — genuinely rare.
26The exam punishes rushing to custom. "Custom sounds impressive" is never a reason; a failed simpler approach is.

14 · The six-step custom-model pipeline

Foundry's fine-tuning workflow — with evaluation as the step you may never skip.

27Evaluate first, deploy second, monitor continuously — deploying without evaluation is always a wrong answer.
28Justify custom with business trade-offs — cost, governance, performance — never with novelty.
