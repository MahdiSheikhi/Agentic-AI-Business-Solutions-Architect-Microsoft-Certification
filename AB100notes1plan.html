A. Analyze requirements for AI-powered business solutions

1. Assess the use of agents in task automation, data analytics, and decision-making

Plain English: An AI agent is software that uses a generative AI model as a reasoning engine to interpret inputs (user messages, events, other agents' messages), decide what to do, and act — unlike traditional automation that follows fixed rules. The architect's first job is deciding whether a use case needs an agent at all, and if so, what type.

Key exam facts:

  • Five core agent components (CAF): generative AI model (reasoning engine), instructions (scope/boundaries), retrieval/knowledge (grounding data), actions/tools (functions, APIs), memory (conversation history and state).
  • Agent spectrum (increasing autonomy/complexity): (1) Retrieval/productivity agents — retrieve and synthesize information to speed decisions (Q&A, knowledge management); (2) Task/action agents — perform specific actions in defined workflows (create ticket, update record); (3) Autonomous/automation agents — manage multi-step processes with minimal oversight, using triggers to run/stop/escalate (supply-chain optimization); these need the most rigorous governance. The Copilot Studio design vocabulary also uses "prompt-and-response agents, task agents, autonomous agents."
  • Agent vs RAG: standard RAG follows a deterministic retrieval pipeline; an agent uses the model to decide which knowledge/tools to use at each step — enabling multi-step reasoning but introducing nondeterminism that requires testing and governance.
  • When NOT to use an agent (CAF decision tree): if the task is structured/predictable with clear repeatable steps, use regular code, Power Automate, or nongenerative AI models (faster, cheaper, more reliable). If the need is static knowledge retrieval over a fixed document set, use a classic RAG solution (FAQ bot). Agents fit when workflows are ambiguous, inputs are unstructured, and dynamic tool selection/reasoning is required.
  • Agent flows vs agents in Copilot Studio: topics and agent flows are deterministic (same input → same output); agents with generative orchestration are non-deterministic. Agent flows are billed via Copilot Credits (no Power Automate license needed); cloud flows need Power Automate licensing. Power Automate cloud flows can be converted (one-way) into agent flows.
  • Business benefits used to justify agents: efficiency (automate repetitive low-value tasks), speed (faster processing/decisions), scalability (elastic handling of demand spikes).
  • Use-case identification method (CAF Strategy): start from business problems → translate to use-case statements → classify as individual work (→ Microsoft 365 Copilot) vs business automation (→ agents/integration) → choose generative (non-deterministic, unstructured input) vs nongenerative AI (deterministic, structured input, prediction/anomaly detection).

Choose X when: deterministic, rules-based process → Power Automate/code; static Q&A → RAG; ambiguous multi-step work needing reasoning + tools → agent; high-risk autonomous actions → autonomous agent + human-in-the-loop guardrails.

2. Review data for grounding (accuracy, relevance, timeliness, cleanliness, availability)

Plain English: Grounding is supplementing a model at inference time with data it wasn't trained on so responses are accurate and business-relevant — usually via RAG. Before building, an architect reviews whether candidate data is good enough to ground agents.

Key exam facts:

  • Definitions (Well-Architected AI guidance): source data (production data — structured, semi-structured, unstructured); grounding data (information supplied to the model at inference, combined with the user query into the prompt); behavioral instructions (style/tuning data); user/turn data (session state — retain as briefly as possible).
  • RAG pipeline concerns: data preparation, chunking vs full-document grounding (tradeoff: full documents = accuracy but high token cost/latency; chunks = scalable but design-sensitive), embeddings/vector search, indexing, reranking, citations.
  • The five review dimensions (exam wording): accuracy (authoritative, correct, tested for groundedness — metrics like groundedness, answer relevance, evidence relevance); relevance (content matches the use case; irrelevant knowledge degrades answers — e.g., adding unrelated content to a Copilot conversational-boosting topic causes misleading answers); timeliness (fresh data; test for stale data/versioning mismatches; in dynamic domains always use the most recent sources; connectors sync on schedules; real-time connectors avoid staleness); cleanliness (deduplicated, well-structured, chunk-friendly, no conflicting versions; noisy data raises hallucination risk); availability (data is reachable and permissioned — agent-user authentication means the agent only surfaces content the user can access; Dataverse search must be enabled; indexes must scale under load).
  • CAF agent guidance: retrieval "provides the grounding data and context required for accurate responses — access to relevant, high-quality data is critical for reducing hallucinations."
  • Testing guidance: functional/integration tests of pipelines, index schema compatibility, freshness/quality checks, index load testing, security testing of ACL-partitioned documents.
  • Groundedness detection (Foundry Tools content safety) can verify responses are supported by sources; Copilot Studio performs grounding, provenance, and semantic-similarity checks and cites sources; the "Allow ungrounded responses" setting loosens citation restrictions.

3. Organize business solution data to be available for other AI systems

Plain English: Make enterprise data "AI-ready" so multiple AI systems (M365 Copilot, Copilot Studio agents, Foundry agents, Fabric data agents) can consume it with governance intact.

Key exam facts:

  • Unified data platform (CAF data architecture for AI agents): Microsoft Fabric OneLake is the central data lake; data domains publish governed data products; agents consume them through Fabric IQ, Foundry IQ (shared knowledge/context layer in Microsoft Foundry), and Copilot Studio — enforcing governance and security boundaries.
  • Dataverse: the structured business-data platform underlying Dynamics 365 and Power Platform; serves as a knowledge source with built-in RAG ("uses a retrieval-augmented generative technique in Dataverse"); requires Dataverse search enabled. Dataverse Link to Fabric exposes Dataverse/D365 data in OneLake in near real time with no ETL/copy (shortcuts, not data movement). Medallion (bronze/silver/gold) reference architecture integrates Dataverse with enterprise data in Fabric.
  • Microsoft 365 Copilot connectors (formerly Graph connectors): bring external/LOB content into the Microsoft Graph semantic index so M365 Copilot, Microsoft Search, Copilot Studio agents, and Researcher can use it. Two models: synced connectors (ingest + semantically index into Graph; items carry content, metadata, and ACLs; permission-trimmed; continuous sync) and federated connectors (MCP-based, query-time fetch, no data movement — for dynamic or regulated data that must stay in source; read-only; no semantic indexing). 100+ prebuilt connectors (ServiceNow, Salesforce, Confluence, etc.); custom connectors via the Copilot connectors API (app registration → connection → schema → ingest items with ACLs).
  • Copilot connectors vs Power Platform connectors as knowledge: Copilot connectors = search-and-index (broad, searchable, citation-friendly knowledge); Power Platform connectors = live API bridges for real-time facts and transactions (also usable as real-time knowledge in Copilot Studio, preview).
  • OneLake shortcuts and mirroring unify data without copying (ADLS, S3, Google Cloud Storage, Dataverse); shortcut transformations (including AI-powered ones: summarize, sentiment, PII redaction) make files analytics-ready.
  • Governance cross-cutting: Microsoft Purview (labels, lineage, DSPM for AI), RBAC, certified semantic models. Apply sensitivity labels so protection follows content into Copilot.
  • Fabric data agents provide conversational Q&A over OneLake (lakehouses, warehouses, semantic models, KQL DBs) and can be plugged into Foundry, Copilot Studio, and M365 Copilot as domain-expert agents.

B. Design overall AI strategy for business solutions

4. Implement the AI adoption process from the Cloud Adoption Framework for Azure

Plain English: CAF has a dedicated AI adoption scenario with six phases and a companion AI agent adoption scenario with four phases. Know both by name.

Key exam facts — CAF AI adoption, six phases: Strategy → Plan → Ready → Govern → Secure → Manage:

  • AI Strategy: (1) AI use-case identification (business problems → use cases → classify individual-work vs business-automation → generative vs nongenerative); (2) AI technology strategy via the Microsoft AI decision tree — four adoption models trading customization for simplicity: ready-to-use Copilots (SaaS) → low-code SaaS development (Copilot Studio, M365 Copilot extensibility) → managed PaaS (Microsoft Foundry — agents, RAG, fine-tuning; Azure ML/Fabric for training; Foundry Tools for prebuilt models) → Azure infrastructure (IaaS) (bring-your-own models, most control/most ops); weigh capabilities, data needed, skills required, cost; (3) Responsible AI strategy (Responsible AI policies); (4) Data strategy (governed, high-quality data; unify data platform).
  • AI Plan: assess AI skills → acquire AI skills → access AI resources → prioritize AI use cases → create AI proof of concepts → implement responsible AI. (Startup checklist is shorter: access resources, establish responsible AI.)
  • AI Ready: AI governance (separate management groups for internet-facing "online" vs internal "corporate" AI workloads; AI-specific Azure Policy for Foundry, Foundry Tools, AI Search; deploy AI resources in workload subscriptions/application landing zones), AI networking, AI reliability, AI foundation, choose architectures.
  • Govern AI: assess AI organizational risks → document AI governance policies → enforce policies → monitor risks. Aligns to the NIST AI Risk Management Framework.
  • Secure AI: discover AI security risks → protect AI resources and data → detect AI security threats.
  • Manage AI: manage AI operations (establish an AI Center of Excellence; choose MLOps for classic ML vs GenAIOps for generative workloads; standardize SDKs; sandbox environments), manage AI deployment, manage AI models, manage AI costs, manage AI data, manage AI business continuity.
  • CAF AI agent adoption, four phases: (1) Plan for agents (business plan, technology plan, organizational readiness, data architecture); (2) Govern and secure agents (responsible AI, governance & security, prepare environment); (3) Build agents (single vs multi-agent, build process); (4) Manage agents (integrate, operate).

5. Design the strategy for building AI and agents in business solutions

Plain English: Turn the CAF guidance into an actionable build strategy: adopt SaaS first, then extend, then build low-code, then pro-code — with success metrics and governance from day one.

Key exam facts:

  • CAF agent decision tree: primary question — does a SaaS (prebuilt) agent meet functional requirements? If yes, adopt it. If no, build: Copilot Studio (SaaS low/no-code, retrieval/task/autonomous agents, built-in responsible AI, fast process transformation), Microsoft Foundry (PaaS, pro-code + low-code declarative agents, workflows, hosted agents, choose your own models, deep integration — strategic transformation), or GPUs & containers (IaaS — compliance-sensitive/fully custom, ACA serverless GPU for SLMs, AKS for LLMs).
  • Hybrid pattern: use Copilot Studio's low-code interface with Foundry's advanced models (bring your own Foundry model into Copilot Studio) to balance speed and capability.
  • Validation: run time-boxed experiments (1–2 weeks per option), document findings with stakeholder review, and assess single-agent vs multi-agent before scaling.
  • Success metrics discipline (business plan): set baseline KPIs, use metrics as go/no-go decision gates at each stage, evaluate post-deployment against targets; retire or redirect underperforming agents.
  • Build process at scale (CAF "process to build agents"): standardized environments, model selection strategy (match model capability to task complexity; use smaller models for routine tasks; use Foundry model catalog/leaderboards; use Model Router for productivity/action agents; enforce quota and cost governance; validate before deployment), evaluations, and CI/CD.
  • The audience profile expects "agentic-first" thinking: define roadmap for agentic-first business processes, use open standards MCP (Model Context Protocol) and A2A (Agent2Agent).

6. Design a multi-agent solution using Microsoft 365 Copilot, Copilot Studio, and Microsoft Foundry

Plain English: Multi-agent systems split responsibilities across specialized agents that delegate to each other. Microsoft supports this within Copilot Studio (child/connected agents), within Foundry (connected agents + workflows), and across platforms via standard protocols.

Key exam facts:

  • Single vs multi-agent (CAF): single-agent = simpler, predictable, less overhead; multi-agent = modularity, separation of concerns, scalability, but more coordination/orchestration. Start single-agent unless the use case crosses security/compliance boundaries, involves multiple teams, or has known future growth.
  • Copilot Studio options to add other agents: (1) Child agents — lightweight agents inside one agent (same config/lifecycle) to logically group tools/instructions/knowledge; (2) Connected agents — other published Copilot Studio agents in the same environment ("Let other agents connect to and use this one" setting; description drives routing; option to pass or withhold conversation history); (3) External agents: agents over the A2A protocol, Microsoft Foundry agents (preview), Fabric data agents (preview), and Microsoft 365 Agents SDK agents (preview, via Activity Protocol).
  • When child vs connected: child agents for a single use case owned by one team, no separate deployment/settings/reuse. Break into connected agents when tool selection degrades — rule of thumb more than 30–40 actions (tools/topics/agents) — or when different teams own agents, need separate ALM, separate channels, separate model settings, or reuse across multiple main agents.
  • Known limits: extra orchestration hops increase latency; testing/governance surface grows; an agent that is a main agent with connected agents cannot itself be a connected agent of another main; Fabric data agents can't be redirected from topics and don't work when the main agent is deployed to M365 Copilot; citations may be lost across agent boundaries; connected agents respect the main agent's general-knowledge setting.
  • How routing works: orchestrator picks a connected agent based on its name/description, the user message/context, and the primary agent's instructions; write distinct, specific descriptions.
  • A2A protocol: open standard contract for agent-to-agent communication — send tasks, structured metadata, predictable responses, multi-turn; use for external-framework agents; use MCP for tools/resources; HTTP/custom connectors for plain APIs.
  • Microsoft Foundry: Foundry Agent Service supports connected agents and multi-agent workflows (sequential logic, conditional branching, state management across agents); supports Activity Protocol and A2A; agents publishable to Microsoft 365 and Microsoft Agent 365 (the control plane for deploying, governing, and managing agents at scale regardless of where they were built).
  • Microsoft 365 Copilot: agents built anywhere (Copilot Studio, Foundry, Agents SDK) can surface in M365 Copilot; declarative agents can be invoked with @mentions; Copilot Studio agents can call Fabric/Foundry agents to compose cross-platform solutions.

8. Define solution rules and constraints when building with Copilot Studio, Microsoft Foundry and Foundry Tools

Plain English: "Rules and constraints" = instructions, guardrails, moderation, and admin policies that bound what an agent can say and do.

Key exam facts (Copilot Studio):

  • Instructions define scope, boundaries, behavior; child/connected-agent descriptions steer routing; prompt modification/custom instructions influence generative answers.
  • Content moderation is configurable at three levels — agent level (Generative AI settings), topic level (generative answers node — takes precedence at runtime), and prompt-tool level. Levels range Lowest → Highest; higher = fewer but safer answers; default is High.
  • Knowledge constraints: Allow ungrounded responses off = block answers not derived from a knowledge source/tool in that turn (fallback topic triggers); Use general knowledge off = restrict to configured knowledge; Use information from the web / Web Search toggle (uses Grounding with Bing Search); Tenant graph grounding with semantic search (requires generative orchestration + "Authenticate with Microsoft"; extra 10-credit billing for non-Copilot-licensed users).
  • Built-in protections: content moderation on all generative requests, checked twice (input and pre-response), covering jailbreak, prompt injection, prompt exfiltration, and copyright; RAG with citation/provenance/semantic-similarity checks; default AI disclosure message. Admins can disable publishing generative-answer agents tenant-wide; DLP policies govern which connectors/knowledge/tools agents may use.
  • Orchestration constraint choice: classic (topic-triggered, deterministic limits per source type) vs generative orchestration (agent picks tools/knowledge; filters sources above 25 via internal model; doesn't support custom data/Bing Custom Search at agent level).
  • Agent Builder (declarative, lite) can only prioritize specified sources ("Only use specified sources"), not fully block general knowledge — use Copilot Studio for stricter control.

Key exam facts (Microsoft Foundry / Foundry Tools):

  • Azure AI Content Safety (Foundry Tools) provides harm-category filters (hate, sexual, violence, self-harm) with severity thresholds, Prompt Shields (jailbreak/indirect injection), groundedness detection, protected-material detection, and blocklists. Content filters are configured on model deployments; with model router, the content filter and rate limits apply once at the router level for all underlying models.
  • Foundry Agent Service constraints: instructions per agent, tool allow-lists (tool catalog, MCP, OpenAPI), basic vs standard setup (network isolation, BYO storage) for compliance; Azure Policy for allowed model deployments/publishers; entra-based RBAC; evaluations and tracing in the agent lifecycle.
  • WAF guardrail patterns: always ground; hallucination filters; tool-use enforcement in code (don't rely on the model to follow instructions); document refusal thresholds.

10. Determine when to build custom agents or extend Microsoft 365 Copilot

Plain English: The classic extensibility decision: declarative agent (extend Copilot) vs custom engine agent (bring your own orchestration/models) vs Copilot Studio custom agent vs Foundry.

Key exam facts:

  • Declarative agents: use Copilot's orchestrator + foundation models; instructions + knowledge + actions; no extra hosting; inherit M365 security/compliance/Responsible AI; user-initiated only; run in M365 surfaces (Copilot Chat, Teams, Word, Outlook, SharePoint). Build with SharePoint agents, Agent Builder (low-code), Copilot Studio, or Agents Toolkit (pro-code). Best for focused scenarios inside Copilot.
  • Custom engine agents: bring your own orchestration and models (any model, e.g., fine-tuned/industry models); support proactive/autonomous triggers, complex workflows, agent-to-agent collaboration; deployable to M365 and external channels (web, custom apps); require your own hosting, security, and RAI compliance. Build with Copilot Studio, Microsoft 365 Agents SDK, Teams SDK, or Microsoft Foundry (Semantic Kernel/LangChain supported).
  • Decision drivers: stay declarative when you want Copilot's orchestration/models, M365-only surfaces, inherited compliance, quick low-code delivery. Go custom engine when you need choice of models/orchestration, autonomy/proactive triggers, non-M365 channels, group collaboration, or advanced integration. A declarative agent can be converted to a custom engine agent (Agents Toolkit path).
  • Copilot Studio agent vs extend M365 Copilot: Copilot Studio custom agents publish to many channels (web, Teams, D365 Contact Center, etc.) with their own orchestration; "agents for Microsoft 365 Copilot" built in Copilot Studio are equivalent to declarative agents. Choose Foundry when you need pro-code control, custom models, hosted agents, or deep Azure integration.

  • CAF framing: "Extensibility tools for Microsoft 365 Copilot" (declarative agents, Agent Builder, Agents Toolkit, Copilot connectors) sit between ready-to-use Copilots and Copilot Studio in the adoption-model spectrum.

11. Determine when custom AI models should be created

Plain English: Escalate only as needed: prompt engineering → RAG/grounding → fine-tuning → custom/trained models.

Key exam facts:

  • Customization ladder: (1) Prompt engineering — first and cheapest lever (in-context learning); (2) RAG — add organizational knowledge at inference; the CAF says "buy and ground prebuilt models" covers most use cases; (3) Fine-tuning (Foundry, LoRA-based) — when you have hundreds-to-thousands of high-quality examples and need: reduced prompt-engineering overhead, consistent style/tone, specific output formats/schemas, better tool-calling accuracy, improved retrieval integration, efficiency (distillation: use a large model's outputs to fine-tune a smaller/cheaper model, e.g., o1 traffic → GPT-4o-mini); benefits = higher quality than prompts alone, train beyond context-window limits, token savings, lower latency; costs = training (tokens × epochs × price; global/developer tiers discounted) + hosting/inference, plus operational commitment to re-tune when base models change; (4) Train your own model — Azure Machine Learning or Microsoft Fabric for classic ML on your data; AKS/ACA GPUs for bring-your-own models.
  • AI Builder (Power Platform): prebuilt models (receipt/invoice/ID reading, sentiment, language detection, entity extraction, text recognition, translation) for common scenarios → use immediately; custom models (document processing, object detection, prediction, category classification, custom entity extraction) when data is unique to your business — build, train, publish; requires Dataverse and AI Builder capacity (consumes Copilot Credits under current licensing). Copilot Studio prompts can also bring your own Foundry model.
  • WAF decision factors custom vs prebuilt: data control, customization need, cost and upkeep (custom = ongoing maintenance, skilled team), performance/scale (prebuilt services are optimized), expertise available. "Usually, choosing a prebuilt model or a managed service is a better option."

Choose custom models when: domain-specific accuracy/compliance can't be met by prompting + RAG over prebuilt models, you have quality training data, and you can fund the MLOps lifecycle. Otherwise use prebuilt Foundry Models/Foundry Tools + grounding.

12. Provide guidelines for creating a prompt library

Plain English: A prompt library is a curated, shared, governed collection of reusable prompts so teams don't reinvent effective prompts and quality/compliance stays consistent.

Key exam facts:

  • Microsoft 365 Copilot Prompt Gallery: Microsoft-curated prompts + user-saved prompts ("Your Prompts") + team-shared prompts ("Teams") inside M365 Copilot/Copilot Chat; supports saving, liking, sharing prompts with teams for consistency and collaboration; admins can track and export saved/liked/shared prompt data for governance/analytics.
  • Copilot Studio / Power Platform prompt library (prompt builder): predesigned prompt templates (document extraction, data transformation, content generation…) accessible from Copilot Studio Tools → Prompt, or Power Apps/Power Automate AI hub → Prompts; makers build, test, and save reusable prompts with input variables (text, image/document) and Dataverse knowledge grounding; prompts run on Foundry-hosted models (e.g., GPT-4o/4o-mini; model + temperature configurable; JSON output supported).

Prompts consume Copilot Credits and require Dataverse in the environment; regional availability limits apply.

13. Develop the use cases for customized small language models (SLMs)

Plain English: SLMs (like Microsoft's Phi family) are compact models that trade breadth for speed, cost, privacy, and deployability — often fine-tuned for a narrow domain.

Key exam facts:

  • Advantages: faster and cheaper to train/run; small storage/memory footprint (easy deploy, including edge/on-device); less prone to overfitting; more interpretable/explainable.
  • Use cases: limited data/resources; well-defined narrow tasks; high-precision/low-recall tasks; sensitive or regulated tasks needing transparency; classification, summarization, entity extraction, simple Q&A.
  • Phi family: Phi-3 (mini 3.8B, small 7B), Phi-4 series incl. Phi-4-mini (built into Edge's on-device Prompt API); MIT-licensed; available in the Foundry model catalog and fine-tunable in Foundry.
  • Deployment patterns: local SLM sidecar on Azure App Service (complete data privacy, zero external dependencies, predictable sub-100ms latency, fixed cost — for healthcare/finance/legal/offline scenarios); Azure Container Apps serverless GPU for SLM hosting; edge/on-device (Edge Prompt API hardware requirements: Win10/11 or macOS 13.3+, 20 GB storage, 5.5+ GB VRAM).
  • Customization: fine-tune an SLM for domain tasks; distillation transfers a large model's task performance to an SLM for lower cost/latency at scale; CAF model-selection strategy: "smaller, optimized models handle routine tasks…reserve premium models for complex reasoning."

Choose an SLM when: cost/latency-sensitive high-volume tasks, edge/offline/data-residency constraints, narrow domain where a fine-tuned SLM matches big-model quality. Choose an LLM when broad reasoning, creativity, or complex multi-step orchestration is required.

14. Provide prompt engineering guidelines and techniques

Plain English: Prompt engineering designs model inputs (instructions, examples, cues, system messages) to get reliable outputs — the first optimization step before RAG or fine-tuning.

Key exam facts (Foundry prompt-engineering guidance):

  • Zero-shot = no examples (relies on model knowledge; cheap; good baseline). One-shot/few-shot = include input→output example pairs to condition behavior (in Chat Completions, add example user/assistant turns after the system message); pick varied examples covering edge cases; beware learned false patterns and token cost.
  • System message / metaprompt: defines role, boundaries, tone, output format (e.g., JSON), safety constraints; appears first; influences but doesn't guarantee compliance — layer with filters and evaluation; make fallback behavior explicit.
  • Techniques: clear instructions; cues (jumpstart text that seeds output format); primary/supporting content; chain-of-thought-style breakdown (break the task down; step-by-step); recency bias means order matters (instructions before and after content can help — "double down"); give the model an out ("respond 'not found' if the answer isn't present") to reduce fabrications; be specific and descriptive; use analogies; temperature/top_p tuning (reasoning models ignore them).
  • Best-practice list (exam-friendly): Be specific. Be descriptive. Double down. Order matters. Give the model an "out."
  • Microsoft 365 Copilot prompt elements: Goal, Context, Expectations, Source(s) — the four "ingredients" taught in Copilot prompting guidance/Prompt Gallery; iterate and refine.
  • Copilot Studio-specific: write agent/tool/topic descriptions carefully (they drive orchestration); use prompt modification for generative answers; configure knowledge per node; test iteratively; for flows: "When X happens, do Y" phrasing, name connectors.
  • Prompt engineering is iterative "art and science" using in-context learning; it's the recommended starting point before fine-tuning.

C. Evaluate the costs and benefits of an AI-powered business solution

17. Select ROI criteria, including total cost of ownership (TCO)

Plain English: ROI = (benefits − costs) ÷ costs. TCO covers the full lifecycle cost: licenses, consumption, implementation, data readiness, and ongoing operations.

Key exam facts — benefit/ROI criteria:

  • Microsoft's agent business-value guidance uses a four-pillar value framework with quantitative + qualitative metrics per pillar, leading vs lagging indicators, and the published Agent Assisted Hours formula (weight sessions by knowledge grounding, multiply by minutes saved, convert to hours × loaded hourly rate → $ value). Copilot Studio has a built-in Savings calculator (time or money saved per run or per tool, added to the Analytics tab; admins can disable money-saving rules).
  • CAF business plan: baseline KPIs before building; business metrics as go/no-go decision gates; post-deployment evaluation. Benefit categories: efficiency/cost reduction, speed/responsiveness, scalability; measurable results: lower operating costs, improved CSAT, faster innovation.
  • Standard financial vocabulary (Azure business-case glossary): TCO, ROI (net benefits ÷ costs), NPV, payback period, CAPEX vs OPEX, net cash flow.

Key exam facts — TCO/licensing components (verify current prices on official pages):

  • Microsoft 365 Copilot: per-user subscription license (list ~$30/user/month) — includes Researcher/Analyst, Copilot in apps, and use of declarative agents by licensed users.
  • Copilot Studio: tenant subscription (capacity pack) = 25,000 Copilot Credits per pack per month (list ~$200/pack); pay-as-you-go via Azure subscription (list ~$0.01 per credit); Copilot Studio user license $0 (maker access). Prepaid credits are consumed first; PAYG handles overage automatically; unused credits don't roll over (monthly reset on the 1st); capacity is pooled tenant-wide and allocated to environments; enforcement/service denial can apply if capacity is exceeded without PAYG.
  • Copilot Credits billing rates (per event): classic answer = 1; generative answer = 2; agent action (tool/autonomous) = 5; tenant graph grounding = 10; agent flow actions = 13 per 100 actions; AI tools: basic text/GenAI 1 per 10 responses, standard 15, premium 100; content processing 8 per page; voice: classic 10/min, GenAI voice 35/min, premium GenAI voice 75/min. Usage by Microsoft 365 Copilot-licensed users of Copilot Studio agents = no charge (a major TCO lever). Test-chat messages aren't billed. Preview features do bill. Forecast with the Copilot Studio agent usage estimator.
  • Other TCO lines: Foundry/Azure consumption (model tokens in/out, storage, compute, grounding connections, AI Search), fine-tuning training + hosting, D365 licenses and agent consumption/pay-as-you-go, implementation and integration effort, data preparation/AI-readiness, testing/evaluation, monitoring/administration, training and change management (the guidance stresses "the license isn't the investment — adoption is").

18. Create an ROI analysis for the proposed AI solution for a business process

Plain English: A repeatable method: define value before building, baseline the process, project benefits and TCO, gate the pilot with metrics, then measure actuals.

19. Analyze whether to build, buy, or extend AI components

Plain English: Buy (adopt SaaS agents/Copilots), extend (customize prebuilt Copilots/agents with your data and actions), or build (Copilot Studio low-code or Foundry pro-code) — decided per use case.

Key exam facts:

  • CAF agent decision tree: "Does a SaaS agent meet your functional requirements?" Yes → buy/adopt (M365 Copilot agents like Researcher/Analyst/App Builder/Workflows, GitHub Copilot, Fabric data agents, Azure Copilot, D365 agents, Security Copilot). No → build (Copilot Studio SaaS low-code for process transformation; Foundry PaaS for strategic/deep integration; GPUs & containers IaaS for compliance-heavy full control).
  • Extend sits between: M365 Copilot extensibility (declarative agents, Copilot connectors, plugins/API actions), extending D365 first-party agents and Copilots (e.g., add knowledge/topics via Copilot Studio), and D365/Copilot Studio agent templates (prebuilt but customizable, "clear and measurable ROI," rapid deployment in days).
  • Tradeoffs (CAF adoption models): Copilots/SaaS = fastest value, least customization; SaaS dev platforms = business-team velocity, but heavy customization hits limits and may force a re-platform to PaaS; PaaS = control + engineering skill needed; IaaS = maximum control, maximum operational ownership. Evaluate each option on capabilities, data needed, skills required, cost.
  • ISV/marketplace angle: partner-built agents from the Agent Store/Marketplace are "buy" options; the well-architected SaaS AI strategy table compares buy prebuilt / customize (fine-tune) / build your own with benefit-tradeoff pairs.
  • Validation practice: time-boxed prototypes comparing low-code vs pro-code before committing; stakeholder-reviewed recommendation.

Choose: buy when a prebuilt agent covers ≥ the functional need; extend when the gap is knowledge/actions on top of an existing Copilot/agent; build low-code when the process is unique but integration is standard; build pro-code when you need custom models, orchestration, isolation, or multi-agent workflows beyond low-code limits.

20. Implement a model router to intelligently route requests to the most suitable model

Plain English: Model router is a deployable Foundry "model" that is actually a trained routing model: it analyzes each prompt in real time and forwards it to the best underlying chat model — cutting cost and latency while keeping quality — behind one endpoint.

Key exam facts:

  • Deployed like any Foundry model (model-router in the catalog); single deployment, single API, single rate limit; the selected underlying model is always disclosed in the response's model field (transparency). It does not store prompts and honors data-zone boundaries (routes only to models allowed by your deployment's geography/compliance and your access).
  • Routing modes: Balanced (default — picks the most cost-effective model within ~1–2% of top quality), Cost (wider ~5–6% quality band, cheapest), Quality (best model regardless of cost). Mode changes take up to 5 minutes.
  • Model subsets: choose which underlying models participate (cost/compliance/performance control); new models are excluded by default until added; router recalibrates automatically.
  • Version 2025-11-18 (GA): supports Global Standard and Data Zone Standard deployments; underlying models include GPT-5 series (gpt-5.2, gpt-5.2-chat), gpt-4o/gpt-4o-mini, Anthropic Claude (claude-haiku-4-5, claude-sonnet-4-5, claude-opus-4-1, claude-opus-4-6 — Claude models must be deployed separately to your Foundry resource first; all other supported models need no separate deployment), grok-4 / grok-4-fast-reasoning, DeepSeek-V3.1/V3.2, gpt-oss-120b, Llama-4-Maverick. Supports tools/agentic scenarios, so it can be used in Foundry Agent Service (fast models for classification subtasks, reasoning models for analysis — per step). reasoning_effort is supported; temperature/top_p are ignored when a reasoning model is selected.
  • Constraints: effective context window is limited by the smallest underlying model (use subsets for large-context needs); content filter and tokens-per-minute limits are set once on the router deployment and apply to all underlying models.
  • When to use router vs direct deployment: router for diverse prompt mixes, cost optimization, latency-sensitive high traffic, single-endpoint simplicity, agents, automatic failover, simplified model lifecycle (deprecated models replaced transparently with auto-update). Direct deployment when you need the same model on every request (parameter-sensitive/compliance-pinned workloads). Hybrid pattern: router as default path + direct deployments for specialized workloads.
  • CAF ties it to agent cost governance: "for productivity and action agents, use the Model Router to dynamically optimize costs while maintaining quality."

Cross-cutting quick-reference (exam triggers)

  • "Fixed, repeatable steps" → Power Automate/agent flow, not an agent. "Static Q&A over fixed docs" → classic RAG. "Ambiguity + tools + reasoning" → agent.
  • "Fastest deployment, standard scenario" → prebuilt/SaaS agent. "Low-code custom" → Copilot Studio. "Own models/orchestration/pro-code" → Foundry. "Full infrastructure control" → GPUs/containers.
  • "Works only inside M365, inherits compliance, no hosting" → declarative agent. "Own models, proactive triggers, external channels" → custom engine agent.
