Security for agents

1 · Entra Agent ID — agents become first-class identities

The blueprint holds the keys; the identity holds nothing. Whether a human is present decides how the agent authenticates.

  • The blueprint is the template/class and holds the credentials (federated identity credentials, certificates, secrets); it can be multitenant.
  • The agent identity is a special service principal with no credentials of its own; single-tenant.
  • Foundry auto-provisions a default blueprint + shared project identity; publishing an agent creates a dedicated identity.
  • Attended = OBO (“the user is the badge”) · autonomous = client credentials (“the agent brings its own badge”).
  • Entra features extend to agents: Conditional Access (P1) · Identity Protection / risky agents (P2) · Identity Governance (ownership/sponsorship, lifecycle workflows, access reviews); all agent sign-ins are logged.
  • Identity inventory spans Foundry + Copilot Studio + third-party agents (Agent 365 registry); external agents onboard via the Entra Auth SDK sidecar or workload identity federation.

2 · The two authentication layers — design both doors

User→agent and agent→tools are separate decisions. Least privilege applies at every hop.

User → agent

  • Three options: No auth (public — DLP can forbid) · Authenticate with Microsoft (the Teams/M365 channel requirement) · Manual OAuth2 — federated credentials recommended over secrets; exposes User.AccessToken.

Agent → tools & data

  • Delegated (user) connections / OBO when actions must respect each user's permissions; application / managed identity for autonomous agents, scoped to minimal RBAC. Foundry tool auth: key · Entra (managed identity) · OAuth identity passthrough.
  • Least privilege everywhere: tool allow-lists (allowed_tools), short-lived just-in-time privileges, Dataverse security roles, human-in-the-loop for consequential actions.

3 · Network security — Copilot Studio locks & Foundry tiers

Studio hardens the SaaS box; Foundry lets you climb to full network isolation.

Copilot Studio locksVNet support for connectors · IP firewall · Continuous Access Evaluation · Direct Line channel security · customer-managed keys (CMK) · Customer Lockbox.

Private MCP endpointsSupported — tool traffic can stay off the public internet too.

Model security

4 · Model hardening — keys, planes, roles & CMK

Five facts that carry every model-security question, including the two classic exam nuances.

Go keylessPrefer Entra (keyless) auth; disable key auth with disableLocalAuth=true enforced via policy.
Two RBAC planesData plane ≠ control plane — control-plane Owner does not automatically get inference rights.
Foundry rolesFoundry User / Foundry Owner / Foundry Account Owner / Foundry Project Manager. Nuance: Foundry User can train a fine-tune (data plane) but deploying it is control-plane — Foundry Owner covers both.
Training data promisesStored in-region, logically isolated, deletable, AES-256 at rest; your fine-tuned models are exclusive to you and never train foundation models.
The CMK nuanceKey Vault must be same region/tenant with soft-delete + purge protection. Revoking CMK blocks new training/deployments — but already-deployed models keep serving until deleted.
Content filtersConfigured at the resource level, associated per deployment; turning filters off (or annotate-only) requires Limited Access approval.

5 · The threat → mitigation map — including prompt manipulation

The exam bullet says “analyze vulnerabilities and mitigations, including prompt manipulation.” This is the whole map.

ThreatMitigations
Direct injection / jailbreak (UPIA)Prompt Shields for user prompts (on by default), safety system messages, input validation, blocklists, red-team testing (PyRIT)
Indirect / cross-prompt injection (XPIA)Prompt Shields for documents + Spotlighting (tags untrusted content low-trust), isolate untrusted content, least-privilege tools, human-in-the-loop for risky actions; Purview audit exposes XPIADetected
Data leakage / oversharingPurview sensitivity labels (encrypted content honored via VIEW + EXTRACT), DLP + Endpoint DLP (blocks pasting into consumer AI), DSPM for AI oversharing assessments, SharePoint Advanced Management, Restricted Content Discovery — note Restricted SharePoint Search retiring (new enablement blocked from 2026-07-31)
Hallucination / ungrounded outputGroundedness detection (preview, with correction), groundedness evaluators, RAG grounding, ungrounded-attributes evaluator
Harmful contentContent filters — 4 categories (hate, sexual, violence, self-harm) × severity safe/low/medium/high, default medium, configurable separately for prompts vs completions
IP / copyright reproductionProtected material detection (text + code) — on by default; ties to the Customer Copyright Commitment
Runtime attacks on AI workloadsDefender for Cloud AI threat protection (GA) — runtime alerts from Prompt Shields signals + threat intel, Defender XDR integration; AI-SPM discovers the AI bill-of-materials and attack paths
Excessive agency / tool misuseTask-adherence checks, prohibited-actions & sensitive-data-leakage evaluators, tool allow-lists, approvals
  • Reference frameworks the docs cite: OWASP Top 10 for LLM applications and MITRE ATLAS.

Governance for agents

6 · The four control planes — who governs what

The most common governance question: match the requirement to the right admin surface.

PPAC (Power Platform admin center)Copilot Studio agents: DLP data policies, environments & groups, sharing limits, quarantine agents (work only in the test canvas), tenant switch to disable publishing of generative-AI agents, gen-AI consent settings, environment routing, inventory incl. drafts, credit capacity + per-agent caps.
M365 admin center — Copilot Control System (CCS)Inventory of everything available to users (first-party + ISV + shared): block/unblock tenant-wide, approve/reject Copilot Studio submissions, deploy/assign to users/groups, pin; agent sharing controls (all / specific / none).
Microsoft Agent 365Cross-origin control plane — Observe · Govern · Secure: unified registry incl. shadow + third-party agents (sync from Bedrock, Vertex AI, Salesforce, Databricks), Agent Map (agent↔person↔data), per-instance blocking, Agent Tools registry for MCP governance; integrates Entra Agent ID, Purview, Defender.
Entra admin centerAgent identities: Conditional Access, risky-agent detection, lifecycle workflows, access reviews.
💡 Hook: “block a rogue agent” → CCS block (M365 agents) or PPAC quarantine (Copilot Studio) · “find shadow agents from other clouds” → Agent 365 registry · “risky sign-ins by an agent” → Entra.

7 · DLP mechanics — the connector traffic light

Data policies are the enforcement muscle of agent governance. Know the classes, the combining rule, and the two error names.

  • Three classes: Business / Non-business / Blocked; business + non-business can't combine in one resource; Microsoft-owned standard connectors can't be blocked.
  • Tenant + environment policies combine with the most restrictive winning.
  • Granular controls: connector action control · endpoint filtering (HTTP, HTTP+Entra, Webhook, SQL Server, Azure Blob, SMTP) · custom-connector URL patterns · virtual connectors switching Copilot Studio capabilities (channels, knowledge sources, auth requirements, App Insights) on/off.
  • Violations block at both moments: publish (DlpViolationError) and runtime (DataLossPreventionViolation).
  • Third-party AI: block/isolate third-party AI connectors via DLP (especially in the default environment); Purview endpoint/browser DLP stops pasting into consumer AI; the governed alternative is bring-your-own frontier model inside tenant governance (Foundry catalog models in prompts / model picker).

Compliance — residency, access & audit

8 · The data residency map — where data sits vs where it's processed

One rule anchors everything: data at rest never moves — only processing location changes. Then learn the toggles.

  • Copilot Studio: data stored in the chosen environment geography; the EU Data Boundary applies when the tenant billing address is EU/EFTA and environments are in EUDB geos.
  • The caveat everyone forgets: M365-powered features store conversation history in the end user's Exchange mailbox region.
  • PPAC cross-geo toggles (per environment, Generative AI settings): “Move data across regions” (needed where Azure OpenAI isn't local; not shown for US/Gov; irreversible for past data) · Bing search consent · M365 services consent · EUDB “flex routing during peak load” — uncheck it to keep all inference in-boundary.
  • Foundry deployment types: at-rest data always stays in the resource's geography — Global / Data Zone / Regional change only processing; fine-tuning training data stored in-region (the Global training tier trades residency for cost).
  • M365 Copilot: interaction content + semantic index live in the tenant's Local Region Geography; Advanced Data Residency / Multi-Geo extend commitments (storage follows Preferred Data Location).
💡 Hook: “EU processing only” = Data Zone EU + EUDB flex-routing unchecked. And Grounding with Bing always leaves the compliance boundary.

9 · Access controls on grounding data & tuning

Agents never out-see their users; search indexes need trimming; tuning splits see/train from deploy.

M365: no new permissionsCopilot & agents retrieve only what the current user's Graph/SharePoint/Teams ACLs allow; sensitivity labels add VIEW + EXTRACT gating; S/MIME and password-protected files are excluded.
DataverseSecurity roles (row + column) enforced for agents and the Dataverse MCP server; connector-file knowledge does live permission checks at query time; transcripts need Bot Transcript Viewer.
Azure AI Search trimmingThe GA pattern: a filterable group_ids-style field + search.in() security filter at query time. Previews: native ACL/RBAC scopes (ADLS Gen2/Blob), SharePoint ACL sync, Purview label enforcement.
Tuning accessFoundry project RBAC separates who can see/train vs deploy; strictest control = BYO storage + CMK; DLP knowledge-source and endpoint controls protect Copilot Studio grounding.

10 · Audit trails — prove who did what, where, to which data

A direct exam bullet: design audit trails for changes to models and data. Four ledgers cover everything.

Responsible AI — principles, standard & red teaming

11 · Six principles → six verifiable checks

“Review the solution for RAI adherence” means turning each principle into something you can inspect.

PrincipleArchitect's verifiable check
⚖ FairnessCohort testing of outputs across demographics; representative grounding data
🛡 Reliability & safetyEvaluations + red teaming as a release gate; failure modes handled; fallback paths
🔒 Privacy & securityPermission trimming, data boundaries, encryption, DLP
🤝 InclusivenessAccessibility and language coverage as design requirements
🔍 TransparencyAI disclosure to users, citations, documented limitations (transparency notes / model cards)
👤 AccountabilityNamed owner per agent, human-in-the-loop for consequential actions, audit trails, incident response
  • Complete an impact assessment at design time, scale the release gate to risk, and monitor post-deployment (feedback, incidents, drift).

12 · The Standard v2 & the shared-responsibility slider

The Standard turns principles into requirements; the layer you buy decides which mitigations are yours.

  • Responsible AI Standard v2: the six principle domains contain 14 goals, each decomposed into concrete engineering requirements across the lifecycle.
  • Anchored practices: RAI Impact Assessments (template published; required for sensitive uses), Transparency Notes for platform services + application cards for apps, annual transparency reporting.
  • Governance chain: Board committee → Responsible AI Council → Office of Responsible AI (policy, sensitive-use review).
  • The shared-responsibility slider: SaaS → Microsoft owns most · PaaS → you own the system message, grounding & filters · IaaS → you own training data and model safety.

13 · Red teaming — automated adversaries with a scorecard

The AI Red Teaming Agent industrializes attacks so the launch decision has a number on it.

  • Built on Microsoft's open-source PyRIT + safety evaluators; output = the Attack Success Rate (ASR) scorecard by category × complexity.
  • Attack strategies: Base64 / ROT13 / leetspeak / morse encodings, character manipulation, jailbreak, XPIA via mock tool outputs, multi-turn / crescendo, compositions.
  • Local runs = single-turn text; cloud runs add agent targets + scheduled/continuous scans. Use alongside human red teaming — the security benchmark expects continuous red teaming.
💡 Final hook: the harm mechanics chant — categories scored 0–7, filter levels safe/low/medium/high, default medium, safety evaluators report a defect rate, red teaming reports ASR.
